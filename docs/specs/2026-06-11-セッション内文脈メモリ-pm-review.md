## 0. 確定事項（本レビューで変更しない不変条件）

- トリガーキーは Ctrl + Shift + Space。
- 保持件数 N は 5 件。
- `[CONTEXT]` の合計上限は 500 文字（UTF-16 長）。
- 既定は OFF（`contextMemoryEnabled = false`）。
- 消去はプロセス終了と ON→OFF 切替のみ。deactivate では消去しない。

## 1. ユーザーストーリーの詳細化

### ペルソナ A: ターミナルで AI コーディングエージェントへ指示するエンジニア

Apple Terminal / Ghostty / iTerm2 で Claude Code や Codex CLI に日本語の指示を書く開発者。特徴は以下のとおり。

- 直前の数メッセージで文脈（対象ファイル・エラー内容・方針）をすでに書いており、次の指示は「あれやっといて」「さっきのやつで」のような省略表現になりがちである。
- 省略表現のままエージェントへ送ると意図が伝わらず往復が増えるため、送信前に具体的な文へ展開したい。
- 文脈つき候補が外したときは、通常変換の候補とその場で見比べて選びたい。
- composition が無いときにキーをアプリに奪われたくない（ターミナルのショートカット環境を壊さない）。

### ペルソナ B: チャットとエディタを行き来するエンジニア

Slack / メール / エディタ / ブラウザを数分単位で行き来しながら作業する開発者。特徴は以下のとおり。

- エディタで書いた設計メモの内容を、直後にチャットで別の人へ説明する場面が多い。アプリをまたいでも「さっき書いたこと」を参照できることに価値がある（全アプリの入力が合流する IME の固有価値）。
- 話題が切り替わったら古い文脈は邪魔になるため、直近 5 件・500 文字の FIFO で自然に流れていく挙動が望ましい。
- 文脈が要らない変換では従来の Shift + Space を使い続け、必要なときだけ Ctrl + Shift + Space を選ぶ。

### ペルソナ C: プライバシー重視ユーザー

入力内容に業務機密・個人情報が含まれうることを前提に IME を選ぶユーザー。特徴は以下のとおり。

- 既定で何も収集されないこと（opt-in）を導入判断の条件にする。
- 有効化しても、文脈がデバイスから出ない・診断ログに残らない・いつでも消せることをドキュメントで確認したい。
- OFF にした瞬間に過去の文脈が消えること、再起動で必ず消えることを信頼の根拠にする。

## 2. 受け入れ基準のテストシナリオ化

仕様書「受け入れ基準」の 7 項目を AC-1〜AC-7 とし、Given/When/Then 形式に展開する。AC-1〜AC-5 と AC-7 は `swift test --package-path KotoInput` で自動検証、AC-6 は自動検証（コードパス検査）と実機計測の併用である。

### AC-1: 既定 OFF で従来挙動と完全一致

#### シナリオ 1-a: OFF ではキーを消費しない

- Given: `contextMemoryEnabled = false`（既定）。composing 状態である。
- When: `keyCode == space && essentialFlags == [.control, .shift]` のキーイベントが InputController に届く。
- Then: `false` を返してアプリへ通す。`.requestContextualConversion` は発行されない。composing でない場合も同様に通す。

#### シナリオ 1-b: OFF では収集ゼロ

- Given: `contextMemoryEnabled = false`。
- When: 任意の非空テキストを commit する（Enter 由来・deactivate 由来の両方）。
- Then: `SessionContextStore` への `append` が発生しない。`snapshot()` は空のまま。

#### シナリオ 1-c: 設定の decode 後方互換

- Given: `contextMemoryEnabled` キーを含まない既存の設定 JSON。
- When: `ConversionSettings` を decode する。
- Then: `contextMemoryEnabled == false` になる（`decodeIfPresent ?? false`）。既存フィールドの値は巻き戻らない。

#### シナリオ 1-d: プロンプトのバイト同一性と instructions の固定性

- Given: `contextEntries` が空の `ConversionRequest`。
- When: `PromptBuilder.prompt(modelInput:contextEntries:)` でプロンプトを生成する。
- Then: 従来の `[INPUT]` のみのプロンプトとバイト単位で同一である。また日本語 instructions は設定値・文脈の有無によらず常に同一文字列である（`[CONTEXT]` の扱いを定める固定 1 行は無条件に含まれる）。これにより prewarm（ADR-0005）が無効化されない。

#### シナリオ 1-e: 既存テストの回帰なし

- Given: 本機能のコード変更を適用したブランチ。
- When: 既存のテストスイート全体を実行する。
- Then: すべて green。ただし instructions の文字列内容を検証する既存テストは、固定 1 行の追加に合わせた期待値更新のみ許される（条件分岐の追加は不可）。

### AC-2: ON のときの FIFO 保持

#### シナリオ 2-a: 非空 commit の追記

- Given: `contextMemoryEnabled = true`。store は空。
- When: `abc` を commit する。
- Then: `snapshot() == ["abc"]`。追記は同期キーハンドリングの外（`Task { @MainActor in ... }`）で行われる。

#### シナリオ 2-b: 空白のみのテキストは無視

- Given: ON。
- When: 空白・改行のみのテキストを commit する。
- Then: `snapshot()` は変化しない。

#### シナリオ 2-c: 件数上限 5 件の FIFO

- Given: ON。5 件保持済み。
- When: 6 件目を `append` する。
- Then: 最古の 1 件が破棄され、直近 5 件が古い順に保持される。

#### シナリオ 2-d: 合計 500 文字（UTF-16）の FIFO

- Given: ON。合計が 500 文字近くまで保持済み。
- When: 合計が 500 文字を超えるエントリを `append` する。
- Then: 合計が 500 文字以内に収まるまで古いエントリから破棄される。

#### シナリオ 2-e: 1 エントリ 500 文字超の切り詰めとサロゲートペア

- Given: ON。UTF-16 長 500 の境界がサロゲートペアの中間に落ちる位置に絵文字等（サロゲートペア文字）を含む 600 文字のテキスト。
- When: `append` する。
- Then: エントリは UTF-16 長 500 以下に切り詰められ、サロゲートペアが分断されない（ペアを壊す場合はペアの手前で切る）。

#### シナリオ 2-f: deactivate 由来の commit も収集対象

- Given: ON。composing 状態でテキストを保持している。
- When: deactivate により commit が発生する。
- Then: そのテキストが `append` される。store は消去されない（確定事項: 消去はプロセス終了と OFF 切替のみ）。別アプリで activate した後も `snapshot()` は保持されたままである。

### AC-3: ON かつ composing 中の Ctrl + Shift + Space で文脈つき日本語 AI 変換

#### シナリオ 3-a: 基本動作（scripted provider）

- Given: ADR-0004 の ScriptedConversionProvider で `CompositionCoordinator` を構築する。ON。store に複数エントリを保持。composing 状態で `areyattoite` を入力済み。
- When: `.requestContextualConversion` コマンドを発行し、変換タスクの完了を待つ。
- Then:
  - provider が受け取った `ConversionRequest.contextEntries` が `snapshot()` と一致する（古い→新しい順）。
  - `ConversionRequest.target` が `.japanese` である（第一版は日本語 target のみ）。
  - プロンプトが `[CONTEXT]`（各エントリを `- ` 付き・古い順）→ 空行 → `[INPUT]`（かな化済み入力）の形である。
  - `CompositionState.sourceText`（Escape 復元用スナップショット）が元のローマ字のまま変わらない。

#### シナリオ 3-b: 結果が候補蓄積に入る

- Given: 通常変換（Shift + Space）の候補が蓄積済みの converted 状態。
- When: `.requestContextualConversion` を発行し完了を待つ。
- Then: 文脈つき変換の結果が既存の候補蓄積（ADR-0012）に追加され、↑/↓ で通常候補と巡回して見比べられる。

#### シナリオ 3-c: attempt 同一性判定キーは（target, useContext）

- Given: `.requestContextualConversion` で converted 状態（attempt 0）。
- When: 再び `.requestContextualConversion` を発行する。
- Then: `ConversionRequest.attempt` が 1 になる（温度 0.8 の再抽選。greedy は attempt 0 のみ）。
- When: 続けて `.requestConversion(.japanese)`（Shift + Space 相当）を発行する。
- Then: `attempt` は 0 から始まる（useContext が異なるため別系列）。さらに `.requestContextualConversion` へ戻った場合も attempt 0 から始まる（直前の attempt 系列と useContext が異なるため）。`CompositionState.conversionUsedContext` がこの判定を担う。

#### シナリオ 3-d: スナップショットの不変性

- Given: ON。変換タスク開始時に store からスナップショット済み。
- When: 変換タスクの実行中に新しいテキストが commit され `append` される。
- Then: 実行中の `ConversionRequest.contextEntries` は変化しない（リクエストは不変）。

#### シナリオ 3-e: store が空のまま ON で押した場合

- Given: ON。store は空。composing 状態。
- When: `.requestContextualConversion` を発行する。
- Then: 変換自体は実行され、プロンプトは `[INPUT]` のみ（従来とバイト単位で同一）。エラーにならない。

#### シナリオ 3-f: ON でも composing が無ければキーを通す

- Given: ON。composition が存在しない。
- When: Ctrl + Shift + Space のキーイベントが InputController に届く。
- Then: `false` を返してアプリへ通す。`.requestContextualConversion` は発行されない。

#### シナリオ 3-g: 設定確認のタイミング

- Given: ON または OFF の任意の状態。
- When: Ctrl + Shift + Space 以外のキーを連続入力する。
- Then: キーストロークごとの設定ロードが発生しない（設定確認は当該キー組み合わせの押下時のみ）。

### AC-4: ON→OFF で次のインタラクション時点の全消去

#### シナリオ 4-a: 読み出し時の消去

- Given: ON で複数エントリを保持した後、設定を OFF に変更する。
- When: composing 中に Ctrl + Shift + Space を押す（OFF 後最初のインタラクション）。
- Then: coordinator が設定 OFF を検知して `store.clear()` を実行し、空を返す。キーは消費されずアプリへ通り、文脈注入は発生しない。

#### シナリオ 4-b: 追記時の消去

- Given: ON で複数エントリを保持した後、設定を OFF に変更する。
- When: テキストを commit する（OFF 後最初のインタラクション）。
- Then: `append` は行われず `store.clear()` が実行される。`snapshot()` は空。

#### シナリオ 4-c: 再 ON 後に過去の文脈が復活しない

- Given: 4-a または 4-b で消去済み。設定を再度 ON にする。
- When: `snapshot()` を読む。
- Then: 空である（空の状態から収集が再開される）。

#### シナリオ 4-d: deactivate は消去トリガーではない

- Given: ON で複数エントリを保持。
- When: deactivate → 別アプリで activate する。
- Then: `snapshot()` は保持されたままである（確定事項。シナリオ 2-f と対）。

### AC-5: 既存安全機構の維持

#### シナリオ 5-a: Escape 復元

- Given: `.requestContextualConversion` で converted 状態。
- When: `restoreSource` コマンド（Escape 相当）を発行する。
- Then: `displayedText` が元のローマ字テキストへ戻る。文脈つき変換の結果が復元先にならない。

#### シナリオ 5-b: stale 結果拒否

- Given: 文脈つき変換タスクの実行中。
- When: composition のテキストが変更される（追加入力等）。
- Then: 後から完了した stale な変換結果は適用されない（既存機構の継承）。

#### シナリオ 5-c: 保護語検証

- Given: 入力に `Claude Code` を含む composing 状態で文脈つき変換を実行。
- When: `ConversionOutputValidator.validate` が出力を検証する。
- Then: `Claude Code` が出力から消えた場合はバリデーション失敗（リトライ or 拒否）になる。`[CONTEXT]` の有無は検証ロジックに影響しない（validator は無変更）。

#### シナリオ 5-d: 膨張率検証は sourceText 基準のまま

- Given: 短い入力（例: `areyattoite`）に長い文脈を付与して変換し、provider が膨張率上限（4.0 × 原文長 + 固定許容）を超える長文を返すよう script する。
- When: `ConversionOutputValidator.validate` が出力を検証する。
- Then: 出力は従来どおり拒否され、Escape で原文へ戻れる。文脈エントリの長さは膨張率の分母に算入されない（sourceText 基準のまま）。

### AC-6: レイテンシ回帰なし

#### シナリオ 6-a: 決定論パス（Tab）の不変

- Given: 本機能のコード変更を適用したブランチ。
- When: 決定論変換（Tab）のコードパスを検査・計測する。
- Then: 10 ms budget のパスに設定ロード・store アクセス・文脈組み立てが追加されていない（ADR-0006 / ADR-0013 Decision 6）。

#### シナリオ 6-b: 通常 AI 変換（Shift + Space）の不変

- Given: 同上。
- When: `.requestConversion(.japanese)` のパスを検査する。
- Then: `useContext == false` のパスでは store の読み出し・文脈組み立てが発生しない。

#### シナリオ 6-c: メモリ書き込みは hot path 禁止

- Given: ON。
- When: commit を伴うキーイベントを処理する。
- Then: `store.append` は同期キーハンドリングの返却後に `Task` で実行される（キーイベント処理時間に算入されない）。

#### シナリオ 6-d: 文脈組み立ての 20 ms budget

- Given: 最大値（5 エントリ・合計 500 文字）の store。
- When: `[CONTEXT]` セクションを組み立てる。
- Then: 組み立てが 20 ms 以内に完了する（ADR-0013 のレイテンシ収支）。

### AC-7: カバレッジ 100% と CI green

#### シナリオ 7-a: カバレッジとゲート

- Given: 全実装とテストが完了したブランチ。
- When: `bun scripts/architecture-harness.ts --staged --fail-on=error` と `make before-commit` を実行する。
- Then: harness invariant 違反ゼロ。lint / typecheck / test / build がすべて green。カバレッジ 100% を維持する。

#### シナリオ 7-b: フィクスチャ契約（ADR-0013 Decision 8）

- Given: `multilingual-quality.json` に `context`（string[]、任意）フィールドを追加し、ja の文脈あり / なしペアを 2 組以上収録する。
- When: `MultilingualQualityFixtureTests` の契約テストを実行する。
- Then:
  - `context` が存在するケースで空配列が無い。
  - 各 entry が 500 文字（UTF-16 長）以内である。
  - 同じ題材の文脈なしケースと id 接尾辞 `-with-context` / `-without-context` でペアになっている。
  - 実在の私的テキストが収録されていない（レビューで確認）。

## 3. 優先度と依存関係

### 実装順序（依存の流れ）

```
P0: ConversionSettings.contextMemoryEnabled     P0: SessionContextStore（KotoCore）
    （decode 後方互換）                              （append / snapshot / clear、UTF-16 FIFO）
            |   ※ 相互依存なし。並行実装可 ※            |
            +--------------------+--------------------+
                                 ↓
P0: ConversionRequest.contextEntries + PromptBuilder.prompt(modelInput:contextEntries:)
    + 日本語 instructions への固定 1 行追加（AC-1-d、AC-3-a）
                                 ↓
P0: 状態機械拡張（.requestContextualConversion / conversionUsedContext /
    Effect.startConversion(useContext:)）（AC-3-c）
                                 ↓
P0: coordinator 統合（commit 時 append・変換時 snapshot・OFF 時 clear）（AC-2、AC-4）
                                 ↓
P0: InputController キールーティング（Ctrl + Shift + Space）（AC-1-a、AC-3-f）
                                 ↓
P0: scripted provider による統合テスト（AC-1〜AC-6 の自動検証分）
                                 ↓
P0: フィクスチャ拡張 + 契約テスト（AC-7-b）
                                 ↓
P1: README・ドキュメント更新 → 実機確認
```

### 実装ユニットの依存まとめ

| 優先度 | 作業 | 依存 | 根拠 |
|---|---|---|---|
| P0 | `ConversionSettings.contextMemoryEnabled`（既定 `false`、`decodeIfPresent ?? false`） | なし | すべての ON/OFF 分岐の前提。後方互換 decode を最初にテストで固定する。 |

### リリース判定

- P0 がすべて green であることが PR マージの必須条件（CI で機械検証可能）。
- 実機確認（文脈あり / なし候補の見比べ・OFF 切替の即時消去・既定 OFF でのキー透過）は、PR 時点で未実施の場合はチェックリストとして PR 本文に残し、マージ前に記録する。
- PR 本文の「Known follow-ups」節に `/follow-up list-pr-body` の出力を貼る（CLAUDE.md の運用）。

## 4. スコープ外の明確化と将来 ADR へ送る判断基準

仕様書「スコープ外」を再確認し、将来 ADR / 別 Issue 化の判断基準を補記する。ADR-0013 の Tradeoff（第一版の採用率・品質改善幅で検証してから判断する）を共通原則とする。

| スコープ外項目 | 今回外す理由 | 将来 ADR / Issue 化の判断基準（再検討トリガー） |
|---|---|---|
| working / vocabulary / project 層、SQLite 永続、セッションをまたぐ記憶 | 検証前のストレージ設計はオーバーエンジニアリング（ADR-0013 Tradeoff）。新ディスク I/O を持たないことが第一版の低リスク性の根拠。 | 第一版の採用率（opt-in 率）と品質改善幅（フィクスチャの文脈あり / なし差分・実利用フィードバック）が永続化の価値を示したとき。TTL・暗号化・消去 UI を含む ADR を起こす。 |
| アプリ別無効化・incognito モード・セキュアフィールド非収集の明示的制御 | 第一版は IMK がパスワード欄で IME を無効化する既定動作に依存する（README に明記）。コントロール追加は設定 UI を伴い第一版の範囲を超える。 | ADR-0013 Decision 7 が掲げるコントロール一覧の残項目。「特定アプリで収集されて困った」という具体的フィードバック、またはターミナル等のセキュアフィールドを使わない機密入力（リスク R-1）が実害として報告されたとき。 |
| window title 等のアプリ連動文脈（Accessibility 権限） | 新権限なしが第一版の方針（ADR-0013 Decision 4）。権限要求は導入障壁とプライバシー懸念を増やす。 | commit テキストのみの文脈では解決できない参照（「この画面の」等）の需要が具体化し、費用対効果が明確になったとき（ADR-0013 Tradeoff）。 |
| push 型の自動提案 | ADR-0013 Decision 1 が pull 型のみと決定済み。自動介入によるタイピング阻害・誤挿入のリスクをゼロにする。 | 原則変更にあたるため、採用する場合は ADR-0013 を supersede する新 ADR が必須。pull 型の利用実績から「提案を見逃して困る」需要が定量的に示されたときのみ検討する。 |
| 翻訳 target への文脈付与 | 第一版は日本語 target のみ。翻訳 instructions・プロンプトは無変更で、多言語変換キー機能との干渉を避ける。 | 日本語での文脈品質（フィクスチャ回帰 + 実利用）が確認できたとき。多言語フィクスチャの `context` ペアを他言語へ拡張する Issue を起こす。 |
| 設定 UI | 既存どおり UserDefaults の JSON 編集で足りる。UI は第一版の検証に不要。 | opt-in 率が「設定手段が JSON 編集のみ」であることに阻害されていると判断できたとき。incognito・全消去ボタン等（Decision 7）とまとめて設計する。 |
