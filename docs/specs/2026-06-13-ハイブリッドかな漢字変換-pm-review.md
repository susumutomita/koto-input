- 対象仕様書: [docs/specs/2026-06-13-ハイブリッドかな漢字変換.md](./2026-06-13-ハイブリッドかな漢字変換.md)（承認済み。本書はこれを正とする）。

- ロール: Product Manager。

## 0. 確定事項（本レビューで変更しない不変条件）

- 辞書ソースは mozc 系（BSD-3-Clause、`dictionary_oss`）。MVP は高頻度語彙にサブセットして同梱する。
- 変換粒度は段階導入。今回は読みの最長一致による単一最良パス変換のみ。文節分割と Viterbi ラティスは後続 PR。
- 辞書が一次変換（決定的・正解を候補集合に必ず含める）、AI が二次変換（文脈再ランク・整形）。
- 既定の変換は単一最良出力のみを確定し、候補リストを常時提示しない。切り替えは外したときだけの任意の保険。これが差別化の核であり受け入れ基準として固定する。
- 辞書はローカル同梱データであり外部送信しない。AI はオンデバイスのまま。入力テキストを外部サービスへ送らない既存 invariant を維持する。
- 出力は AI 経路・辞書フォールバック経路のいずれでも `ConversionOutputValidator` を必ず通す（プロンプトはセキュリティ境界ではない）。

## 1. ユーザーストーリーの詳細化

### ペルソナ A: 日本語を多用するが候補選択を嫌うエンジニア

ターミナルやエディタで日本語の文章を継続的に打つ開発者。特徴は以下のとおり。

- 「ほうほう」「けっきょく」のような頻出語を打って一発で漢字交じり文を得たい。候補ウィンドウを開いて選ぶ操作が思考を止めるため、従来 IME の候補選択を最大の不満点だと感じている。
- 現状の純 AI 変換が読みを据え置く no-op（「ほうほう」がそのまま出る）を、機能の信頼性を損なう致命的な体験だと捉えている。
- まれに変換を外したときだけ、その場で別表記へ切り替えて救済したい。日常的には候補を見たくない。

### ペルソナ B: Apple Intelligence を有効化できない環境のユーザー

実機が未対応・モデル未ダウンロード・設定で AI を無効化している等の理由で、オンデバイス AI が使えない状態の利用者。特徴は以下のとおり。

- AI が不可用でも、辞書だけで読みが漢字化され、最低限の一発変換が成立してほしい。「AI が無いと何も変換されない」状態は導入を断念する理由になる。
- AI 不可用時の挙動が決定的で予測可能であること（同じ読みが常に同じ漢字になる）を信頼の根拠にする。

### ペルソナ C: 配布物の健全性を重視するユーザー / レビュアー

Homebrew の公開 tap 経由で導入する利用者と、配布物のライセンス・プライバシーを点検するレビュアー。特徴は以下のとおり。

- 同梱辞書のライセンス（BSD-3-Clause）が遵守され、著作権表示と出典が配布物に明記されていることを確認したい。
- 辞書がローカルデータであり、入力テキストが外部へ出ないことを設計上の不変条件として保証してほしい。
- 配布サイズが過大にならないこと（MVP は高頻度サブセット）を導入判断に含める。

## 2. 受け入れ基準のテストシナリオ化

仕様書「受け入れ基準」の 8 項目を AC-1〜AC-8 とし、Given/When/Then 形式に展開する。AC-1〜AC-6 は `swift test --package-path KotoInput`（toolchain の無い環境では CI の swift ジョブ）で自動検証、AC-7・AC-8 はゲート実行とコードパス検査で検証する。No Mock 原則により、辞書を用いるテストは実 mozc 由来の小規模サブセット辞書を使う。

### AC-1: 代表読みの決定的な漢字化（no-op 撲滅）

#### シナリオ 1-a: 単一語の決定的変換

- Given: 実 mozc サブセット辞書をロードした `DictionaryConverter`。
- When: 読み `ほうほう` を `convert` する。
- Then: 単一最良表記が `方法` である。全かな未変換（`ほうほう` のまま）ではない。同様に `けっきょく` → `結局` を検証する。

#### シナリオ 1-b: 連接読みの最長一致セグメント分割

- Given: 同上の辞書。
- When: 複数語が連接した読み（例: 辞書に収録された 2 セグメント分の読み）を `convert` する。
- Then: 読みが左から最長一致でセグメント分割され、各セグメントの最小コスト表記が連結される。境界が決定的である。

#### シナリオ 1-c: 辞書ヒットしないセグメントの残置

- Given: サブセット辞書に存在しない読みを含む入力。
- When: `convert` する。
- Then: ヒットしないセグメントはかなのまま残り、ヒットするセグメントは漢字化される。例外やクラッシュにならず、後段の AI とフォールバックに委ねられる。

#### シナリオ 1-d: ハイブリッド経路での no-op 撲滅（統合）

- Given: ADR-0004 の ScriptedConversionProvider で AI 役を構成した `HybridConversionProvider`。AI が読みをそのまま返す（no-op）よう script する。
- When: `ほうほう` を `convert` する。
- Then: AI 出力が検証で弾かれ、dictBest（`方法`）が確定する。最終出力が全かな未変換にならない。

### AC-2: 既定は単一最良・候補リスト非提示・切り替えは任意

#### シナリオ 2-a: 既定確定は単一最良のみ

- Given: `HybridConversionProvider` が dictBest と複数の dictCandidates を持つ読みを変換する。
- When: 変換キーを 1 回押して確定相当の状態に至る。
- Then: 確定値（displayedText）は単一最良出力のみ。候補リストは表示されず、別候補が自動で割り込まない。

#### シナリオ 2-b: 候補プールへの蓄積順序

- Given: AC-2-a の converted 状態。
- When: 候補プールの中身を検査する。
- Then: AI 出力・dictBest・dictCandidates の代替表記・別 attempt の順で `ConversionCandidate` が蓄積される。蓄積された候補はすべて `ConversionOutputValidator` 通過済みである（未検証出力を候補にしない不変条件、ADR-0012）。

#### シナリオ 2-c: ↑↓ の任意操作でのみ巡回

- Given: converted かつ候補プールが 2 件以上。
- When: 修飾キーなしの ↑（keyCode 126）/ ↓（keyCode 125）を押す。
- Then: 候補プールを wrap around で巡回し displayedText を差し替える（ADR-0012）。候補が 2 件未満、または composing でなければキーは消費されずアプリへ通る。

#### シナリオ 2-d: 切り替えは Escape / Enter の既存規則と直交

- Given: ↑↓ で別候補を表示中。
- When: Escape を押す。
- Then: sourceText（原文ローマ字）へ戻る。Enter は表示中の候補のみを commit する。dictCandidates の混在で復元先・commit 対象が変わらない。

### AC-3: AI 不可用時の辞書フォールバック

#### シナリオ 3-a: availability が unavailable

- Given: `availability()` が `.unavailable(reason:)` を返す AI 役を script した `HybridConversionProvider`。
- When: `ほうほう` を `convert` する。
- Then: AI を呼ばず dictBest（`方法`）を確定フォールバックとして返す。例外を投げない。

#### シナリオ 3-b: availability が preparing

- Given: `availability()` が `.preparing` を返す AI 役。
- When: `convert` する。
- Then: dictBest を確定フォールバックとして返す。後で再試行可能な状態として扱う。

#### シナリオ 3-c: AI 例外時のフォールバック

- Given: `convert` で例外を投げる AI 役。
- When: `convert` する。
- Then: 例外が伝播せず dictBest を確定フォールバックとして返す。

#### シナリオ 3-d: AI 出力が検証 NG のときのフォールバック

- Given: 検証で弾かれる出力（全かな未変換・膨張率超過・保護語喪失のいずれか）を返す AI 役。既存 attempt 機構の自動 retry も全て検証 NG。
- When: `convert` する。
- Then: 最終的に dictBest を確定フォールバックとして返す。原文ローマ字や読みのままにはならない。

### AC-4: ConversionOutputValidator の no-op 検出拡張

#### シナリオ 4-a: 全かな未変換を failure 扱い

- Given: target が `.japanese`。出力が読み（全かな）と実質同一の未変換である。
- When: `ConversionOutputValidator.validate(output:source:settings:target:)` を呼ぶ。
- Then: `.failure` を返す。これにより no-op が converted として確定しない。

#### シナリオ 4-b: 正常な漢字交じり出力は success

- Given: target が `.japanese`。出力が漢字交じり文（例: `方法`）。
- When: `validate` する。
- Then: `.success` を返す。漢字化が成立した出力を誤って failure にしない。

#### シナリオ 4-c: 自動エスカレーションへの接続

- Given: 初回 attempt の出力が全かな未変換で failure。
- When: coordinator の自動 retry 機構（ADR-0015、最大追加 2 回）が走る。
- Then: attempt を増やして再変換する。全 retry が検証 NG でも、ハイブリッド経路では最終的に dictBest が確定フォールバックになる（AC-3-d と接続）。validator は緩めない。

#### シナリオ 4-d: 非日本語 target には適用しない

- Given: target が翻訳（`.japanese` 以外）。出力が全かなを含む正当な訳文。
- When: `validate` する。
- Then: no-op 判定を適用せず、従来どおり検証する。日本語固有のルールが翻訳経路を壊さない。

#### シナリオ 4-e: 既存検証ロジックの回帰なし

- Given: 既存の膨張率・保護語・頭字語・末尾句点・括弧 unwrap の各テストケース。
- When: no-op 検出を追加した `validate` を実行する。
- Then: 既存テストが全て green。no-op 検出が既存の判定と干渉しない。

### AC-5: 辞書引きと最良パス選択の決定性

#### シナリオ 5-a: 同一読みの繰り返し変換

- Given: 同一の辞書とサブセットを 2 回ロードした `DictionaryConverter`。
- When: 同じ読みを複数回 `convert` する。
- Then: 単一最良出力が毎回完全に一致する。

#### シナリオ 5-b: 同コスト表記の安定ソート

- Given: 同一読みに対し最小コストが同点の表記が複数ある辞書ケース。
- When: 最良表記を選ぶ。
- Then: 安定ソートで一意の表記が選ばれ、実行ごとに結果が揺れない。

#### シナリオ 5-c: dictCandidates の順序安定性

- Given: 代替表記を複数持つセグメント。
- When: `convert` を複数回実行する。
- Then: dictCandidates の集合と順序が毎回同一である（候補巡回の再現性のため）。

### AC-6: 既存テスト全 green とカバレッジ 100% 維持

#### シナリオ 6-a: スイート全体の green

- Given: 本機能のコード変更を適用したブランチ。
- When: `swift test --package-path KotoInput`（または CI の swift ジョブ）を実行する。
- Then: 既存テストを含む全テストが green。

#### シナリオ 6-b: カバレッジ 100%

- Given: 全実装とテストが完了したブランチ。
- When: カバレッジを計測する。
- Then: 新規 `DictionaryConverter`・`HybridConversionProvider`・validator 拡張を含めてカバレッジ 100% を維持する。辞書ヒットしない分岐・フォールバック分岐・no-op 検出分岐がテストで網羅される。

### AC-7: architecture-harness の invariant 違反ゼロ

#### シナリオ 7-a: staged ゲート

- Given: 実装をステージしたブランチ。
- When: `bun scripts/architecture-harness.ts --staged --fail-on=error` を実行する。
- Then: invariant 違反ゼロで通る。

#### シナリオ 7-b: No Mock 原則の遵守

- Given: 辞書を用いるテスト群。
- When: harness の `INVARIANT_NO_MOCK_DATA` 検査を通す。
- Then: スタブ辞書・`mockData` / `stubApi` 系を含まない（実 mozc サブセットを使う）。ADR-0004 の ScriptedConversionProvider は AI 役の例外として既存どおり許容される。

### AC-8: 入力テキスト非送信 invariant の維持

#### シナリオ 8-a: 辞書はローカル参照のみ

- Given: `DictionaryConverter` と同梱辞書リソース。
- When: 変換を実行する。
- Then: 辞書引きはオンメモリのローカルデータ参照のみ。ネットワーク I/O・外部サービス呼び出しが発生しない。

#### シナリオ 8-b: AI はオンデバイスのまま

- Given: `HybridConversionProvider` の AI 役（`AppleFoundationModelsProvider`）。
- When: 再ランク・整形を実行する。
- Then: 既存どおりオンデバイス推論であり、入力テキストを外部へ送らない（ADR-0002 維持）。

## 3. 優先度と依存関係

### 実装順序（依存の流れ）

CLAUDE.md の作業順序原則（ドキュメント更新 → リファクタリング → 機能追加）に従い、先行リファクタの validator 拡張を最初に入れる。これは辞書導入前でも独立して価値がある（no-op を確定させない安全機構の強化）ためである。

```
P0: ドキュメント先行（ADR-0016 起草、NOTICE / THIRD-PARTY-LICENSES、harness invariant 追記の要否判断）
                                 ↓
P0: ConversionOutputValidator 拡張（全かな未変換 no-op を .japanese で failure）（AC-4）
    ※ 辞書導入前でも独立して価値がある先行リファクタ。最初に入れる。
                                 ↓
P0: DictionaryConverter（KotoCore・決定的・I/O 非依存）（AC-1, AC-5）
    + 辞書データのビルド前処理とサブセット同梱（AC-1, AC-8, US-7）
                                 ↓
P0: HybridConversionProvider（DictionaryConverter + AppleFoundationModelsProvider 合成）（AC-2, AC-3）
                                 ↓
P0: IME 統合（候補プール蓄積・prewarm での辞書ロード・↑↓ 巡回の踏襲）（AC-2, US-9）
                                 ↓
P0: scripted provider による統合テスト（AC-1〜AC-5 の自動検証分）
                                 ↓
P0: ゲート（harness / before-commit / カバレッジ）（AC-6, AC-7, AC-8）
                                 ↓
P1: README・ドキュメント仕上げ → 実機確認
```

| P0 | `HybridConversionProvider`（`TextConversionProvider` 実装、`DictionaryConverter` と `AppleFoundationModelsProvider` を合成） | validator 拡張 + DictionaryConverter + 辞書データ | フォールバック判定（availability / 例外 / 検証 NG → dictBest）が AC-3 の核。AI を再ランクへ格下げする配線の中心。 |

### なぜこの順序か（依存の理由）

- validator → DictionaryConverter → HybridProvider → IME 統合の順は「安全機構を先に固め、決定的な一次変換を据え、二次の AI を合成し、最後にユーザー体験へ配線する」流れである。
- validator 拡張を先頭に置くのは、no-op を failure 扱いにする判定が HybridProvider のフォールバック判定（検証 NG → dictBest）の前提だからである。validator を後回しにすると、no-op が success として確定し AC-1 / AC-4 を同時に満たせない。
- DictionaryConverter は I/O とモデルに依存しない純粋ロジックなので、provider・IME より前に単体で決定性（AC-5）を固定できる。逆順だと決定性のバグが上位層のテストに紛れて切り分けが難しくなる。
- HybridProvider は validator と辞書の両方が揃って初めてフォールバック分岐をテストできる。IME 統合は HybridProvider の出力（dictBest・dictCandidates）が確定してから候補プールへ積む配線になる。

### リリース判定

- P0 がすべて green であることが PR マージの必須条件（CI で機械検証可能）。
- 実機確認（一発変換の体感・候補非提示・AI 不可用時のフォールバックの自然さ・prewarm 維持）は、PR 時点で未実施の場合はチェックリストとして PR 本文に残し、マージ前に記録する。
- PR 本文の「Known follow-ups」節に `/follow-up list-pr-body` の出力を貼る（CLAUDE.md の運用）。

## 4. スコープ外の明確化と follow-up にする判断

仕様書「スコープ外」を再確認し、follow-up（別 PR）にする理由と将来の再検討トリガーを補記する。共通原則は段階導入であり、MVP の一発変換品質と配布サイズの実測を見てから次段を判断する。いずれも `/follow-up add` で記録し、別 PR で処理する（`INVARIANT_FOLLOWUP_TRACKED`）。

| スコープ外項目 | 今回外す理由（follow-up にする判断） | 将来 ADR / Issue 化の判断基準（再検討トリガー） |
|---|---|---|
| 文節分割と Viterbi ラティスによる多文節最小コスト変換 | 最長一致の単一最良パスで no-op 撲滅という主目的は達成できる。ラティスはコスト学習・接続コスト表・探索実装を伴い MVP の範囲を大きく超える。同 PR に混ぜるとスコープクリープになり CI と検証が肥大する。 | MVP の最長一致で「文節境界を誤る」誤変換が実利用・実機で頻発したとき。接続コストとビーム探索を含む ADR を起こし、ADR-0016 を補完する。 |
| ユーザー変換履歴による学習・重み調整 | 検証前の学習機構はオーバーエンジニアリング。永続ストレージ・プライバシー設計を伴い、第一版の低リスク性（新規ディスク I/O なし）を崩す。 | 固定コストの辞書だけでは選択を外す同音異義が定量的に多いと判明したとき。文脈メモリ（ADR-0013）との関係を含め永続化の ADR を起こす。 |
| mozc 全辞書の同梱 | 配布サイズが過大になり、公開 tap 配布の導入障壁を上げる。MVP は高頻度サブセットで一発変換の価値を検証する。 | 高頻度サブセットでヒットしない読みの取りこぼしが実利用で問題化し、サイズ増を上回る価値が示されたとき。サブセット基準とサイズ上限を ADR に記録する。 |
| 送り仮名・用言活用の厳密処理 | 活用形の厳密処理は辞書設計と分割規則の複雑度を上げる。MVP は語幹中心の高頻度語で価値を出す。 | 活用語の誤変換（送り仮名のずれ等）が一発変換の体感品質を損なうと実機で確認できたとき。活用処理の規則を ADR と辞書スキーマに反映する。 |
