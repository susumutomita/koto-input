# ローマ字かな変換層 仕様書

- 日付: 2026-06-10
- 関連 Issue: Issue 6（要件）、Issue 7（UI/UX）、Issue 8（変換ロジック）、Issue 9（統合）、Issue 10（テスト）、Issue 13、Issue 15
- 実装: PR 14（Tab 即時変換・前段かな正規化）、PR 16（保護語除外・アポストロフィ対称化）
- 設計判断の正本: [ADR-0006](../adr/0006-決定論ローマ字かな変換の前段化.md)

## 概要

生のローマ字をそのままオンデバイスモデルへ渡すと分かち書きに失敗して誤変換される（実測: `kyouhaiihida` → 「きょうかいいひだ」「京橋駅」等）。ローマ字 → ひらがなは規則で決定論的に解決できるため、コード側でかな化してからモデルへ渡し、モデルの仕事を「かな漢字変換 + 整文」に絞る。

## ユーザーストーリー

- ターミナルで Claude Code を使う開発者として、ローマ字を素打ちした文章を待ちゼロでひらがな化したい（Tab）。
- 同じ開発者として、Shift + Space の AI 変換が毎回同じ品質で決まってほしい（前段かな正規化 + greedy sampling）。

## 受け入れ基準

実装済みのテストとの対応を併記する（すべて `KotoInput/Tests/KotoCoreTests/`）。

- [x] `kyou ha ame` → `きょう は あめ`、分かち書きなしの `kyouhaiihida` → `きょうはいいひだ`（RomajiKanaConverterTests）
- [x] 促音 `gakkou`/`kitte`/`matcha`、撥音 `shimbun`/`kon'ya`/`onna`/`konnyaku`、小書き・長音（同上）
- [x] `application` / `Claude` / `scripts/foo.sh` / `user_id` / `don't` が破壊されない（同上）
- [x] 保護語（小文字含む）はかな化から除外され、原文のまま [INPUT] に残る（PromptBuilderTests）
- [x] Tab は composition 中のみ消費し、ひらがな化する。composition が無ければターミナルへ素通し（CompositionTransitionTests + InputController）
- [x] モデルへ渡る [INPUT] がかな化されている（PromptBuilderTests の `promptNormalizesRomaji`）
- [ ] 実機 E2E: `kyouhaiihida` → Shift + Space → 「今日はいい日だ」相当（docs/terminal-compatibility.md のシナリオに従いオーナーが確認）

## UI/UX の決定事項（Issue 7）

- 入力中の表示はローマ字のまま（逐次かな変換はしない）。
- Tab で明示的にひらがな化、Shift + Space で AI 変換。どちらも待ち時間の扱いは状態機械（converting 表示・タイプ先行）に従う。
- Escape の復元対象は打たれたままのテキスト（`sourceText`）。前段かな正規化はプロンプト層のみで行い、表示・復元系の挙動を変えない。
- 状態表示は marked text の下線のみ（MVP の制約を維持）。

## 技術設計（実装済みの構成）

- `RomajiKanaConverter.normalize(_:protecting:)`: 語単位 all-or-nothing の決定論変換。保護語区間は語の途中でも原文保持。
- 統合点はプロンプト構築（`PromptBuilder.prompt(sourceText:settings:)`）。`ConversionRequest.sourceText` は打たれたままを保持し、Escape 復元・タイプ先行 splice・出力検証はこれを基準にする。
- 出力検証（保護語）との層間整合: 保護語をかな化から除外することで「保護語は原文どおり出力に残る」検証が成立する。

ローカルセッション（Issue 15）の設計案との差分: `ConversionRequest` を sourceText / modelInputText に分離する案に対し、本実装はかな化をプロンプト層に置いた。validator は元テキスト基準（同案と同じ）。挙動上の要件はどちらも満たす。

## テスト計画（Issue 10）

- 正常系: 基本文・促音・撥音・拗音・小書き・長音・分かち書きなし長文。
- 異常系・境界値: 空文字（normalize は恒等）、英語のみの語（保持）、記号・パス隣接（保持）、語頭 `'`・語頭 `-`（変換しない）、保護語（除外）。
- 統合: プロンプト層でのかな化（PromptBuilderTests）、Tab の reducer 遷移、変換中の Tab はタイプ先行ルールでキャンセル。
- 実機 E2E シナリオ: `docs/terminal-compatibility.md` の「かな正規化の E2E シナリオ」節。

## スコープ外

- かな → 漢字の決定論変換（辞書）。漢字化は LLM の仕事のまま。
- カタカナ・全角記号モード、キーのカスタマイズ。
- 逐次（タイプ中の）かな変換。
