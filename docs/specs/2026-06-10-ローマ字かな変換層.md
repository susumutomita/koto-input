# ローマ字かな変換層 仕様書

## 概要

Shift + Space の変換要求時に、composition のローマ字をコード側で決定的にひらがなへ変換してから Apple Foundation Models に渡す層を追加する。小型オンデバイスモデルが苦手とするローマ字の分かち書きを排除し、変換品質を改善する。

## 背景・解決したい課題

- 実測（2026-06-10、実機の SystemLanguageModel）: 生のローマ字 `kyouhaiihida` を渡すと「きょうかいいひだ」（ユーザー環境では「京橋駅」）に誤変換される。ひらがな `きょうはいいひだ` を渡すと「今日は良い日だ」に正しく変換される。
- 原因: モデルは「かな→漢字・整文」は得意だが、区切りのないローマ字の解読は苦手。現状の Koto はローマ字をそのままモデルに渡している。

## ユーザーストーリー

- ターミナルで Claude Code / Codex に日本語プロンプトを書く開発者として、ローマ字で打った文を正しく日本語へ変換するために Koto を使いたい。
- 同じ開発者として、英語コマンド・識別子（`git`、`Claude Code` 等）が混在した文でも、それらが崩れずに変換されてほしい。

## 振る舞い（ヒアリング決定事項）

1. **変換タイミング**: Shift + Space 時に一括変換する。入力中の表示はローマ字のまま（現状維持）。Escape の復元対象も従来どおり元のローマ字テキスト。
2. **英語混在の扱い**: 空白区切りの語ごとに判定し、「子音が余らず完全にひらがなへ変換できる語」だけかな化する。完全変換できない語（`git`、`Claude` 等）、大文字英字を含む語、protectedTerms に一致する部分は原文を維持する。語の区切り空白（スペース・タブ・改行）は原文どおり保持する。
   - 例: `Claude Code wo tukau to git mo benri` → `Claude Code を つかう と git も べんり`（モデルへ渡す中間表現であり、最終出力ではない）
3. **かな表**: 一般的な日本語 IME 互換とする。ヘボン式・訓令式の両方を受理（`shi`/`si`、`tsu`/`tu` 等）、促音（子音重ね → っ）、拗音（`kya` 等）、小書き（`xtu`/`ltu` 等）。撥音は `n`/`nn` → ん、語末の `n` も ん。`n'` は直後に母音 / `y` が続く場合のみ区切りとして ん（`han'i` → はんい。語末や子音前の `'` は原文維持で、`goin'` の `'` を消さない）。`nn` の直後に母音 / `y` が続く場合は最初の `n` のみ ん（`minna` → みんな）、`n` が 3 連続する場合は `nn` を ん として消費（`konnnichiha` → こんにちは）、`nn'` + 母音も ん（`zenn'in` → ぜんいん）。長音記号 `-` → ー は語内でかなに後続する場合のみ変換し、語頭の `-`（CLI フラグ等）は原文維持。`・`（U+30FB）は通過させるが長音判定の「かな」には含めない。句読点 `.` → 。、`,` → 、 も変換可能文字として扱う（数字や他の記号を含む語は完全変換不能として原文維持。変換でかなを 1 文字も生成しない語は原文維持とし、記号のみの語はこの一般規則で守る）。既存のひらがな・カタカナ・漢字（々〆 を含む）は語内をそのまま通過させる（`人々ha` → 人々は）。
4. **protectedTerms の保護**: 大文字小文字を区別した完全一致で、複数語フレーズ（例: `Claude Code`）も含めて原文の出現箇所を保護する（語単位一致ではフレーズ保護語が一致し得ないため）。語境界はテキスト末尾・空白に加えて句読点 `.` `,` も含む（`sudo,` の `sudo` も保護）。各層（かな化・出力検証・プロンプト）で使う protectedTerms は trim と空要素除去を一元化した `sanitizedProtectedTerms` を用いる。

## 受け入れ基準

- [ ] `RomajiKanaConverter` の単体テストが基本五十音・濁音・半濁音・拗音・促音・撥音・長音・完全変換不能語の原文維持・protectedTerms の原文維持を網羅している。
- [ ] `kyouhaiihida` → `きょうはいいひだ` の決定的変換テストが通る。
- [ ] `Claude Code wo tukau to git mo benri` → `Claude Code を つかう と git も べんり` のテストが通る。
- [ ] `han'i` → `はんい` 等、`n'` の区切り明示のテストが通る。語末の `n'`（`goin'`）は原文維持される。
- [ ] `konnnichiha` → `こんにちは`、`zenn'in` → `ぜんいん` のテストが通る。
- [ ] `-a` などの語頭 `-` を含む語が原文維持されるテストが通る。
- [ ] 複数語の protectedTerms（`Claude Code`）が保護されるテストが通る。
- [ ] 句読点が後続する保護語（`sudo,`）が保護されるテストが通る。
- [ ] 保護語の安全網が元テキスト基準で働く（語境界なし出現の保護語が出力から消えると変換失敗になり、かな化で合成された文字列への偽陽性は起きない）テストが通る。
- [ ] `人々ha` → `人々は` のテストが通る。
- [ ] モデルへ渡る `ConversionRequest.sourceText` がかな化済みであることを scripted provider（ADR-0004）で検証するテストが通る。
- [ ] Escape で元のローマ字テキストへ復元される既存挙動が回帰しない。
- [ ] 実機確認: Koto で `kyouhaiihida` → Shift + Space → 「今日はいい日だ」相当の変換になる。
- [ ] `docs/terminal-compatibility.md` の対象ターミナルで E2E 確認し、結果を同ドキュメントへ追記する。

## 非機能要件

- パフォーマンス: かな変換は O(n) の純関数。モデル呼び出し前の同期処理として体感遅延を生まない。
- セキュリティ・プライバシー: 入力テキストをログ・デバイス外へ出さない方針を維持する（ADR-0002）。
- アクセシビリティ: 表示挙動（marked text の下線のみ）は現状から変更しない。

## 技術設計

- **新モジュール**: KotoCore に `RomajiKanaConverter`（純粋関数の enum、状態を持たない）を追加する。
- **統合点**: `CompositionCoordinator.startConversion` で `sourceText` をかな化してから `ConversionRequest` を構築する。`CompositionState` と表示（composition・restoreSource）はローマ字のまま変更しない。
- **validator**: `ConversionOutputValidator.validate(source:)` にはかな化後のテキストを渡す（maximumExpansionRatio の基準もかな化後の長さになる）。
- **PromptBuilder**: `[INPUT]` にかな化済みテキストが入るため構造は変更不要。instructions の `Convert romaji, ...` の文言は「かな・英語混じりテキストの変換」前提に更新する。
- **protectedTerms の判定**: 大文字小文字を区別した完全一致で、複数語フレーズを含む出現箇所をかな化対象から除外する。
- **ConversionRequest の分離**: `sourceText`（ユーザーが打った元テキスト）と `modelInputText`（かな化後のモデル入力）を分離する。モデルへは `modelInputText` を渡し、ConversionOutputValidator の保護語検査と膨張率基準は元テキスト（`sourceText`）基準を維持する。かな化で合成された文字列への保護語の誤検知（偽陽性）と、語境界なし出現の保護語消失の見逃し（偽陰性）を防ぐため。
- **かな化の実行位置**: かな化はキーイベントの同期パスではなく、変換タスク内（メインアクター外）で行う。
- **データモデル / API / UI**: 変更なし（UI コンポーネントの追加はない）。

## 既知の制限

- 偶然完全変換できる小文字英単語（例: `sudo` → すど、`bun` → ぶん）はかな化される。protectedTerms への登録で防げる。緩和策（英単語辞書等）はフォローアップ。
- 変換失敗時の表示は従来どおり変化しない（marked text の下線のみという既存制約）。失敗状態の可視化はフォローアップ。

## スコープ外

- 打鍵と同時のかな表示（ライブ変換）。
- カタカナ語の明示変換、変換候補 UI。
- ローマ字テーブルのユーザーカスタマイズ設定。
- 文中の一部だけ完全変換できない語が含まれる場合のスペルミス補正。
- 小文字英単語の誤かな化の自動緩和（英単語辞書・ヒューリスティック）。
- 変換失敗状態の可視化 UI。

## 実装合流時の補記（2026-06-10）

本仕様の承認後、クラウド側セッションが ADR-0006（決定論ローマ字かな変換の前段化）として同目的の実装を main へ先行マージした（PR 14・16）。最終実装はクラウド実装（文脈規則ベースの語分割・Tab 即時変換・prewarm）を基盤とし、本仕様のレビューで実証されたバグ修正を移植して合流した（ADR-0007）。本仕様との差分は以下のとおり。

- `Shift + Space`（AI 変換）に加えて `Tab`（AI を使わない即時かな変換）がある。AI 変換のかな化はプロンプト前段の正規化として行われる。
- 語分割は空白区切りの all-or-nothing ではなく、ASCII 英字連続 + 文脈規則（パス・識別子検出）による。`kyou!` → きょう! のように記号前のローマ字も変換される。
- かな表のカバレッジが本仕様より広い（ヘボン式 `m`+b/m/p、`tch`、`thi` 等）。
- 語末の `n'`（`kon'` 等）は原文維持（本仕様どおり。クラウド初期実装の `kon'` → こん から変更）。
- 検証（保護語・膨張率）は元テキスト基準、モデル入力は `ConversionRequest.modelInputText`（本仕様どおり）。
