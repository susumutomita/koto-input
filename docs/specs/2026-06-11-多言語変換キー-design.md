# 多言語変換キー インタラクション設計

- 対象機能: 多言語変換キー（仕様書: [`docs/specs/2026-06-11-多言語変換キー.md`](./2026-06-11-多言語変換キー.md)）。
- 担当 Issue: https://github.com/susumutomita/koto-input/issues/25
- ロール: Designer。
- 日付: 2026-06-11。

## 前提となる表示制約

ローマ字かな変換層の設計（[`docs/specs/ローマ字かな変換層-design.md`](./ローマ字かな変換層-design.md)）と同じく、Koto の表示チャネルは marked text の下線スタイルのみである。ポップアップ・候補ウィンドウ・スピナー・通知は存在せず、本機能でも追加しない（仕様の「データモデル / UI コンポーネント追加なし」に従う）。

翻訳は「プロンプトと検証の言語分岐 + キールーティング」の追加であり、状態機械の構造・表示メカニズムは現行から変更しない。

## 1. キーインタラクション仕様：全状態 × キー操作マトリクス

縦軸は `CompositionPhase`（idle / composing / converting / converted / failed）、横軸は本機能で追加・変更されるキー操作および既存操作。

「消費」= IMK が `true` を返してホストアプリへキーイベントが届かない。「通す」= IMK が `false` を返してホストアプリへ届く。

### 1-1. 多言語変換キー（Ctrl + Shift + E/C/K/F/G/S）

| 状態 | 動作 | 消費 | 備考 |
| --- | --- | --- | --- |
| idle | 何もしない。 | 通す | composition が無い場合は消費しない（仕様 5）。ターミナルのショートカットが通る。 |
| composing | `.requestConversion(target)` を発行する。displayedText をスナップショットし converting へ遷移する。表示はローマ字スナップショットのまま（細下線）。 | 消費 | Shift + Space と同じライフサイクル。 |
| converting | 既存タスクをキャンセルし、現在の sourceText で新しい target の変換を開始する（新 requestID）。表示は不変。 | 消費 | タスクが重なった場合の重複起動を防ぐため既存タスクを cancelConversion してから再開始する。 |
| converted | 同じ target を再押下: attempt + 1 で再抽選（retryCount + 1）。displayedText を sourceText に戻して converting へ。別の target を押下: attempt 0 から新しい target へ変換し直す（sourceText スナップショットは維持）。表示はどちらも sourceText（原文）に戻って細下線。 | 消費 | 原文スナップショットが維持されるため、どの言語間を行き来してもローマ字へ Escape 復元できる。 |
| failed | composing と同じ扱い。`.requestConversion(target)` で再試行する。 | 消費 | エラー後の再試行を多言語でも統一する。 |

### 1-2. Shift + Space（日本語変換）

| 状態 | 動作 | 消費 | 備考 |
| --- | --- | --- | --- |
| idle | 何もしない。 | 通す | 既存挙動を変えない。 |
| composing | `.requestConversion(.japanese)` を発行する。 | 消費 | 既存挙動を変えない。 |
| converting | 既存タスクをキャンセルし `.requestConversion(.japanese)` で日本語変換を再開始する。 | 消費 | 既存挙動を変えない。 |
| converted（翻訳後） | attempt 0 から日本語変換に戻る（仕様 2 「Shift + Space: 日本語変換へ戻す」）。sourceText スナップショットは維持。 | 消費 | 翻訳後から Shift + Space で日本語候補へ差し替えられる。 |
| failed | `.requestConversion(.japanese)` で再試行する。 | 消費 | 既存挙動を変えない。 |

### 1-3. Tab（かな正規化）

| 状態 | 動作 | 消費 | 備考 |
| --- | --- | --- | --- |
| idle | 何もしない。 | 通す | ターミナルの補完を奪わない。既存挙動を変えない。 |
| composing（修飾なし） | `.normalizeToKana` を発行する。即時かな化して composing のまま。 | 消費 | 言語と無関係（仕様 4）。既存挙動を変えない。 |
| converting | 変換タスクをキャンセルし、かな化した上で composing へ戻る（applyEdit 経由）。 | 消費 | 既存挙動を変えない。 |
| converted | composing へ戻る（applyEdit が isSourcePreserved をクリアする）。 | 消費 | 既存挙動を変えない。 |
| failed | composing へ戻る（applyEdit 経由）。 | 消費 | 既存挙動を変えない。 |
| （修飾キーあり） | 何もしない。 | 通す | 既存挙動を変えない。 |

### 1-4. Escape

| 状態 | 動作 | 消費 | 備考 |
| --- | --- | --- | --- |
| idle | 何もしない。 | 通す | 既存挙動を変えない。 |
| composing（isSourcePreserved = false） | `.cancel`。composition を破棄し idle へ戻る。 | 消費 | 既存挙動を変えない。 |
| composing（isSourcePreserved = true かつ displayedText ≠ sourceText） | `.restoreSource`。sourceText（元のローマ字）へ復元し composing へ。 | 消費 | 翻訳後から編集した場合も同様。 |
| converting | タイプ先行中（displayedText ≠ sourceText）: 変換をキャンセルし追記分を保持したまま composing へ戻る。追記なし: sourceText（ローマ字）へ復元して composing へ（仕様 3）。 | 消費 | 既存挙動を変えない。翻訳変換中も同じ。 |
| converted | `.restoreSource`。元のローマ字へ復元して composing へ戻る（仕様 3）。 | 消費 | 翻訳後の Escape でローマ字が戻ることを保証する。 |
| failed | `.restoreSource`。sourceText へ復元して composing へ戻る。 | 消費 | 既存挙動を変えない。 |

### 1-5. Enter / Ctrl + Enter

| 状態 | 動作 | 消費 | 備考 |
| --- | --- | --- | --- |
| idle | 何もしない。 | 通す | 既存挙動を変えない。 |
| composing / converting / converted / failed（修飾なし） | `.commit`。displayedText をそのまま確定して idle へ。 | 消費 | 言語に関わらず共通。既存挙動を変えない。 |
| composing（Ctrl + Enter） | `.insert("\n")`。composition 内に改行を挿入する。 | 消費 | 既存挙動を変えない。 |

### 1-6. Ctrl + C

| 状態 | 動作 | 消費 | 備考 |
| --- | --- | --- | --- |
| composing / converting / converted / failed | `.cancel` で composition を破棄してから通す。 | 通す | 既存挙動を変えない。翻訳変換中も同じ。 |
| idle | 何もしない。 | 通す | 既存挙動を変えない。 |

### 1-7. 状態遷移サマリー（翻訳追加後）

```
idle
  ↓ insert
composing
  ↓ Shift+Space / Ctrl+Shift+E/C/K/F/G/S
converting(requestID, target)
  ↓ conversionSucceeded
converted(requestID, target)
  ↓ Ctrl+Shift+同 target   → converting（attempt+1）
  ↓ Ctrl+Shift+別 target   → converting（attempt 0、同 sourceText）
  ↓ Shift+Space            → converting(.japanese、attempt 0）
  ↓ Escape                 → composing（sourceText 復元）
  ↓ Enter                  → idle（commit）
  ↓ 編集                   → composing（isSourcePreserved クリア）

converting(requestID, target)
  ↓ conversionFailed       → failed
  ↓ Escape                 → composing（sourceText 復元）
  ↓ 編集（スナップショット破壊） → composing（cancelConversion）
```

## 2. ターミナル既定ショートカットとの衝突調査

対象ショートカット: Ctrl + Shift + E、Ctrl + Shift + C、Ctrl + Shift + K、Ctrl + Shift + F、Ctrl + Shift + G、Ctrl + Shift + S。

### 2-1. Terminal.app（Apple 標準ターミナル）

| ショートカット | Terminal.app の既知の割当 | 衝突の有無 |
| --- | --- | --- |
| Ctrl + Shift + E | なし（macOS 標準では未割当） | なし |
| Ctrl + Shift + C | なし | なし |
| Ctrl + Shift + K | なし | なし |
| Ctrl + Shift + F | なし | なし |
| Ctrl + Shift + G | なし | なし |
| Ctrl + Shift + S | なし | なし |

Terminal.app は Ctrl + Shift 修飾キーの組み合わせをほぼ使わない。macOS システムメニューのキーボードショートカットには Ctrl + Shift 系が割り当てられていないため、衝突なし。

### 2-2. iTerm2

| ショートカット | iTerm2 の既知の割当 | 衝突の有無 |
| --- | --- | --- |
| Ctrl + Shift + E | デフォルト未割当（ユーザー設定次第） | なし（デフォルト） |
| Ctrl + Shift + C | デフォルト未割当 | なし（デフォルト） |
| Ctrl + Shift + K | デフォルト未割当 | なし（デフォルト） |
| Ctrl + Shift + F | デフォルト未割当 | なし（デフォルト） |
| Ctrl + Shift + G | デフォルト未割当 | なし（デフォルト） |
| Ctrl + Shift + S | デフォルト未割当 | なし（デフォルト） |

iTerm2 のデフォルトショートカットは主に Cmd 修飾を使う（Cmd + Shift + E がアクティビティ等）。Ctrl + Shift 系はデフォルトで未割当だが、ユーザーがカスタム設定している可能性はある。

### 2-3. Ghostty

| ショートカット | Ghostty の既知の割当 | 衝突の有無 |
| --- | --- | --- |
| Ctrl + Shift + E | デフォルト未割当 | なし（デフォルト） |
| Ctrl + Shift + C | コピー（`copy_to_clipboard`） | あり |
| Ctrl + Shift + K | デフォルト未割当（一部バージョンでスクロール操作） | 条件付きあり |
| Ctrl + Shift + F | デフォルト未割当 | なし（デフォルト） |
| Ctrl + Shift + G | デフォルト未割当 | なし（デフォルト） |
| Ctrl + Shift + S | デフォルト未割当 | なし（デフォルト） |

Ghostty は Ctrl + Shift + C をコピーとして使うケースがある。ただし IMK のキーイベントは composition が存在する場合に InputController が先行して消費するため、composing 中であれば Ghostty のコピーが起動しない。composition がない idle 状態では Ctrl + Shift + C を通すため、コピーは正常に機能する。

### 2-4. VS Code 統合ターミナル

VS Code は Ctrl + Shift 修飾を多用するエディタである。統合ターミナルは InputController を経由しない（VS Code は独自の IME 統合を持つ）ため、Koto の InputController が VS Code 統合ターミナルのキーイベントを受け取るかどうかは実装依存である。ただし、エディタ本体（ターミナル外のエディタペイン）で Koto を使う場合は衝突が問題になる。

| ショートカット | VS Code の既知の割当 | 衝突の有無 |
| --- | --- | --- |
| Ctrl + Shift + E | サイドバー: エクスプローラー（Explorer）表示切替 | あり（composition 外） |
| Ctrl + Shift + C | ターミナル: 新しい外部ターミナルを開く | あり（composition 外） |
| Ctrl + Shift + K | エディタ: 行を削除（Delete Line） | あり（composition 外） |
| Ctrl + Shift + F | サイドバー: 検索（Search）表示切替 | あり（composition 外） |
| Ctrl + Shift + G | サイドバー: ソース管理（Source Control）表示切替 | あり（composition 外） |
| Ctrl + Shift + S | ファイル: 名前を付けて保存（Save As） | あり（composition 外） |

特に Ctrl + Shift + E / F / G はサイドバー切替として広く知られる VS Code の代表的なショートカットである。ユーザーが日常的に使うキーと重なる。

**回避設計の評価**: 仕様 5「composition が無いときは Ctrl + Shift + 言語キーを消費しない（ターミナルのショートカットを奪わない）」により、idle 状態では全キーをホストアプリへ通すため、VS Code 側のショートカットは composition なしの状態では正常に動作する。composition 中（composing / converting / converted / failed）はキーを消費するが、その状態ではユーザーは文字入力に集中しており、サイドバー切替を同時に行う頻度は低い。したがって、**「composition 中のみ消費する」設計で VS Code の衝突は実用上回避できる**と評価する。

### 2-5. tmux

| ショートカット | tmux の既知の割当 | 衝突の有無 |
| --- | --- | --- |
| Ctrl + Shift + E | なし（tmux プレフィックスは Ctrl + B がデフォルト） | なし |
| Ctrl + Shift + C | なし | なし |
| Ctrl + Shift + K | なし | なし |
| Ctrl + Shift + F | なし | なし |
| Ctrl + Shift + G | なし | なし |
| Ctrl + Shift + S | なし | なし |

tmux はデフォルトプレフィックスが Ctrl + B であり、Ctrl + Shift 修飾を直接使わない。ユーザーがカスタム設定している場合はあるが、デフォルトでは衝突なし。

### 2-6. 衝突まとめと設計方針

| ターミナル | 衝突するキー | composition 中のみ消費で回避可能か |
| --- | --- | --- |
| Terminal.app | なし | - |
| iTerm2 | なし（デフォルト） | - |
| Ghostty | Ctrl + Shift + C（コピー） | 可能（idle では通す） |
| VS Code 統合ターミナル | E / C / K / F / G / S すべて（エディタ側） | 可能（idle では通す） |
| tmux | なし（デフォルト） | - |

結論: **すべての既知の衝突は「composition 中のみ消費する」設計で回避可能である**。idle 状態ではキーを必ず通すため、VS Code サイドバー切替（Ctrl + Shift + E/F/G）や Ghostty のコピー（Ctrl + Shift + C）等は Koto を入力ソースとして選択している状態でも通常どおり機能する。

実機検証時は特に VS Code 統合ターミナルと Ghostty で Ctrl + Shift + C の動作を確認し、`docs/terminal-compatibility.md` に結果を追記すること。

## 3. 変換中・失敗時の表示仕様

### 3-1. 基本方針

翻訳変換においても表示挙動を現行から変更しない（仕様の非機能要件「アクセシビリティ: 表示挙動は現状から変更しない」）。ローマ字かな変換層設計で確立した下線スタイル 2 値ルールをそのまま踏襲する。

| 状態 | marked text の内容 | 下線スタイル | 本機能での変更 |
| --- | --- | --- | --- |
| idle | なし | - | 変更なし。 |
| composing | 打鍵したローマ字そのまま | 細（single） | 変更なし。 |
| converting（翻訳中） | 変換要求時のローマ字スナップショットのまま | 細（single） | 変更なし。英語変換中も表示テキストはローマ字のまま。 |
| converted（翻訳後） | モデル出力（訳文、validator 通過済み） | 太（thick） | テキスト内容のみが変わる。スタイルは既存ルールと同じ。 |
| failed（翻訳失敗） | ローマ字スナップショットのまま（訳文は表示しない） | 細（single） | 変更なし。 |
| Escape 復元後 | sourceText（元のローマ字） | 細（single） | 変更なし。 |

### 3-2. 変換中（converting）の挙動

- `Ctrl + Shift + E` を押した瞬間、displayedText は現在の composing テキスト（ローマ字）のままで相変わらず細下線で表示される。
- かな正規化（`RomajiKanaConverter.normalize`）は同期・即時実行されるが、変換中の表示テキストには反映しない（ローマ字かな変換層設計と同様の設計判断）。
- モデルへは `modelInputText`（かな化済み）が送られるが、ユーザーからは不可視である。
- 変換完了まで「静止した細下線マーカー」のみが変換進行中を示す。スピナーや通知は追加しない。
- 変換中でも編集・Escape・Enter・Ctrl + C はすべて受け付ける（ユーザーがロックされない）。

### 3-3. 失敗時（failed）の挙動

- 訳文を表示しない。ローマ字スナップショット（sourceText と同一）を細下線で保持する。
- テキストは一切失われない（既存の最優先保証と同じ）。
- 失敗時と composing 時は視覚上同一になる（エラーメッセージ表示チャネルが無い）。この既知の制約は変更しない（現行と同じ）。
- 翻訳失敗後の回復経路。
  1. 同じ言語キーを再押下する: 同じテキストで再試行する。モデル準備中の場合の主経路。
  2. 別の言語キーを押下する: 別言語で変換を試みる。
  3. Shift + Space を押下する: 日本語変換で再試行する。
  4. Enter を押下する: ローマ字のまま commit する。
  5. Escape を押下する: sourceText へ復元して composing へ戻る。

## 4. エラー状態の挙動

翻訳変換で発生しうるエラーを系統別に整理する。日本語変換と同じ `KotoError` / `CompositionPhase.failed` を使い、エラー系統固有の追加 UI は持たない。

### 4-1. モデル利用不可

- 発生条件: `LanguageModelSession` の生成または推論呼び出しで Apple Intelligence が利用不可（設定無効・デバイス非対応・準備未完了）。
- 検出: `AppleFoundationModelsProvider` が `KotoError.modelUnavailable` を返す。
- `CompositionPhase` は `.failed(message: error.userMessage)` へ遷移する。
- 表示: ローマ字スナップショットを細下線で保持（テキスト不変）。
- 回復: 上記 3-3 の回復経路と同じ。
- 6 言語すべてで同条件で発生するため、言語ごとの個別処理はしない。

### 4-2. 検証拒否（ConversionOutputValidator による拒否）

- 発生条件:
  - 出力が空（空文字列・空白のみ）。
  - 膨張率が閾値を超える（極端に長い出力）。
  - 保護語・頭字語が消失している。
  - 翻訳固有: 日本語固有の unwrap 処理（末尾句点・鉤括弧）は `.japanese` のみに適用し、他言語には適用しない。
- 検出: `ConversionOutputValidator` が出力を拒否し `KotoError.invalidOutput` を返す。
- `CompositionPhase` は `.failed(message: error.userMessage)` へ遷移する。
- 表示: ローマ字スナップショットを保持。
- 回復: 3-3 の回復経路と同じ。温度 0.8 で再試行（Ctrl + Shift + 同言語キーの 2 回目以降）するか、別言語を試みるか、Shift + Space で日本語変換へ戻す。

### 4-3. 空出力

- 発生条件: モデルが空文字列または空白のみを返した場合。
- 空出力は `ConversionOutputValidator` で検出し、「検証拒否」として `.failed` へ遷移する。
- 翻訳対象言語に関わらず、空または空白のみの訳文を確定することはない。
- 入力テキスト自体が空白のみの場合は変換要求前に `noop` として処理し、`failed` にすらならない（既存挙動と同じ）。

### 4-4. セッションキャッシュ失敗

- `AppleFoundationModelsProvider` は target をキーにセッションキャッシュを持つ（仕様の技術設計）。セッション生成失敗（初回アクセス時）は `KotoError.modelUnavailable` として扱う。
- キャッシュされたセッションが無効化された場合（Apple Intelligence 設定変更等）も同様に `modelUnavailable` で失敗させる。
- 再試行時はキャッシュを無効化して再生成を試みることが望ましいが、当該ポリシーは `AppleFoundationModelsProvider` 内の実装詳細とし、UI / 状態機械には影響させない。

## 5. アクセシビリティ確認

### 5-1. 表示挙動を変更しないことの明記

本機能はキールーティングの追加とプロンプト・検証の言語分岐のみであり、`InputController.render` の実装を変更しない。具体的には以下が現行から変わらない。

- `NSUnderlineStyle.single`（細下線）と `NSUnderlineStyle.thick`（太下線）の割り当て基準（converted のみ太、それ以外は細）。
- `setMarkedText` の呼び出し経路。
- `insertText` によるコミット経路。
- marked text のカーソル位置の扱い。

翻訳後の変換結果（訳文）が converted 状態として表示されることで、ローマ字かな変換層設計と同様に「テキスト内容が置き換わること」が converted の第一シグナルになる。

### 5-2. ローマ字かな変換層設計からの継承

ローマ字かな変換層の設計（[`docs/specs/ローマ字かな変換層-design.md`](./ローマ字かな変換層-design.md) セクション 4）で確認した達成基準の評価は、翻訳においてもそのまま成立する。後退は発生しない。

| 達成基準 | 翻訳機能での位置付け |
| --- | --- |
| 1.4.1 色の使用 | 状態の区別に色を使わない。変更なし。 |
| 1.4.3 / 1.4.11 コントラスト | ホストアプリ依存。変更なし。 |

### 5-3. 既知の制約（変更なし）

以下はローマ字かな変換層設計から引き継ぐ既知の制約であり、本機能では解消しない。

- failed と composing の視覚的区別が不能（エラーメッセージ表示チャネルが無い）。翻訳失敗時も同様。
- converting 中の進行フィードバックが無い（無反応との区別が付かない）。翻訳変換中も同様。
- 下線スタイルの変化（細 → 太）は読み上げでは伝わらない。converted の通知はテキスト内容の変化に依存する。

## 既知の制約とフォローアップ候補

本機能のスコープでは解決しない事項。

- VS Code 統合ターミナルで composition 中に誤って Ctrl + Shift + F 等を押した場合、サイドバーではなく翻訳変換が起動する。構造上の回避策はなく、「composition 中はエディタショートカットより翻訳が優先される」ことをドキュメントに明記することで対処する。
- ターミナル側でユーザーが Ctrl + Shift + E 等をカスタム設定している場合は idle 状態で正常に動作するが、composing 中は翻訳が優先される。
- 翻訳中と日本語変換中の進行フィードバックは同様に無い（既知の制約の継承）。

これらは `/follow-up add` で記録し、別 Issue・別 PR で扱う。
