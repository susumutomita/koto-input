## 1. ユーザーストーリーの詳細化

### 主ペルソナ

ターミナル（Apple Terminal / Ghostty / iTerm2）で Claude Code や Codex CLI に日本語プロンプトを書く開発者。特徴は以下のとおり。

- 英語キーボードでローマ字入力し、Shift + Space で一括変換する。
- プロンプトには `git` などのコマンド名、`Claude Code` などの製品名、ファイルパスや識別子が日常的に混在する。
- 変換結果が意図と違ったら Escape で元のローマ字に戻し、打ち直すか原文のまま送る。
- 入力内容には業務上の機密が含まれうるため、テキストがデバイス外へ出ないことを前提に使う。

### 利用文脈ごとのストーリー

| ID | ストーリー | 対応する受け入れ基準 |
|---|---|---|
| US-1 | 開発者として、区切りなしのローマ字文（例: `kyouhaiihida`）を Shift + Space で正しい日本語（「今日はいい日だ」相当）に変換したい。なぜなら現状はモデルがローマ字の分かち書きに失敗し「きょうかいいひだ」等へ誤変換されるからである。 | AC-2、AC-6 |
| US-2 | 開発者として、英語コマンド・識別子（`git`、`npx` 等）が混在した文を変換しても、それらの語が原文のまま残ってほしい。なぜならコマンド名がかな化・漢字化されるとプロンプトとして壊れるからである。 | AC-3 |
| US-3 | 開発者として、protectedTerms に登録した固有名詞（`Codex` 等）は、たとえローマ字として読める綴りでも変換されずに残ってほしい。なぜなら製品名の表記が崩れると意図が伝わらないからである。 | AC-1（protectedTerms 原文維持） |
| US-4 | 開発者として、`.` と `,` を含む文を変換したとき句読点「。」「、」として扱われてほしい。なぜなら文の区切りまで含めて日本語プロンプトとして自然にしたいからである。 | AC-1（かな表の網羅） |
| US-5 | 開発者として、変換結果が気に入らないとき Escape で元のローマ字テキストへ正確に戻りたい。なぜなら打ち直しや原文のままの送信を素早く選びたいからである。 | AC-5 |
| US-6 | 開発者として、かな変換層の追加によって Shift + Space の体感が遅くなってほしくない。なぜならターミナルでの入力リズムを崩したくないからである。 | 非機能要件（O(n) 純関数） |
| US-7 | 開発者として、入力テキストがログやデバイス外に出ない現行方針が維持されてほしい。なぜならプロンプトに機密情報を含めることがあるからである。 | 非機能要件（ADR-0002 維持） |
| US-8 | 開発者として、入力中の表示はこれまでどおりローマ字のままで構わない。なぜなら打鍵中の表示変化はターミナルの行レンダリングを乱すリスクがあるからである。 | 振る舞い 1（現状維持）、スコープ外（ライブ変換） |

## 2. 受け入れ基準のテストシナリオ化

仕様書「受け入れ基準」の各項目を AC-1〜AC-7 とし、前提・操作・期待結果の形式に展開する。AC-1〜AC-4 は `swift test --package-path KotoInput` で自動検証、AC-5 は既存テストの回帰確認、AC-6・AC-7 は実機手動検証である。

### AC-1: RomajiKanaConverter 単体テストの網羅

すべて前提は「`RomajiKanaConverter` に入力文字列と protectedTerms を与える（純粋関数呼び出し）」、操作は「変換を実行する」である。期待結果のみ表に示す。

| シナリオ | 入力例 | 期待結果 |
|---|---|---|
| 基本五十音 | `aiueo`、`kakikukeko` | `あいうえお`、`かきくけこ` |
| 濁音 | `gagigugego`、`dadidudedo` | `がぎぐげご`、`だぢづでど` |
| 半濁音 | `papipupepo` | `ぱぴぷぺぽ` |
| ヘボン式・訓令式の両受理 | `shi` / `si`、`tsu` / `tu`、`fu` / `hu` | いずれも `し`、`つ`、`ふ` |
| 拗音 | `kya`、`sha` / `sya`、`cha` / `tya` | `きゃ`、`しゃ`、`ちゃ` |
| 促音（子音重ね） | `gakki`、`zasshi` | `がっき`、`ざっし` |
| 小書き | `xtu` / `ltu`、`xya` / `lya` | `っ`、`ゃ` |
| 撥音（子音前・語末・nn） | `kantan`、`hon`、`konnichiha` | `かんたん`、`ほん`、`こんにちは` |
| 長音記号 | `ko-hi-` | `こーひー` |
| 句読点 | `souda.`、`iie,demo` | `そうだ。`、`いいえ、でも` |
| 完全変換不能語（子音余り） | `git` | `git`（原文維持） |
| 完全変換不能語（大文字混在） | `Claude` | `Claude`（原文維持） |
| 完全変換不能語（数字・記号混在） | `v2`、`docs/specs` | 原文維持 |
| protectedTerms 原文維持 | protectedTerms に `sudo` を登録し `sudo` を入力 | `sudo`（`すど` へ変換可能だが保護語のため原文維持） |
| 空ではない混在文 | `kore wo git de kanri` | `これ を git で かんり`（語単位の判定、空白保持） |

補足: `konnichiha` は「`nn` → ん」を貪欲に適用すると「こんいちは」になり一般的な IME と乖離するため、解析規則を固定する基準シナリオとして必須にする（後述の指摘 5 を参照）。

### AC-2: 実測誤変換ケースの決定的変換

- 前提: `RomajiKanaConverter` を直接呼び出す単体テスト。
- 操作: `kyouhaiihida` を変換する。
- 期待結果: `きょうはいいひだ` が返る。モデルを介さない決定的な結果である。

### AC-3: 英語混在文の語単位判定

- 前提: `RomajiKanaConverter` を直接呼び出す単体テスト。protectedTerms は既定値。
- 操作: `Claude Code wo tukau to git mo benri` を変換する。
- 期待結果: `Claude Code をつかう と git も べんり` が返る。`Claude`・`Code`・`git` は原文維持、`wo`・`tukau`・`to`・`mo`・`benri` はかな化され、語間の空白が保持される。
- 注記: この期待結果はモデルへ渡す中間テキストであり、ユーザーが最終的に見る変換結果（モデルが整文した「Claude Code を使うと git も便利」相当）ではない（後述の指摘 6 を参照）。

### AC-4: モデルへ渡る sourceText のかな化検証

- 前提: ADR-0004 の ScriptedConversionProvider を provider として `CompositionCoordinator` を構築し、受信した `ConversionRequest` を記録できるようにする。composition にローマ字（例: `kyouhaiihida`）を入力済み。
- 操作: `requestConversion` コマンド（Shift + Space 相当）を発行し、変換タスクの完了を待つ。
- 期待結果: provider が受け取った `ConversionRequest.sourceText` が `きょうはいいひだ` である。また `CompositionState.sourceText`（Escape 復元用スナップショット）と表示中の composition はローマ字のまま変わらない。
- 注記: ScriptedConversionProvider の検証対象は「Coordinator がかな化済みテキストを渡すこと」という統合配線であり、かな変換の品質検証は AC-1〜AC-3 の純粋関数テストが担う（ADR-0004 の適用範囲と整合）。

### AC-5: Escape 復元の回帰なし

- 前提: AC-4 と同じ構成。ローマ字を入力し変換を要求して `converted`（または `converting` / `failed`）状態にある。
- 操作: `restoreSource` コマンド（Escape 相当）を発行する。
- 期待結果: `displayedText` が元のローマ字テキストへ戻る。かな化されたテキストや変換結果が復元先にならない。既存の `CompositionTransitionTests` / `CompositionCoordinatorTests` の該当テストがすべて Green のまま。

### AC-6: 実機確認（手動）

- 前提: macOS 26 / Apple Silicon / Apple Intelligence 有効の実機。Koto を入力ソースとして選択し、ターミナルを開く。
- 操作: `kyouhaiihida` と入力し Shift + Space を押す。
- 期待結果: 「今日はいい日だ」相当の変換が表示される。「きょうかいいひだ」「京橋駅」のような分かち書き失敗由来の誤変換が起きない。判定基準は「ひらがな列『きょうはいいひだ』の意味を保った漢字かな交じり文であること」とする（モデル出力は非決定的なため、文体差は許容する）。

### AC-7: ターミナル E2E 確認とドキュメント追記

- 前提: AC-6 と同じ実機。`docs/terminal-compatibility.md` のマトリクス（Apple Terminal / Ghostty / iTerm2 × Claude Code / Codex CLI）を対象とする。
- 操作: 各組み合わせで同ドキュメントの検証手順（marked text 表示、変換が送信を起こさない、commit と送信の分離、Escape 復元、Unicode 耐性、カーソル移動、入力ソース切替、変換中のアプリ切替）に加え、ローマ字混在文の変換品質を確認する。
- 期待結果: 各組み合わせの結果（検証日・状態・備考）が `docs/terminal-compatibility.md` のマトリクスへ追記されている。実施できない組み合わせは「未検証」のまま残してよいが、最低 1 組み合わせは検証済みになる。
- 注記: 同ドキュメントの例文 `kono authentication no sekinin han'i ga aimai` にはアポストロフィが含まれ、本仕様の規則では `han'i` が原文維持になる（後述の指摘 3 を参照）。E2E 時の期待値を事前にすり合わせること。

## 3. 優先度と依存関係

### 実装順序（依存の流れ）

```
P0: RomajiKanaConverter 単体（AC-1, AC-2, AC-3）
      ↓ 依存
P0: CompositionCoordinator 統合（AC-4, AC-5）+ validator 基準・PromptBuilder 文言更新
      ↓ 依存
P1: 実機確認（AC-6）
      ↓ 依存
P1: ターミナル E2E + docs/terminal-compatibility.md 追記（AC-7）
```

| 優先度 | 作業 | 依存 | 根拠 |
|---|---|---|---|
| P0 | `RomajiKanaConverter`（KotoCore 内の純粋関数 enum）と単体テスト | なし | 本機能の価値の源泉。純粋関数なので CI で完結し、TDD の起点になる。 |
| P0 | `CompositionCoordinator.startConversion` での統合、`ConversionOutputValidator.validate` へのかな化後テキスト適用、`PromptBuilder` instructions の文言更新、scripted provider による検証 | Converter 完成 | Converter が存在しないと統合点を書けない。Escape 回帰（AC-5）はこの統合の副作用検証なので同フェーズで行う。 |
| P1 | 実機確認（AC-6） | 統合完了 + Apple Intelligence 有効の実機 | CI では検証不能（ADR-0004 の前提）。コード完成後でないと意味がない。 |
| P1 | ターミナル E2E と互換性ドキュメント追記（AC-7） | 実機確認の環境が整っていること | 実機確認と同一環境・同一セッションでまとめて実施するのが効率的。マトリクス全埋めは必須としない（未検証残置を許容する既存運用に従う）。 |

### リリース判定

- P0 がすべて Green であることが PR マージの必須条件（CI で機械検証可能な範囲）。
- AC-6・AC-7 は実機でしか検証できないため、PR 時点で未実施の場合はチェックリストとして PR 本文に残し、マージ前に実施結果を記録する。実施不能な組み合わせは `docs/terminal-compatibility.md` の「未検証」運用に従う。

## 4. スコープ外の明確化と将来条件

| スコープ外項目 | 今回外す理由 | 将来必要になる条件（再検討トリガー） |
|---|---|---|
| 打鍵と同時のかな表示（ライブ変換） | marked text の毎打鍵更新はターミナルの行レンダリング互換性リスクが高く、現行の「入力中はローマ字表示」で課題（変換品質）は解決できるため。 | (1) 入力中にかなを確認したいという利用者フィードバックが実際に出ること、(2) `docs/terminal-compatibility.md` の対象ターミナルで毎打鍵の setMarkedText が安定すると検証できること。両方を満たしたら別仕様書を起こす。 |
| カタカナ語の明示変換・変換候補 UI | カタカナ化はモデルの整文に委ねられる。候補 UI は MVP の「下線のみ」表示方針（terminal-compatibility.md 既知の制限）と矛盾する。 | モデルがカタカナ語を安定して出せない実測が蓄積する、または第一候補のみでは満足できないという要望が出ること。表示方針の変更になるため ADR での意思決定を伴う。 |
| ローマ字テーブルのユーザーカスタマイズ設定 | 一般的な IME 互換テーブルで主ペルソナの打鍵流儀をカバーできる想定のため。設定画面・永続化の追加コストに対して現時点の価値が薄い。 | AZIK 等の拡張ローマ字配列や独自綴りの要望が具体的に出ること。`SettingsRepository` / `ConversionSettings` の拡張と設定 UI が前提になる。 |
| 完全変換不能語が混在する場合のスペルミス補正 | 「原文維持」は決定的で安全な挙動であり、補正は誤爆（正しい英語コマンドの書き換え）のリスクを持ち込むため。 | 原文維持された打ち間違い語（例: `tukau` を `tukua` と打つ）が変換品質を体感的に下げているという実測・フィードバックが出ること。補正は決定的アルゴリズムかモデル委譲かの設計判断を伴うため ADR を起こす。 |

## 5. 仕様書との矛盾・曖昧さ（PM 指摘事項）

実装着手前に Developer / QA と解消方針を合意すべき点。いずれも仕様書本文の意図（実測誤変換の解消）を覆すものではない。

1. **protectedTerms の「語単位の完全一致」と複数語保護語の不整合**。仕様書は「語単位の完全一致（大文字小文字を区別）」とするが、`ConversionSettings.defaultProtectedTerms` には空白を含む `Claude Code` がある。空白区切りの語単位照合では複数語の保護語は永遠に一致しない。受け入れ基準の例では `Claude` と `Code` がそれぞれ大文字を含み完全変換不能なので結果は偶然一致するが、保護語ロジックとしては仕様どおりに動かない。語単位照合に加えてフレーズ（複数語）一致を扱うのか、複数語保護語は照合対象外と割り切るのかを決める必要がある。
2. **大文字英字を含む語の扱いが未明記**。仕様書は「数字や他の記号を含む語は完全変換不能として原文維持」とするが、大文字英字には触れていない。`Code` を小文字化して照合すると `code` → `こで` と完全変換できてしまい、受け入れ基準の例（`Code` 原文維持）と矛盾する。「変換表は小文字のみ受理し、大文字英字を含む語は完全変換不能」という規則を明文化すべき。
3. **アポストロフィ `'` の扱い**。一般的な IME では `n'` は撥音の区切り（`han'i` → はんい）として受理されるが、仕様書のかな表に `'` の記載がなく、「他の記号を含む語は原文維持」規則に従うと `han'i` は原文のままになる。`docs/terminal-compatibility.md` の標準検証例文 `sekinin han'i` がこの語を含むため、AC-7 の期待値に直接影響する。「一般的な日本語 IME 互換」との整合をどちらに倒すか決定が必要。
4. **空白の保全規則が未規定**。「空白区切りの語ごとに判定」とあるが、連続スペース・タブ・改行（Control + Enter で composition に改行が入る既知挙動）を分割・再結合時に原文どおり保持するかが書かれていない。Unicode 耐性（複数行が壊れない）の E2E 期待動作と整合させるため、「語間の空白文字列は原文を厳密に保持する」ことを明記すべき。
5. **撥音の解析規則の細部**。「`n` / `nn` → ん」を字面どおり貪欲に適用すると `konnichiha` が「こんいちは」になり一般的な IME（「こんにちは」）と乖離する。「子音（`y` を除く）の前および語末の `n` は ん、母音・`y` の前は次のかなの一部」という規則をテストで固定する必要がある（AC-1 のシナリオに `konnichiha` を含めたのはこのため）。
6. **受け入れ基準の例が中間テキストであることの明示不足**。`Claude Code をつかう と git も べんり` は RomajiKanaConverter の出力（モデルへ渡す中間テキスト）であり、ユーザーが見る最終結果ではない。仕様書の文脈からは読み取れるが、QA がユーザー可視出力として E2E の期待値に使うと誤判定するため、本書 AC-3 の注記で区別した。
7. **実機確認の合否基準の曖昧さ（軽微）**。AC-6 の「『今日はいい日だ』相当」はモデル出力が非決定的な前提では妥当な書き方だが、合否判定者によるブレを防ぐため、本書では「ひらがな列の意味を保った漢字かな交じり文であること」を判定基準として補った。
