# 多言語品質フィクスチャ 仕様書

## 概要

セマンティック多言語出力モード（ADR-0010、Issue 36）の品質を、ゴールデン一致ではなくフィクスチャ契約で評価する仕組み。フィクスチャは `KotoInput/Tests/KotoCoreTests/Fixtures/multilingual-quality.json` に置き、`MultilingualQualityFixtureTests` がモデルを呼び出さずにスキーマと契約を機械検証する。実モデルの出力品質を人間が評価するときの参考データ（acceptableOutputs）も兼ねる。

## スキーマ

フィクスチャは以下のフィールドを持つオブジェクトの JSON 配列。

| フィールド | 型 | 必須 | 意味 |
| --- | --- | --- | --- |
| `id` | string | 必須 | ケースの一意な識別子。`<targetLanguage>-<内容>` の形式を推奨する。 |
| `source` | string | 必須 | ユーザーが打つ元テキスト（ローマ字・混在日本語）。実在の私的テキストは使わない。 |
| `targetLanguage` | string | 必須 | `ConversionTarget.localeIdentifier` と一致する BCP 47 識別子（ja / en / zh-Hans / ko / fr / de / es / ar）。 |
| `profile` | string | 必須 | `OutputProfile` の rawValue（neutral / polite / business / casual / technical）。 |
| `acceptableOutputs` | string[] | 必須（1 件以上） | 許容できる出力の例。人間評価の参考であり、完全一致は要求しない。 |
| `rejectedOutputs` | string[] | 任意 | validator が必ず拒否すべき出力（保護語消失・空出力等）。 |
| `mustPreserve` | string[] | 必須 | 出力に原文どおり残るべき保護語・識別子。`source` に原文どおり含まれていなければならない。 |
| `mustNotAdd` | string[] | 必須 | 入力に無い断定として出力に混入してはならない語。大文字小文字を区別せずに照合する。 |

## 機械検証する契約

`MultilingualQualityFixtureTests` は `Bundle.module` からフィクスチャを decode し、各ケースについて以下を検証する。

1. `targetLanguage` が `ConversionTarget` の `localeIdentifier` に解決できる。
2. `profile` が `OutputProfile` に解決できる。
3. `mustPreserve` の各語が `source` に原文どおり含まれる（保護語検証が必ず束縛される）。
4. 各 `acceptableOutputs` が `mustPreserve` をすべて含み、`mustNotAdd` をひとつも含まない。
5. 各 `acceptableOutputs` が、`mustPreserve` を `protectedTerms` に設定した `ConversionOutputValidator` を通る。
6. 各 `rejectedOutputs` が同じ validator で必ず失敗する。

加えてコーパス全体として、10 件以上・id の一意性・日本語（ja）と英語（en）の収録・キー割当の無いアラビア語（ar）2 件以上を要求する。

## ケースの追加手順

1. `multilingual-quality.json` にオブジェクトを追記する。`id` は既存と重複させない。
2. `mustPreserve` の語が `source` に原文どおり含まれることを確認する（含まれない語は validator の検証対象にならない）。
3. `acceptableOutputs` は忠実な訳・整文のみとする。入力に無い情報・文末句読点を足した例を許容例にしない（Issue 22 の教訓）。
4. 保護語消失のような「必ず拒否すべき出力」を再現したい場合は `rejectedOutputs` に入れる。
5. `swift test --package-path KotoInput` で `MultilingualQualityFixtureTests` が Green になることを確認する（toolchain の無い環境では CI の swift ジョブが代行）。

## 評価方針

- 自動テストはモデルを呼び出さず、スキーマと契約（mustPreserve の残存・mustNotAdd の不在・validator の通過と拒否）だけを決定論的に検証する。
- 実モデルの出力品質は、実機で `source` を打って得た出力を `acceptableOutputs` と見比べて人間が評価する。乖離が大きいケースはプロンプト（`PromptBuilder`）の改善か、2 段生成の再検討（ADR-0010 の Tradeoff）の根拠とする。
- フィクスチャはプライバシー制約（ADR-0010 の決定 9）に従い、実在の私的テキスト・個人情報を含めない。
