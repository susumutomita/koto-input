# セッション内文脈メモリ ユーザーフィードバック

- 対象仕様: [docs/specs/2026-06-11-セッション内文脈メモリ.md](./2026-06-11-セッション内文脈メモリ.md)（承認済み、これを正とする）。
- 設計判断の正本: [ADR-0013](../adr/0013-ローカル文脈メモリは-opt-in-のセッション内記憶から始める.md)。
- 担当 Issue: <https://github.com/susumutomita/koto-input/issues/46>。
- ロール: User（実ユーザーペルソナによるシミュレーション）。
- 前提: シミュレーション時点で実装は未完了。挙動はすべて仕様書・ADR-0013・既存 InputController の記述から決定的に追跡した。
- 確定事項の扱い: トリガー = Ctrl + Shift + Space、保持 5 件 / 500 文字、既定 OFF、消去はプロセス終了と OFF 切替のみ、push 型提案なし、設定 UI なし、はこのフィードバックで覆さない。これらへの不満は第 4 節で「将来への送り先」として整理した。

ペルソナは 3 人を使い分ける。

- ペルソナ A: Ghostty / iTerm2 で Claude Code に指示を書く日本人エンジニア。ターミナルを離れずに issue 説明・指示・確認をすべて打つ。
- ペルソナ B: Slack で依頼を書き、直後にエディタへ移って詳細をまとめるテックリード。アプリをまたぐ作業が多い。
- ペルソナ C: プライバシー重視のセキュリティエンジニア。新機能はまず「何が収集され、いつ消えるか」を確認してから使う。

## 1. 主要シナリオのウォークスルー

### シナリオ 1（ペルソナ A）: issue 説明を書いた直後に「arewoyatteoite」を文脈つきで展開する（機能の核心）

- 事前準備:
  1. README の設定節に従い、ターミナルで JSON 全体を書き直す。

     ```bash
     defaults write com.susumutomita.inputmethod.Koto conversionSettings -string \
       '{"style":"polite","customInstruction":"","protectedTerms":["Claude Code","Codex"],"maximumExpansionRatio":4,"contextMemoryEnabled":true}'
     ```

  2. 反映されたかどうかを確認する手段は無い（後述 2-1）。
- 操作（キー入力単位）:
  1. Ghostty で Claude Code を起動し、Koto を選択する。
  2. `nyuuryoku chuu no kouho hyouji ga chiratsuku fuguai wo naoshite hoshii` と打鍵する（下線付き composition）。
  3. `Shift + Space` を押す。無音の待ち時間の後、「入力中の候補表示がちらつく不具合を直してほしい。」（太下線）。
  4. `Enter` で確定する。設定が ON なので、確定テキストが文脈メモリの 1 件目として非同期に追記される。
  5. `Enter` をもう一度押して Claude Code へ送信する。
  6. 続けて `genin wa marked text no koushin junjo da to omou` → `Shift + Space` → 「原因は marked text の更新順序だと思う。」→ `Enter` → `Enter`。文脈 2 件目。
  7. Claude Code の応答を読み、しばらく別の調査を眺める（このあいだ何も打たなければ文脈は保持されたまま）。
  8. `arewoyatteoite` と打鍵する（composition: `arewoyatteoite`）。
  9. `Ctrl + Shift + Space` を押す。
  10. 無音の待ち時間の後、文脈つき候補「さっき報告した、入力中の候補表示がちらつく不具合の修正をやっておいて。」（太下線）が表示される。
  11. 比較したければ続けて `Shift + Space` を押す。通常変換「あれをやっておいて。」が候補に積まれ、`↑` / `↓` で見比べられる。
  12. `Enter` で確定し、`Enter` で送信する。
- 体感: 手順 10 が成功した瞬間の体験は強烈で、「IME が自分の作業を覚えている」という Koto にしかない価値を一発で理解できる。Shift + Space との操作対称性（同じ修飾系・同じ候補蓄積・同じ Escape 復元）も学習コストを下げている。
- 懸念点 1（無音の待ち時間）: 多言語変換キーのフィードバックで指摘した「変換中に画面が変化しない」問題がそのまま乗る。文脈つき変換はプロンプトが長くなるぶん体感の待ちも伸びる可能性があり、無音期間の不安は通常変換より大きい。
- 懸念点 2（膨張率検証との衝突）: 原文「あれをやっておいて」は仮名で 9 文字しかない。文脈展開の成果物は 30〜40 文字級の具体文になるのが本来の成功形で、これは `maximumExpansionRatio`（既定 4.0）× 原文長 + 固定許容を超えうる。仕様は「許容を超える長文出力は従来どおり拒否され、Escape で原文へ戻れる」と明記しているが、ユーザーから見えるのは「Ctrl + Shift + Space を押したのに何も変わらない」だけ。機能の目的（短い参照を長い具体文へ展開する）と検証器の前提（出力は原文の高々 4 倍）が構造的に衝突しており、しかも拒否は無音。このシナリオの成功率を実質的に決めるのはモデル品質ではなく膨張率上限になりかねない（2-6・第 4 節参照）。

### シナリオ 2（ペルソナ B）: Slack で書いた依頼をエディタで参照する（アプリ横断の価値）

- 操作:
  1. Slack のメッセージ欄で `asu no release note no shitagaki wo kyou juu ni junbi shite oite hoshii` と打ち、`Shift + Space` → 「明日のリリースノートの下書きを今日中に準備しておいてほしい。」→ `Enter`（確定 = 文脈へ追記）→ `Enter`（Slack へ送信）。
  2. `Cmd + Tab` でエディタへ移動する。composition が残っていれば deactivate 時に commit され、これも文脈へ追記される。
  3. エディタの TODO メモで `anoken no shousai wo koko ni matomeru` と打ち、`Ctrl + Shift + Space` を押す。
  4. 「Slack で頼んだリリースノート下書き準備の件の詳細をここにまとめる。」のような、依頼内容を踏まえた候補が表示される。
- 体感: 文脈ストアがプロセス全体で 1 つ（全アプリの入力が合流する）という設計のおかげで、「Slack で書いたことをエディタが知っている」というアプリ単位のツールには不可能な体験が成立する。これは IME という立ち位置の差別化として仕様どおり機能している。
- 懸念点: Slack では短い相づち（`ryoukai desu`、`arigatou gozaimasu` 等）も全部 Enter 確定するので、1 件ずつ文脈の枠を消費する。依頼文を書いてから相づちを 5 回打つと、肝心の依頼文が FIFO から流れている（シナリオ 5 参照）。「Slack で書いた依頼をエディタで参照する」というこのシナリオは、依頼直後に移動すれば成功し、雑談を数往復してから移動すると黙って失敗する。成否を分ける条件がユーザーから一切見えない。

### シナリオ 3（ペルソナ C）: 一度試して OFF に戻す（プライバシーユーザーの検収）

- 操作:
  1. README のプライバシー節と ADR-0013 を読み、「既定 OFF・デバイス外送信なし・ログ記録なし・再起動で消える」を確認してから ON にする（シナリオ 1 と同じ defaults write）。
  2. 試しに 2〜3 回変換・確定し、`anoken doushita` → `Ctrl + Shift + Space` で文脈つき候補が出ることを確認する。
  3. OFF に戻す。JSON 全体を `contextMemoryEnabled: false` で書き直すか、`defaults delete` で設定ごと消す。
  4. 次のインタラクション時点で保持済み文脈が全消去され、以後 Ctrl + Shift + Space は消費されずアプリへ通る。
- 体感: 「既定 OFF・オンデバイス・opt-in・OFF で全消去・再起動で消える」という束は、このペルソナが新機能に求める条件をほぼ満たしている。診断ログ（ADR-0002）と作業記憶を別概念として定義し直した点も、ドキュメントを読む側として信頼できる。
- 懸念点 1（消去の遅延と確認不能性）: 消去は「OFF へ切り替えた瞬間」ではなく「次のインタラクション時点」。in-memory なので実害は薄いが、「OFF にした、消えた」を確認する手段が何も無い。UI なし・ログなし（ADR-0002 で文脈テキストのログは禁止）なので、消去されたことは仕様を信じるしかない。
  - 懸念点 2（defaults delete の巻き添え）: `defaults delete` で OFF に戻すと `protectedTerms` など他の設定も消える。ON/OFF のたびに JSON 全体を書き直す運用は、このペルソナのように「使うときだけ ON にしたい」ユーザーと最も相性が悪い。
- 懸念点 3（ADR と第一版の落差）: ADR-0013 の Decision 7 にはアプリ別無効化・UI からの全消去・incognito・セキュアフィールド非収集が「プライバシーコントロール一覧」として並ぶが、第一版にあるのは全体 ON/OFF だけ。ADR を読んでから使うユーザーほど「一覧の残りはどこにあるのか」と探すことになる。README で第一版の範囲を明示してほしい。
- 懸念点 4（セキュアフィールドの限界）: 第一版は「IMK がパスワード欄で IME を無効化する既定動作」への依存で、セキュア指定されていない欄に打ったワンタイムコードや秘密情報は普通に収集される。仕様自身が README 明記を求めており、これは絶対に落とせない。

### シナリオ 4（ペルソナ A 派生）: 設定ミスと「OFF のとき何も起きない」が合流する最悪ケース

- 操作:
  1. README で機能を知り、defaults write で有効化したつもりが、キー名を `contextMemoryEnable` と typo する。
  2. decode は `decodeIfPresent ?? false` なので、エラーも警告も出ず黙って OFF のまま。
  3. composing 中に `Ctrl + Shift + Space` を押す。OFF なのでキーは消費されずアプリへ通る。
  4. Ghostty では何も起きない。JetBrains 系 IDE では `Ctrl + Shift + Space` が Smart Completion に割り当たっているため、補完ポップアップという無関係な動作が起きる。
- 体感: ユーザーには「機能が壊れている」「この環境では使えない」「設定が効いていない」の区別がつかない。正常な OFF・JSON の typo・アプリによるキー横取りの 3 つが、すべて同じ「変換が起きない」という見た目に縮退する。多言語変換キーのときの VS Code 衝突（既報のシナリオ 5）と同型の問題で、しかもこちらは「既定 OFF」なので新規ユーザー全員が最初にこの無反応を通る。
- 補足: OFF のときにキーを素通しする設計自体は正しい（既存キーと同じ原則で、ターミナルのショートカットを奪わない）。問題は素通しという正しい挙動が、ドキュメント無しでは故障と見分けられないこと。

### シナリオ 5（ペルソナ A）: 意図した文脈が 5 件 FIFO から流れているケース

- 操作:
  1. Claude Code に issue 説明（180 文字）を確定・送信する。文脈 1 件目。
  2. その後のやりとりで `y`、`susumete`、`OK desu`、`tsugi he`、`git status` のような短い確定を 5 回行う。Koto を選択したままなら、シェルコマンドや 1 文字の応答も Enter 確定のたびに 1 件として記録される。
  3. `arewoyatteoite` → `Ctrl + Shift + Space` を押す。
  4. [CONTEXT] に残っているのは直近 5 件の相づちとシェルコマンドだけで、「あれ」は解決されない。候補は通常変換とほぼ同じ「あれをやっておいて。」になる。
- 体感: 5 件の上限は「件数」ベースなので、確定 1 回 = 1 件と数える CLI 用途では数十秒で枠が回転する。500 文字の総量に余裕があっても件数 5 で頭打ちになり、長さ 1 文字の `y` と 180 文字の issue 説明が同じ 1 枠を占める。逆に 500 文字級の長文を 1 回確定すると、それまでの文脈が総文字数制限で一掃される。どちらの方向でも「さっき書いたのに覚えていない」が起き、しかも保持内容を確認する手段が無いので、ユーザーは失敗してから推測するしかない。
- 補足: N = 5 / 500 文字という値自体は確定事項なので覆さない。ここで求めたいのは、(1) この限界が README に書かれていること、(2) 文脈が空・無関係のときの挙動（通常変換と同等の候補が返る）が仕様どおり安全に倒れること、の 2 点。値の再検討と保持戦略の高度化（短い確定の除外等）は第 4 節で将来へ送る。

## 2. 「使いにくい」「わかりにくい」点

### 2-1. 発見可能性: 設定が UserDefaults の JSON 手編集（high）

機能の入口が「README を読んで JSON 文字列を defaults write で書き直す」しかない。具体的には以下が重なる。

- 既存設定を含む JSON 全体を書き直す必要があり、`protectedTerms` 等を消す事故が起きやすい。
- キー名を typo しても `decodeIfPresent ?? false` で黙って OFF になり、エラーが出ない（シナリオ 4）。
- ON になったかどうかを確認する手段が「実際に Ctrl + Shift + Space を押して候補が変わるか見る」しかなく、その確認自体が 2-3（空文脈では通常変換と同一）に阻まれる。

設定 UI なしは確定事項なので、第一版では README 側で吸収するしかない。最低限、(1) `contextMemoryEnabled` を含む完全な JSON 例、(2) `defaults read com.susumutomita.inputmethod.Koto conversionSettings` での確認手順、(3) typo は無警告で OFF になるという注意、の 3 点を README に載せてほしい。

### 2-2. OFF のとき Ctrl + Shift + Space が「何も起きない」（high）

既定 OFF の機能で、トリガーキーを押した結果が「無反応（アプリへ素通し）」なのは、新規ユーザー全員が通る最初の体験として分かりにくい。正常な OFF・設定の typo・アプリのキー横取り（JetBrains の Smart Completion 等）が同じ見た目に縮退する（シナリオ 4）。素通し設計自体は既存キーとの一貫性があり正しいので、対策はドキュメントに限られる。

- README のキー表で「設定 OFF のときはアプリへそのまま通す」を明記する。
- `docs/terminal-compatibility.md` に Ctrl + Shift + Space の既知衝突（JetBrains 系の Smart Completion 等）を追記する。多言語変換キーのときに VS Code の `Ctrl + Shift + E` 衝突を文書化要求したのと同じ扱い。

### 2-3. 文脈が空のとき通常変換と同一結果になる戸惑い（medium）

仕様では `contextEntries` が空なら prompt はバイト単位で従来と同一。つまり ON にした直後（まだ何も確定していない状態）に Ctrl + Shift + Space を試すと、Shift + Space と同じ候補が返る。機能を有効化した直後の「動作確認」がまさにこの状態で行われるため、「設定が効いていないのでは」という誤認を誘発する。2-1 の確認不能性と合成すると、ユーザーは ON / OFF どちらの状態にいるのかを判定できない。

さらに、Shift + Space と Ctrl + Shift + Space は attempt 同一性が（target, useContext）で別管理なので、空文脈で両方押すとどちらも attempt 0 の greedy 変換になり、同一文字列の候補が 2 件積まれる可能性がある。↑/↓ で巡回したとき同じ候補が 2 回出るのか、重複が排除されるのかが仕様から読み取れない。実装時に確認し、どちらの挙動かをテストで固定してほしい。

### 2-4. 5 件 / 500 文字で意図した文脈が流れている（high）

シナリオ 5 のとおり。件数ベース FIFO は CLI のような「短い確定が高頻度」の環境で意図した文脈をすぐ押し出し、逆に長文 1 件が総文字数制限で全部を押し出す。値そのものは確定事項なので、第一版では「限界の文書化」と「失敗時に安全（通常変換相当）に倒れること」を求め、保持戦略の改善（空白以外の最小長フィルタ・重み付け・working 層）は将来へ送る。なお仕様の「空白のみのテキストは無視」は正しい最低ラインだが、`y` 1 文字は通過する。

### 2-5. 文脈つき候補と通常候補を ↑/↓ で見比べる体験（medium）

候補蓄積に合流する設計（ADR-0012 拡張）のおかげで「文脈ありとなしを並べて選ぶ」操作自体は自然にできる。これは良い。一方で巡回中の候補には「どれが文脈を使った候補か」を示す手がかりが無い。文脈つき候補が期待外れだったとき、いま見ているのが文脈ありの再抽選なのか通常候補なのかを、ユーザーは押したキーの履歴を記憶して判別するしかない。既報（多言語変換キー 2-6「converted 状態の種類が見えない」）と同根で、言語に加えて useContext の軸が増えたぶん組み合わせが広がった。marked text のみという表示制約の中での解決は難しいため、第一版では諦めるが、将来の候補表示改善の検討材料として記録する。

### 2-6. 膨張率検証が機能の目的と衝突する（high）

シナリオ 1 懸念点 2 のとおり。「あれをやっておいて」（9 文字）を文脈で具体化した出力は容易に 4.0 倍を超え、無音で拒否される。検証器を変更しないことは仕様で明示されているので第一版のコードは仕様どおりでよいが、ユーザーがこの壁に当たることは確実なので、以下を求める。

- README に「文脈つき変換で長い具体文へ展開したい場合は `maximumExpansionRatio` を上げる」というガイドを書く（設定キーは既存。新規実装不要）。
- 品質フィクスチャの文脈ありケースに「原文が短く出力が長い」ペアを含め、既定の膨張率上限で成功するか・拒否されるかを実装時に把握する。
- 文脈変換時の許容再設計（例: 文脈ありのときの上限を別係数にする）は invariant に関わるため、ADR で将来判断する。

### 2-7. 消去タイミングと「消えたことの確認不能性」（medium）

OFF 切替後の消去が「次のインタラクション時点」である点、消去を確認する手段が無い点（シナリオ 3）。消去手段がプロセス終了と OFF 切替のみという点は確定事項なので、第一版では README に消去タイミングを正確に書く（「OFF にすると次の入力時に消える」「Mac の再起動・ログアウト・Koto プロセス終了でも消える」）ことを求める。「いま何を覚えているか」の確認手段と即時全消去操作は将来へ。

### 2-8. 「セッション」という言葉の境界が分かりにくい（low）

機能名の「セッション」がユーザー語彙では多義的（ログインセッション、ターミナルのセッション、アプリの起動単位）。実体は「Koto の入力メソッドプロセスの生存期間」で、アプリを切り替えても消えず（むしろそれが価値）、再ログインや Mac 再起動で消える。README ではプロセスやセッションという実装語彙を避け、「Mac を再起動すると消える」「アプリをまたいで覚えている」という観測可能な言葉で書いてほしい。

## 3. 受け入れ基準のユーザー視点での充足確認

仕様書の受け入れ基準を 1 つずつ、「ユーザーとして満足か」の観点で確認した。

1. **既定 OFF で従来挙動と完全一致。Ctrl + Shift + Space は消費されずアプリへ通り、収集ゼロ。既存テスト green。**
   - 評価: 満足。opt-in の約束をテストで固定する基準として必須で、ペルソナ C の信頼の土台。ただし「消費されず通る」が正しい挙動であることはテストで保証できても、ユーザーがそれを故障と誤認する問題（2-2）はこの基準では救えない。README 記載とセットで初めて充足とみなしたい。
2. **ON のとき、commit した非空テキストが直近 5 件・合計 500 文字（UTF-16 長）まで FIFO で保持される。**
   - 評価: 仕様としては満足、体験としては条件付き。保持規則は明確だが、ユーザーから保持内容が見えないため「覚えているはず」と「実際に覚えている」の差を確認できない（2-4・シナリオ 5）。Enter 確定だけでなく deactivate 由来の commit も対象になる点は、アプリ切替の多いペルソナ B にとって正しい挙動。
3. **ON かつ composing 中の Ctrl + Shift + Space で `[CONTEXT]` 付き日本語 AI 変換が実行され、候補蓄積に入り ↑/↓ で見比べられる。**
   - 評価: 満足。シナリオ 1 の核心の保証であり、既存の候補蓄積へ合流させる設計はユーザーの学習コストをほぼゼロにしている。空文脈時の候補重複の有無（2-3）をこの基準のテストケースに含めてほしい。
4. **ON→OFF で次のインタラクション時点に全消去、以後の注入・収集なし。**
   - 評価: 満足。ペルソナ C の検収項目そのもの。「追記時と読み出し時の両方で設定を確認し OFF なら clear」という二重の消去経路は安心材料。消去の遅延（2-7）は README に正確なタイミングが書かれることを充足条件としたい。
5. **Escape 復元・stale 結果拒否・保護語検証・膨張率検証は従来どおり機能する。**
   - 評価: 概ね満足だが、最大の懸念を含む。Escape でいつでも原文へ戻れることは文脈変換を「試して気に入らなければ捨てる」操作にしてくれる必須の安全網。一方、膨張率検証が従来どおりであることは、文脈展開の成功形（短い参照 → 長い具体文）を黙って拒否する構造的衝突を孕む（2-6）。基準として正しいが、フィクスチャでこの衝突が実際に起きる頻度を可視化してほしい。
6. **決定論変換（Tab）・通常 AI 変換（Shift + Space）のレイテンシに回帰がない。**
   - 評価: 満足。文脈メモリを ON にしても普段のタイピングと Tab の即時性が変わらないことは、「機能を入れたら全体が重くなった」という最悪の体験を防ぐ。メモリ書き込みを hot path から外す設計（ADR-0013 Decision 6）の検証として適切。
7. **カバレッジ 100 % を維持し、CI green。**
   - 評価: 必要条件。ユーザーから直接は見えないが、既存機能のリグレッション防止として当然の基準。

総評: 受け入れ基準は「opt-in の約束」「既存安全網の継承」「候補蓄積への合流」という機能の骨格を正しく押さえている。一方で、(1) ユーザーが機能の存在と ON / OFF 状態を知れるか（発見可能性）、(2) 文脈が流れた・空だったときに何が起きたか分かるか（失敗の透明性）、(3) 膨張率検証と文脈展開の衝突が実用上どの頻度で起きるか、に対応する基準が無い。1 と 2 は README 整備（仕様のドキュメント節に既定）を具体化することで、3 はフィクスチャ収録ケースの選定で、それぞれ第一版のうちに手当てできる。

### 文脈つき変換（セッション内文脈メモリ）

`contextMemoryEnabled` を true にすると、Koto は確定したテキストの直近 5 件（合計 500 文字まで）をメモリ上にだけ記憶する。composition 中に `Ctrl + Shift + Space` を押すと、その記憶を参照情報として使った日本語 AI 変換を実行し、「あれ」「例の件」のような省略表現を直前に書いた内容を踏まえた具体的な文へ展開する。候補は通常の変換候補と同じく `↑` / `↓` で見比べられる。

```bash
defaults write com.susumutomita.inputmethod.Koto conversionSettings -string \
  '{"style":"polite","customInstruction":"","protectedTerms":["Claude Code","Codex"],"maximumExpansionRatio":4,"contextMemoryEnabled":true}'

# 反映されているかの確認
defaults read com.susumutomita.inputmethod.Koto conversionSettings
```

- 既定は OFF。OFF のとき `Ctrl + Shift + Space` は Koto が消費せずアプリへそのまま通る。
- キー名の綴りを誤ると警告なしで OFF のままになる。`defaults read` で確認すること。
- 記憶が空のとき（ON にした直後など）は、`Shift + Space` の通常変換と同じ結果になる。
- 確定 1 回が 1 件として数えられる。短い確定（`y` など）も 1 件を消費するため、参照したい内容を書いてから時間が経つと記憶から流れていることがある。
- 短い入力を長い具体文へ展開すると出力長の上限（`maximumExpansionRatio`）で拒否されることがある。その場合は値を上げる。
- 記憶はこの Mac の外へ送信されず、ディスクにも書かれない。OFF に切り替えると次の入力時に全消去され、Mac の再起動でも消える。
- パスワード欄は macOS が IME を無効化するため記憶されないが、セキュア指定の無い入力欄に打った内容は記憶対象になる。秘密情報を扱う作業では OFF を推奨する。
- JetBrains 系 IDE など、`Ctrl + Shift + Space` をアプリ側が先取りする環境がある（[docs/terminal-compatibility.md](./terminal-compatibility.md) を参照）。
