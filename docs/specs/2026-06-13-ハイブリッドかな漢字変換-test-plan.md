## テスト方針

- TDD で進める。各テストを先に Red で書き、実装で Green にする。カバレッジ 100% を維持する。
- BDD スタイルで `@Test` / `@Suite` のタイトルを日本語で記述し、振る舞いを表現する。
- No Mock を守る。辞書は実 mozc 由来の小規模サブセット（`dictionary_oss`、BSD-3-Clause）を SwiftPM リソースとして同梱し、実 I/O・実データで検証する。スタブ辞書は使わない。
- AI（Apple FoundationModels）は CI ランナーで動かず非決定的なため、ADR-0004 に基づきテストターゲット限定の `ScriptedConversionProvider` で不可用・失敗・遅延を再現する。変換品質の検証には使わず、フォールバックや状態遷移の制御にのみ使う。
- 純粋ロジック（`DictionaryConverter`・`ConversionOutputValidator`）は同期の決定的テストで直接検証する。`HybridConversionProvider` の合成挙動はスクリプト化した AI と実辞書を組み合わせて検証する。

## テスト配置

- `KotoInput/Tests/KotoCoreTests/DictionaryConverterTests.swift`（新規・決定的ロジック）。
- `KotoInput/Tests/KotoCoreTests/ConversionOutputValidatorTests.swift`（拡張・no-op 検出）。
- `KotoInput/Tests/KotoCoreTests/HybridConversionProviderTests.swift`（新規・辞書 + スクリプト AI の合成）。
- `KotoInput/Tests/KotoCoreTests/CompositionCoordinatorTests.swift`（拡張・候補プールと巡回）。
- セキュリティ観点は静的検査と既存 invariant で担保し、専用テストを補助的に置く。

## 1. 正常系: 代表読みが辞書経由で決定的に漢字化される

辞書バックボーンが「正解の表記を候補集合に必ず含める」一次変換として機能し、代表的な読みが決定的に漢字化されることを確認する。`DictionaryConverter.convert(reading)` を実辞書サブセットに対して直接検証する。

- Given: 実 mozc サブセット辞書をロードした `DictionaryConverter`。
- When: 読み「ほうほう」「けっきょく」「かんじ」を変換する。
- Then: それぞれ「方法」「結局」「漢字」を単一最良表記として返し、全かな未変換にならない。

想定テスト名（`@Suite("DictionaryConverter の決定的なかな漢字変換")`）。

- `@Test("代表読み「ほうほう」が辞書経由で「方法」に決定的に変換される")`。
- `@Test("代表読み「けっきょく」が辞書経由で「結局」に決定的に変換される")`。
- `@Test("代表読み「かんじ」が辞書経由で「漢字」に決定的に変換される")`。
- `@Test("複数セグメントの読みを最長一致で分割し、各セグメントの最良表記を連結する")`。

加えて、合成経路で AI が辞書最良を素通しせず文脈整形しても受理されることを確認する。

- Given: 実辞書 + 整形済み出力を返すスクリプト AI を合成した `HybridConversionProvider`。
- When: 「ほうほう」を変換し、AI が「方法」を返す。
- Then: `ConversionOutputValidator` を通過し、確定値が「方法」になる。

想定テスト名（`@Suite("HybridConversionProvider のハイブリッド変換")`）。

- `@Test("辞書最良を AI が再ランクして返した出力が検証を通り確定する")`。

## 2. no-op 検出: 全かな未変換を ConversionOutputValidator が failure 化する

出力が読み（全かな）と実質同一の未変換である場合を `target` が `.japanese` のときに failure 扱いにし、既存の attempt 機構へ接続して自動エスカレーションさせる。検証ロジックは `ConversionOutputValidator.validate(output:source:settings:target:)` を直接検証し、エスカレーションは `CompositionCoordinator` 経由で検証する。

- Given: `ConversionSettings.default` と `target: .japanese`。
- When: 出力「ほうほう」を source「houhou」で検証する。
- Then: `.failure(.generationFailed(...))` を返し、メッセージに source / output の本文を混入させない（ADR-0010 のプライバシー制約を踏襲）。

想定テスト名（`ConversionOutputValidatorTests` へ追加）。

- `@Test("全かな未変換の日本語出力は no-op として failure になる")`。
- `@Test("漢字交じりに変換された出力は no-op 判定を受けず受理される")`。
- `@Test("一部だけ漢字化された出力は未変換とみなさず受理する")`。
- `@Test("英語ターゲットでは全かな出力でも no-op 判定を適用しない")`。
- `@Test("no-op の failure メッセージに source / output の本文を混入させない")`。

エスカレーションの接続は coordinator で確認する。`exhaustValidationRetries` の既存パターンを踏襲し、`receivedAttempts == [0, 1, 2]` で自動再抽選を観測する。

- Given: スクリプト AI を合成した coordinator。
- When: AI が全かな未変換「ほうほう」を返し続ける。
- Then: 自動 retry が上限まで走り、`receivedAttempts == [0, 1, 2]`・`retryCount == 2` になる。

想定テスト名（`CompositionCoordinatorTests` へ追加）。

- `@Test("全かな未変換の出力は自動 retry でエスカレーションされ、確定しない")`。

## 3. フォールバック: AI 不可用・失敗・検証 NG 時に辞書最良で一発成立する

AI が不可用・未準備・失敗・検証 NG のとき、`HybridConversionProvider` が辞書最良（dictBest）を確定フォールバックとして返し、候補リストを開かずに一発で成立することを確認する。AI の不可用・失敗は ADR-0004 の `ScriptedConversionProvider` で再現する（`setAvailability`・`failOldest`・全かな出力の resolve）。辞書は実サブセットを使う。

- Given: 実辞書 + `availability` を `.unavailable` にしたスクリプト AI を合成した provider。
- When: 「ほうほう」を変換する。
- Then: AI を呼ばず dictBest「方法」を確定フォールバックとして返し、`.converted` 相当で一発成立する。

- Given: 実辞書 + `convert` が `KotoError.generationFailed` を投げるスクリプト AI。
- When: 「けっきょく」を変換する。
- Then: dictBest「結局」へフォールバックして成立する。

- Given: 実辞書 + 全かな未変換「ほうほう」を返し続けるスクリプト AI（検証 NG を再現）。
- When: 「ほうほう」を変換する。
- Then: 検証 NG が上限に達した後、dictBest「方法」へフォールバックして成立する。

想定テスト名（`HybridConversionProviderTests`）。

- `@Test("AI が不可用なとき辞書最良で一発フォールバック成立する")`。
- `@Test("AI が未準備（preparing）のとき辞書最良で一発フォールバック成立する")`。
- `@Test("AI が KotoError で失敗したとき辞書最良で一発フォールバック成立する")`。
- `@Test("AI 出力が検証 NG を繰り返したとき辞書最良で一発フォールバック成立する")`。
- `@Test("フォールバックでも候補リストを開かず単一最良のみを確定する")`。

## 4. 決定性: 同一入力に対し常に同一の既定出力を返す

辞書引きと最良パス選択が決定的であり、最小コスト選択を安定ソートで一意に定めることを確認する。

- Given: 実辞書をロードした `DictionaryConverter`。
- When: 同一読み「ほうほう」を 100 回連続で変換する。
- Then: すべて同一の単一最良「方法」を返す。

- Given: 同音異義を複数表記持つ読み（辞書サブセットに含むもの）。
- When: 繰り返し変換する。
- Then: 既定の単一最良が一意に定まり、代替候補集合の並びも安定する。

想定テスト名（`DictionaryConverterTests`）。

- `@Test("同一読みの繰り返し変換は常に同一の単一最良を返す")`。
- `@Test("同コストの同音異義でも安定ソートで既定出力が一意に定まる")`。
- `@Test("代替候補集合の列挙順序が決定的である")`。

フォールバック経路の決定性も固定する。

- Given: AI 不可用のスクリプト AI を合成した provider。
- When: 同一入力「けっきょく」を繰り返しフォールバックさせる。
- Then: 確定値「結局」が毎回同一になる。

想定テスト名（`HybridConversionProviderTests`）。

- `@Test("AI 不可用時のフォールバック確定値が同一入力に対し決定的である")`。

## 5. 境界値

辞書ミス・空入力・保護語/頭字語の保持・膨張率上限を確認する。

### 5-1. 未知語（辞書ミス）

- Given: 辞書サブセットに無い読み「ぬんちゃくぽ」。
- When: `DictionaryConverter.convert` する。
- Then: ヒットしないセグメントはかなのまま残し、後段の AI とフォールバックへ委ねる（例外を投げない）。

想定テスト名。

- `@Test("辞書ヒットしないセグメントはかなのまま残して後段へ委ねる")`。
- `@Test("既知語と未知語が混在する読みは既知部分だけ漢字化し未知部分を保持する")`。

### 5-2. 空入力

- Given: 空文字列の読み。
- When: `DictionaryConverter.convert("")` する。
- Then: 空の単一最良を返し、クラッシュしない。

- Given: 合成 provider に空 source を渡す。
- When: 変換する。
- Then: `ConversionOutputValidator` の `emptyResponse` 契約に沿って failure になり、辞書フォールバックも空のため元テキストを保持する。

想定テスト名。

- `@Test("空入力の辞書変換は空を返しクラッシュしない")`（`DictionaryConverterTests`）。
- `@Test("空入力では合成変換が emptyResponse として失敗し元テキストを保持する")`（`HybridConversionProviderTests`）。

### 5-3. 保護語・頭字語の保持

`ConversionOutputValidator` の既存契約（保護語・頭字語の喪失検出）が辞書段の介在後も維持されることを確認する。辞書は保護語をかな化・漢字化せずに保持する。

- Given: `protectedTerms = ["Claude Code"]` を含む設定。
- When: 「Claude Code を直す」を辞書段に通す。
- Then: 「Claude Code」はそのまま残り、辞書はラテン保護語を変換対象にしない。

- Given: 頭字語「SWIFT」を含む読み。
- When: 合成変換し、AI 出力で頭字語が喪失する。
- Then: `ConversionOutputValidator` が喪失を検出し、辞書フォールバックでも頭字語を保持する。

想定テスト名。

- `@Test("辞書段は保護語をかな化・漢字化せず原文のまま保持する")`（`DictionaryConverterTests`）。
- `@Test("辞書フォールバックでも保護語が原文のまま残る")`（`HybridConversionProviderTests`）。
- `@Test("頭字語が喪失した AI 出力は拒否され、辞書フォールバックで頭字語を保持する")`（`HybridConversionProviderTests`）。

### 5-4. 膨張率上限

`ConversionOutputValidator.maximumExpansionRatio` の既存上限が、辞書最良を再ランクした AI 出力にも適用されることを確認する。

- Given: `maximumExpansionRatio = 1.0` の設定。
- When: 上限ぎりぎりの出力と上限超過の出力を検証する。
- Then: ぎりぎりは受理し、超過は `.failure(.generationFailed(...))`（メッセージに「長すぎます」を含む）で拒否する。

想定テスト名（`HybridConversionProviderTests`、検証単体は既存 `ConversionOutputValidatorTests` を流用）。

- `@Test("膨張率上限を超える AI 出力は拒否され辞書最良へフォールバックする")`。
- `@Test("膨張率上限以内の AI 出力はそのまま受理する")`。

## 6. 切り替え: ↑↓ で候補プールを巡回（任意）。既定は候補リストを常時提示しない

既定の確定値は単一最良出力のみであり、候補リストを常時提示しないこと、切り替えは ↑↓ の任意操作のときだけ候補プールを巡回することを確認する（ADR-0012 踏襲）。これは koto-input の差別化の核であり受け入れ基準として固定する。`CompositionCoordinatorTests` の既存パターン（`canCycleCandidates`・`selectCandidate(offset:)`）を踏襲する。

- Given: 合成 provider を持つ coordinator。
- When: 「ほうほう」を 1 回変換する。
- Then: 確定表示は「方法」のみで、候補が 1 件の間は `canCycleCandidates == false`（↑↓ はアプリへ通す）。

- Given: 変換後に再抽選（または別表記の蓄積）で候補プールに 2 件以上溜まった状態。
- When: `selectCandidate(offset:)` で巡回する。
- Then: `canCycleCandidates == true` になり、↑↓ で候補プール（AI 出力・dictBest・dictCandidates の代替・別 attempt）を順に巡回し、`commit` が選択中候補を確定する。

想定テスト名（`CompositionCoordinatorTests` へ追加）。

- `@Test("既定の変換は単一最良のみを表示し候補リストを常時提示しない")`。
- `@Test("候補が 1 件の間は巡回できず上下キーをアプリへ通す")`。
- `@Test("候補プールが 2 件以上のとき ↑↓ で巡回し commit が選択候補を確定する")`。
- `@Test("候補プールに dictBest と dictCandidates の代替表記が積まれる")`。

## 7. セキュリティ観点: 入力テキストを外部送信しない invariant の確認方法

辞書はローカル同梱データ、AI はオンデバイスであり、入力テキストを外部サービスへ送信しない invariant を維持する。確認は次の多層で行う。

- 静的検査: `bun scripts/architecture-harness.ts --staged --fail-on=error` を invariant 違反ゼロで通す。`DictionaryConverter`・`HybridConversionProvider` の新規コードが `INVARIANT_NO_MOCK_DATA` ほか既存 invariant に違反しないことを確認する。辞書段の不変条件（辞書はローカル・外部送信なし、既定出力は決定的）を harness に追記する場合は ADR-0016 で記録する。
- 契約の固定: `TextConversionProvider` の doc コメント「実装は入力テキストを外部サービスへ送信してはならない」を `HybridConversionProvider` が遵守する。`DictionaryConverter` は I/O とモデルに依存しない純粋ロジックであり、ネットワーク API を import しないことをレビューと依存関係で担保する。
- データソース: 辞書は SwiftPM リソースとしてバンドルに同梱したローカルファイルからのみロードし、リモートから取得しない。BSD-3-Clause の著作権表示は `NOTICE` もしくは `THIRD-PARTY-LICENSES` に記載する。
- プライバシー: 検証失敗メッセージに source / output の本文を混入させない既存契約（ADR-0010）を no-op 検出にも適用する（本計画 2 節）。

想定テスト名。

- `@Test("DictionaryConverter は外部 I/O を行わず同梱辞書のみで変換する")`（`DictionaryConverterTests`、ネットワーク非依存・決定性で間接確認）。
- `@Test("辞書ロードは同梱リソースから行われリモート取得しない")`（`DictionaryConverterTests`、Bundle リソース参照で確認）。

注記: ネットワーク送信の不在はユニットテストだけでは立証しきれないため、静的検査（harness）・依存関係レビュー・`docs/terminal-compatibility.md` のチェックリストによる実機一気通貫検証を併用する。

## 受け入れ基準との対応

| 受け入れ基準 | 対応セクション |
| --- | --- |
| 代表読みが辞書経由で決定的に漢字化され no-op が起きない | 1, 2 |
| 既定は単一最良のみ表示し候補リストを常時提示しない | 6 |
| AI 不可用・未準備でも辞書のみで一発成立 | 3 |
| 全かな未変換を failure 化し attempt で自動エスカレーション | 2 |
| 辞書引きと最良パス選択が決定的 | 4 |
| 既存テストが全て green でカバレッジ 100% | 全セクション |
| harness が invariant 違反ゼロで通る | 7 |
| 入力テキストを外部送信しない invariant 維持 | 7 |

## 実行手順

```bash
make swift-test   # Swift テスト（toolchain の無い環境では CI の swift ジョブで確認）
nr lint           # biome check
nr lint:text      # textlint
bun scripts/architecture-harness.ts --staged --fail-on=error   # invariant 違反検出
```

すべて Green になるまで完了としない。失敗したら原因を特定してコードを修正する（設定ファイルや invariant を変更しない）。
