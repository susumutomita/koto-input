## 1. ユーザーストーリーの詳細化

### 主ペルソナ

ターミナル（Apple Terminal / Ghostty / iTerm2）で Claude Code や Codex CLI に英語・多言語の指示を書く開発者。特徴は以下のとおり。

- 英語キーボードでローマ字入力し、日本語で考えた内容をそのまま英語（または他言語）へ変換したい場面が日常的に発生する。
- コミットメッセージの summary は英語で書きたいが、思考は日本語のため、ローマ字のまま打って Ctrl + Shift + E で変換したい。
- プロンプトには `Claude Code`、`git`、`SWIFT` などのコマンド名・製品名・頭字語が混在するため、これらが訳されると困る。
- 変換結果の質に不満があれば同じキーを連打して再抽選し、言語を変えたいときは別キーで切り替えられると便利である。
- 入力内容には業務上の機密が含まれうるため、テキストがデバイス外へ出ないことを前提に使う。

### 利用文脈ごとのストーリー

| ID | ストーリー | 頻度 | 成功基準 | 対応する受け入れ基準 |
|---|---|---|---|---|
| US-1 | 開発者として、ローマ字で打った日本語の指示（例: `kyouhaiihida`）を Ctrl + Shift + E で英語の自然文に変換したい。なぜならコミットメッセージや Pull Request の説明文を日本語で考えてからそのまま英語に出したいからである。 | 1 セッションあたり数十回 | AI が文意を保ちつつ自然な英文を返す。`Claude Code` 等の保護語が原文のまま残る。 | AC-1、AC-5 |
| US-2 | 開発者として、converted（英語）状態から同じキーを連打して別の候補を得たい。なぜなら最初の候補が文体的に堅すぎたり、ニュアンスが合わないことがあるからである。 | US-1 の 20% 程度 | attempt が増えるたびに出力が変わる（温度 0.8 の再抽選）。attempt 数に上限はない（既存の再変換機構と同じ）。 | AC-2 |
| US-3 | 開発者として、英語に変換した後でやはり中国語にしたいと思ったとき、Ctrl + Shift + C でその場でターゲットを切り替えたい。なぜなら変換前のローマ字に戻して打ち直す手間をなくしたいからである。 | US-1 の 5% 程度 | 別言語キーを押した時点で原文スナップショットから attempt 0 で変換が始まる。前の converted テキストが残らない。 | AC-3 |
| US-4 | 開発者として、converted 状態から Shift + Space を押すことで日本語変換に戻りたい。なぜならターゲット言語を間違えたとき、または日本語で入力したくなったときに素早く戻れる手段が必要だからである。 | US-3 と同程度 | Shift + Space で日本語変換（attempt 0）が始まる。converted テキストが Escape での復元対象にならない。 | AC-3 |
| US-5 | 開発者として、変換を取り消して元のローマ字に戻したいとき Escape で復元できてほしい。なぜならすべての変換先言語で Escape の挙動が一致していないとメンタルモデルが崩れるからである。 | US-1 の 10% 程度 | すべての言語ターゲットで Escape → 元のローマ字テキストへ正確に復元される。 | AC-4 |
| US-6 | 開発者として、`Claude Code`、`SWIFT`、`git` のようなコマンド・頭字語・識別子が翻訳されないことを信頼したい。なぜなら翻訳された識別子をそのままターミナルに貼り付けると動かないからである。 | 全変換の多数 | 2 文字以上の大文字連続（頭字語）と保護語登録済み語が訳文に原文のまま現れる。消えた場合はリトライ / 拒否される。 | AC-5 |
| US-7 | 開発者として、ターミナルで composition が無いときに Ctrl + Shift + E を押しても、ターミナルのショートカットを奪われたくない。なぜなら Ctrl + Shift + E が別の操作（例: 新しいタブを開く等）に割り当たっている環境があるからである。 | 誤操作時 | composition が無いときはキーイベントをアプリへ通す。 | AC-6 |
| US-8 | 開発者として、6 言語すべてでモデルへのプロンプトがターゲット言語を明示していることを確認したい。なぜなら言語指定が曖昧だとモデルが出力言語を誤ることがあるからである。 | CI 自動検証 | scripted provider でターゲット言語付きリクエストが届くことをテストが網羅している。 | AC-7 |
| US-9 | 開発者として、入力テキストがデバイス外に出ないという既存のプライバシー方針が多言語変換でも維持されてほしい。なぜならプロンプトに機密情報を含めることがあるからである。 | 常時 | 完全オンデバイス（ADR-0002 維持）。ログにテキストが出ない。 | 非機能要件 |

## 2. 受け入れ基準のテストシナリオ化

仕様書「受け入れ基準」の各項目を AC-1〜AC-8 とし、Given/When/Then 形式に展開する。AC-1〜AC-7 は `swift test --package-path KotoInput` で自動検証、AC-8 は実機手動検証である。

### AC-1: 英語変換の基本動作（scripted provider）

- Given: ADR-0004 の ScriptedConversionProvider を provider として `CompositionCoordinator` を構築する。composition に `kyouhaiihida` を入力済み（composing 状態）。
- When: Ctrl + Shift + E 相当の `.requestConversion(.english)` コマンドを発行し、変換タスクの完了を待つ。
- Then:
  - provider が受け取った `ConversionRequest.target` が `.english` である。
  - provider が受け取った `ConversionRequest.modelInputText` が `きょうはいいひだ`（かな化済み）である（既存のかな化前段が多言語変換でも適用されることの確認）。
  - `PromptBuilder` が生成した instructions に英語のターゲット言語指定が含まれる。
  - `CompositionState.sourceText`（Escape 復元用スナップショット）が `kyouhaiihida`（元のローマ字）のまま変わらない。

### AC-2: converted 状態での再抽選（同言語連打）

- Given: AC-1 と同じ構成で `.english` で converted 状態にある。`attempt` は 0。
- When: 同じ Ctrl + Shift + E を再び発行する。
- Then:
  - `ConversionRequest.attempt` が 1 に増える。
  - 2 回目以降のリクエストに温度 0.8 が適用される（greedy は attempt 0 のみ）。
  - `ConversionRequest.target` は `.english` のまま変わらない。
  - 原文スナップショット（`kyouhaiihida`）は維持される。

### AC-3: converted 状態での言語切り替えおよび日本語変換へ戻る

以下の 3 シナリオを個別に検証する。

#### シナリオ 3-a: converted（英語）→ 別言語キー（中国語）

- Given: `.english` で converted 状態にある。
- When: Ctrl + Shift + C 相当の `.requestConversion(.chineseSimplified)` を発行する。
- Then:
  - `ConversionRequest.target` が `.chineseSimplified` である。
  - `ConversionRequest.attempt` が 0 から始まる。
  - 原文スナップショットは同一の `kyouhaiihida` が維持される。

#### シナリオ 3-b: converted（英語）→ Shift + Space → 日本語変換

- Given: `.english` で converted 状態にある。
- When: `.requestConversion(.japanese)` コマンドを発行する（Shift + Space 相当）。
- Then:
  - `ConversionRequest.target` が `.japanese` である。
  - `ConversionRequest.attempt` が 0 である。
  - 原文スナップショットは維持される。

#### シナリオ 3-c: 6 言語すべてについて言語ターゲットの切り替えが attempt 0 で始まる

- Given: 任意の converted 状態。
- When: `.english` 以外の 5 言語（`.chineseSimplified`、`.korean`、`.french`、`.german`、`.spanish`）それぞれの `.requestConversion(target)` を発行する。
- Then: 各 `ConversionRequest.attempt` が 0 である。各 `ConversionRequest.target` がそれぞれの言語に対応している。

### AC-4: Escape 復元（全言語共通）

以下の前提・操作・期待結果を各言語ターゲットで繰り返す（E 英語、C 中国語、K 韓国語、F フランス語、G ドイツ語、S スペイン語）。

- Given: ターゲット言語で converted 状態にある。`CompositionState.sourceText` がローマ字テキスト（例: `kyouhaiihida`）。
- When: `restoreSource` コマンド（Escape 相当）を発行する。
- Then:
  - `displayedText` が元のローマ字テキストへ戻る。
  - translated text が復元先にならない。
  - 既存の `CompositionTransitionTests` / `CompositionCoordinatorTests` の Escape 関連テストがすべて Green のまま（回帰なし）。

### AC-5: 保護語・頭字語の原文維持

#### シナリオ 5-a: 保護語の維持

- Given: ScriptedConversionProvider を使用。入力が `Claude Code wo tsukau` で `.english` で変換済み。
- When: `ConversionOutputValidator.validate` が出力を検証する。
- Then: `Claude Code` が出力に原文のまま含まれる。`Claude Code` が出力から消えた場合はバリデーション失敗（変換リトライ or 拒否）になる。

#### シナリオ 5-b: 頭字語（2 文字以上の大文字連続）の維持

- Given: 入力に `SWIFT` が含まれ `.english` で変換。
- When: `ConversionOutputValidator.validate` が出力を検証する。
- Then: `SWIFT` が訳文に原文のまま含まれる。翻訳されて消えた場合は変換失敗扱いになる。

#### シナリオ 5-c: 日本語固有検証（末尾句点 strip・鉤括弧 unwrap）は `.japanese` のみ適用

- Given: ターゲットが `.english`、`.french` 等の非日本語で変換。
- When: `ConversionOutputValidator.validate` が出力を検証する。
- Then: 末尾句点 strip・鉤括弧 unwrap のロジックが実行されない（日本語固有検証の誤適用がない）。

### AC-6: composition が無いときのキーの非消費

- Given: composition が存在しない状態（composing 状態でない）。
- When: Ctrl + Shift + E / C / K / F / G / S の各キーイベントが InputController に届く。
- Then: いずれのキーイベントも `false` を返す（アプリへ通す）。`.requestConversion` は発行されない。

### AC-7: 6 言語すべてのプロンプトがターゲット言語を明示する

- Given: `PromptBuilder.instructions(settings:target:)` を 6 言語ターゲット（`.english`、`.chineseSimplified`、`.korean`、`.french`、`.german`、`.spanish`）で呼び出す単体テスト。
- When: 各 `target` で instructions を生成する。
- Then:
  - 各言語の instructions に対応言語名が明示されている（例: 英語なら `English`、中国語なら `Simplified Chinese` 等）。
  - 保護語・頭字語・コード・パスの原文維持指示が含まれる。
  - 入力に無い引用符・括弧で包まない指示が含まれる。
  - 訳文のみを返す指示が含まれる。
  - `.japanese` の instructions が翻訳指示ではなく従来の整文指示を使っている（既存テストの回帰）。

### AC-8: 実機確認（手動）

- 前提: macOS 26 / Apple Silicon / Apple Intelligence 有効の実機。Koto を入力ソースとして選択し、ターミナルを開く。
- 操作 8-a（英語変換）: `kyouhaiihida` と入力し Ctrl + Shift + E を押す。期待: 文意を保った自然な英文が表示される。
- 操作 8-b（再抽選）: 8-a の converted 状態でさらに Ctrl + Shift + E を押す。期待: 別の英文が表示される（非決定的なため必ずしも異なる内容とは限らないが、2〜3 回の連打で変化を確認できること）。
- 操作 8-c（言語切り替え）: 8-a の converted 状態で Ctrl + Shift + F（フランス語）を押す。期待: フランス語の自然文が表示される。
- 操作 8-d（Escape 復元）: 8-a または 8-c の converted 状態で Escape を押す。期待: 元の `kyouhaiihida` に戻る。
- 操作 8-e（保護語確認）: `Claude Code wo tsukau` と入力し Ctrl + Shift + E を押す。期待: 英文に `Claude Code` が原文のまま含まれる。
- 操作 8-f（空 composition）: composition なしの状態で Ctrl + Shift + E を押す。期待: ターミナルのショートカットやデフォルト動作がそのまま発動する（Koto が消費しない）。
- 確認結果を `docs/terminal-compatibility.md` の該当行（英語 + 最低 1 言語の 2 組み合わせ以上）に追記すること。合否判定基準: 「文意を保ち、ターゲット言語として自然な文であること」（モデル出力は非決定的なため、文体差は許容する）。

## 3. 優先度と依存関係

### 実装順序（依存の流れ）

```
P0: ConversionTarget enum + CompositionCommand 拡張（.requestConversion(ConversionTarget)）
      ↓ 依存
P0: CompositionTransition / CompositionState への target 保持・reduce 拡張（AC-2, AC-3）
      ↓ 依存
P0: PromptBuilder の言語分岐 + ConversionOutputValidator の言語分岐（AC-5, AC-7）
      ↓ 依存
P0: AppleFoundationModelsProvider のセッションキャッシュ拡張（target キー）
      ↓ 依存
P0: InputController のキールーティング（AC-6）+ scripted provider による統合テスト（AC-1 〜 AC-7）
      ↓ 依存
P1: 実機確認（英語: AC-8-a〜f）
      ↓ 依存
P1: 実機確認（追加 1 言語以上）+ docs/terminal-compatibility.md 追記
```

### 言語検証優先順位

| 優先度 | 言語 | 根拠 |
|---|---|---|
| P0 | 英語（E） | 主ペルソナの最頻用途（コミットメッセージ・Pull Request 説明文・Claude Code 指示）。実機確認必須。 |
| P1 | 中国語簡体字（C） / 韓国語（K） | アジア圏チームとのチャット用途。文字セットが日本語と異なるため、PromptBuilder の指示が効いているか実機で確認する価値がある。 |
| P2 | フランス語（F） / ドイツ語（G） / スペイン語（S） | CI では instructions の網羅テストのみ実施。実機確認は優先度 P1 の環境が整ってから行う。 |

### 実装ユニットの依存まとめ

| 優先度 | 作業 | 依存 | 根拠 |
|---|---|---|---|
| P0 | `ConversionTarget` enum と `CompositionCommand.requestConversion(ConversionTarget)` の拡張（既存呼び出しを `.japanese` へ移行） | なし | すべての後続実装の型基盤。型変更が先に確定しないと他ユニットを並行作業できない。 |
| P0 | `CompositionTransition` / `CompositionState` の target 保持・reduce 拡張 | `ConversionTarget` 完成 | 「同 target → attempt + 1、異 target → attempt 0」のロジックが状態機械の核心。テスト先行（TDD）。 |
| P0 | `PromptBuilder.instructions(settings:target:)` の言語分岐 | `ConversionTarget` 完成 | 言語ごとの instructions が正しくないと実機でも CI でも品質を保証できない。単体テスト（AC-7）が CI で完結する。 |
| P0 | `ConversionOutputValidator` の言語分岐（日本語固有検証を `.japanese` のみに限定） | `ConversionTarget` 完成 | 翻訳結果に日本語の句点 strip が誤適用されるバグを防ぐ。 |
| P0 | `AppleFoundationModelsProvider` のセッションキャッシュ（target をキーとする拡張） | PromptBuilder 完成 | instructions が target ごとに異なるため、キャッシュキーに target を含めないとセッションが混在する。 |
| P0 | `InputController` のキールーティング（Ctrl + Shift + 言語キー → `.requestConversion(target)`） + scripted provider による統合テスト | 状態機械・PromptBuilder・Validator 完成 | 統合テストは全 P0 ユニットの配線を確認する。キールーティングは composition 存在判定も含む（AC-6）。 |
| P1 | 実機確認（英語必須、追加 1 言語以上） | P0 完成 + Apple Intelligence 有効の実機 | CI では検証不能。コード完成後でないと意味がない。 |
| P1 | `docs/terminal-compatibility.md` への結果追記 | 実機確認 | 実機確認と同一セッションで実施するのが効率的。 |

### リリース判定

- P0 がすべて Green であることが PR マージの必須条件（CI で機械検証可能）。
- AC-8 は実機でしか検証できないため、PR 時点で未実施の場合はチェックリストとして PR 本文に残し、マージ前に英語 + 最低 1 言語の実機確認結果を記録する。

## 4. スコープ外の再確認と将来 Issue 化すべき項目

仕様書「スコープ外」を再確認し、将来 Issue 化の条件を補記する。

| スコープ外項目 | 今回外す理由 | 将来 Issue 化の条件（再検討トリガー） |
|---|---|---|
| オンデバイスモデル非対応言語（アラビア語等） | Foundation Models の対応言語外。品質を担保できない。 | Apple が対応言語を拡張し、実機で品質確認できたとき。Translation framework との 2 段変換も同時に設計を検討する。 |
| Translation framework との 2 段変換 | アーキテクチャが複雑になる。完全オンデバイスの方針（ADR-0002）との整合を別途判断する必要がある。 | オンデバイスモデルの翻訳品質に対するフィードバックが蓄積し、Translation framework の併用が明示的に要求されたとき。ADR で決定する。 |
| 言語キーマップのカスタマイズ UI・設定化 | 初期リリースの 6 言語固定割当で主ペルソナの用途をカバーできる想定。設定画面・永続化の追加コストが現時点の価値に対して高い。 | 固定割当との衝突（別アプリのショートカットとの干渉）が具体的に報告されたとき。`ConversionSettings` と設定 UI の追加を伴う ADR を起こす。 |
| composition 内の部分選択を訳す操作 | 変換は composition 全体を対象とする現行設計で主ペルソナの用途はカバーできる。部分変換は状態機械の大幅拡張を要する。 | 「一文の後半だけ英語に変換したい」というフィードバックが具体的に出たとき。状態機械の選択範囲モデルと UI（テキスト選択の表示）の設計を伴う。 |
| 訳調（フォーマル度等）の設定 | プロンプト設計の複雑度が増す。初期リリースは「自然な訳」で十分と判断。 | 「フォーマルな英語が欲しい」「カジュアルな英語が欲しい」というフィードバックが蓄積したとき。PromptBuilder の instructions 拡張と `ConversionSettings` への持ち込みを ADR で判断する。 |
| 翻訳セッションの prewarm 最適化 | 6 言語分の常駐は不要。初回要求時生成 + キャッシュで十分。 | 翻訳変換の初回レイテンシが体感上問題になるユーザーフィードバックが出たとき。composition 開始時の言語予測ヒューリスティックとともに検討する。 |

## 5. リスクと未決事項

### リスク

| ID | 内容 | 影響度 | 発生可能性 | 対応方針 |
|---|---|---|---|---|
| R-1 | ターミナルの Ctrl + Shift + キーとの衝突 | 高 | 中 | Designer の調査待ち。仕様書「振る舞い 5」で composition なし時は消費しないと決定済みだが、composing 中の衝突（例: Ghostty の Ctrl + Shift + E がターミナル機能を使う場合）は `docs/terminal-compatibility.md` の実機検証で確認が必要。衝突が確認されたら言語キーの変更または回避策を ADR で決定する。 |
| R-2 | キーコードではなく `charactersIgnoringModifiers` による文字判定のキーボードレイアウト依存 | 中 | 低 | 仕様書で「レイアウト差を吸収するため文字で判定」と決定済み。ただし JIS キーボードや AZIK 等の非 US 英語レイアウトで期待どおり動くかは実機確認が必要。実機確認（AC-8）のマトリクスに JIS キーボードを含めることが望ましい。 |
