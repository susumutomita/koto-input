# ローマ字かな変換層 ユーザーフィードバック

- 対象仕様: [docs/specs/2026-06-10-ローマ字かな変換層.md](./2026-06-10-ローマ字かな変換層.md)（承認済み、これを正とする）。
- 担当 Issue: <https://github.com/susumutomita/koto-input/issues/6>。
- ロール: User（実ユーザーペルソナによるシミュレーション）。
- ペルソナ: Ghostty / iTerm2 で Claude Code / Codex に日本語プロンプトを書く日本人開発者。英語コマンドと日本語指示文が高頻度で混在する。IME が遅い・予測不能なら即座に使うのをやめる。
- 前提: シミュレーション時点で `RomajiKanaConverter` は未実装。挙動はすべて仕様書の記述から決定的に追跡した。実装済みの状態遷移（`CompositionTransition` / `CompositionCoordinator`）と UI 描画（`InputController.render`）は現行コードを根拠にした。

## 1. 利用シナリオのシミュレーション

### シナリオ 1: `kyouhaiihida` を変換する（機能の核心）

- 操作: `kyouhaiihida` と打鍵 → Shift + Space。
- 仕様どおりの挙動: 1 語として判定され、`kyo-u-ha-i-i-hi-da` がすべてかな表に載るため完全変換可能。`きょうはいいひだ` がモデルへ渡り、実測根拠（仕様書の背景）から「今日はいい日だ」相当が返る。表示は変換完了まではローマ字のまま下線、成功で太下線の日本語に置き換わる。
- 体感: これが一番欲しかった挙動。従来の「京橋駅」事故（生ローマ字をモデルが誤読する）が決定的なかな化で消えるなら、毎日使う動機になる。かな化は同期 O(n) でモデル呼び出し前に終わるため、体感遅延が増えない点も仕様として明記されており安心できる。
- 小さな不安: 変換中の表示がローマ字のまま（下線スタイルも single のまま）なので、Shift + Space が「効いたのか」数百ミリ秒のあいだ分からない。仕様の「表示挙動は現状維持」は理解するが、体感としては既知のストレス源が残る。

### シナリオ 2: `Claude Code wo tukau to git mo benri` を変換する（英語混在）

- 操作: 上記を打鍵 → Shift + Space。
- 仕様どおりの挙動: 空白区切りの語ごとに判定する。`Claude` と `Code` は大文字 `C` がかな表に無く完全変換不能で原文維持。`git` は `gi` のあとに `t` が余るため原文維持。`wo` → `を`、`tukau` → `つかう`、`to` → `と`、`mo` → `も`、`benri` → `べんり`。かな化済みテキストがモデルへ渡り、「Claude Code を使うと git も便利」相当に整文される。
- 体感: 期待どおり。英語の固有名詞・コマンドが崩れないことが、このペルソナにとって採用可否を分ける最低ラインで、それを「子音余り判定」という決定的なルールで担保しているのは納得感がある。
- 気づいた点 1: この例で `Claude` と `Code` が守られているのは大文字のおかげであって、protectedTerms のおかげではない。仕様の protectedTerms 判定は「語単位の完全一致」だが、README のデフォルト設定例にある保護語 `Claude Code` は 2 語なので、語単位一致では永遠にマッチしない。小文字で `claude code` と打つ癖のあるユーザー（自分は CLI ではよく小文字で打つ）は保護されない。
- 気づいた点 2: 仕様書の期待値文字列は `Claude Code をつかう と git も べんり` で、`を` と `つかう` の間だけ空白が無い。語単位変換で空白を保持するなら `を つかう` になるはずで、期待値の空白規則が例から読み取れない。この文字列は受け入れ基準のテスト期待値そのものなので、曖昧なまま実装に渡ると QA と Developer の解釈がずれる。

### シナリオ 3: タイポした語を変換する（`kyouhaiihidda` / `kyouhaiihidqa`）

- 操作 A: `kyouhaiihidda`（`d` を重ねたタイポ）→ Shift + Space。
- 挙動 A: `dda` は促音規則で `っだ` に解釈でき完全変換可能。`きょうはいいひっだ` がモデルへ渡る。モデルが気を利かせて「今日はいい日だ」へ直すかは非決定的で、「今日はいいひっだ」のような不自然な結果も起こり得る。
- 体感 A: タイポが「っ」混じりの変なかなに化けて出てくる。Escape で元のローマ字に戻して打ち直せるので許容範囲だが、どこが悪かったのかは自分で気づく必要がある。
- 操作 B: `kyouhaiihidqa`（子音が余るタイポ）→ Shift + Space。
- 挙動 B: 語全体が完全変換不能となり、生ローマ字 `kyouhaiihidqa` がそのままモデルへ渡る。これはこの機能が解決したかった「生ローマ字の分かち書き誤読」がそのまま再発する経路で、「京橋駅」型の誤変換が返ってくる可能性が高い。
- 体感 B: ここが一番怖い。日本語のローマ字入力では空白を入れずに長く打つのが普通（看板例の `kyouhaiihida` 自体がそう）なので、文全体が 1 語扱いになり、1 文字のタイポで文全体がかな化対象から外れる。スコープ外（スペルミス補正をしない）と仕様に明記されているのは理解するが、起きたときに UI が何も説明しない（後述）ことと組み合わさると、「ときどき理由なく滅茶苦茶になる IME」という体感になる。これは即アンインストール要因になり得る。

### シナリオ 4: 英語コマンドだけ打って Enter（`git status`）

- 操作: `git status` と打鍵 → Enter（Shift + Space は押さない）。
- 仕様どおりの挙動: 変換要求をしていないので、ローマ字かな変換層は一切関与しない。既存挙動どおり 1 回目の Enter で composition が確定し、2 回目の Enter でターミナルへ送信される。
- 体感: 今回の機能が英語のみの行に余計なことをしないのは正しい。一方で、英語コマンドしか打っていない行でも Enter が 2 回必要なのは、ターミナル常駐のペルソナには毎回の摩擦で、Koto を常用入力ソースにするのを躊躇する最大の既存理由。今回のスコープ外であることは README とキー操作表から理解できるが、将来要望として残す（第 4 節）。

### シナリオ 5: 変換途中で Escape

- 操作: `kyouhaiihida` → Shift + Space → モデル応答前に Escape。
- 仕様・実装どおりの挙動: `CompositionTransition.restoreSource` が converting 中でも動き、変換タスクはキャンセルされ、表示は `sourceText`（元のローマ字）に戻って composing に復帰する。かな化はモデルへ渡す直前の変換だけで、`CompositionState` と restoreSource はローマ字のまま、という仕様の統合方針どおり。
- 体感: 挙動は正しく、復元対象が「かな」ではなく「打ったままのローマ字」なのは打ち直しの観点で正解。ただし converting 中の表示はもともと元ローマ字のスナップショットなので、Escape を押しても画面上は 1 ピクセルも変化しない。「キャンセルできたのか、まだ変換が走っているのか」が分からず、もう一度 Escape を叩いて composition ごと消してしまいそうになる。

### シナリオ 6: README の看板例文を変換する（`sekinin han'i` を含む文）

- 操作: README 冒頭の入力例 `kono authentication no sekinin han'i ga aimai dakara ...` → Shift + Space。
- 仕様どおりの挙動: `han'i` はアポストロフィを含み、仕様の「数字や他の記号を含む語は完全変換不能として原文維持」に該当して生ローマ字のままモデルへ渡る。`kono` → `この`、`no` → `の`、`sekinin` → `せきにん`、`ga` → `が`、`aimai` → `あいまい` などはかな化される。
- 体感: プロダクトの README が一番目立つ場所に掲げている例文が、新機能のハッピーパスから外れている。`'` は MS-IME・ことえり以来の標準的な撥音区切り（`han'i` = はんい、`hon'ya` = ほんや、`kin'yuu` = きんゆう）で、これを打つ習慣のあるユーザーほど裏切られる。さらに `'` 無しの `hani` は `はに`、`konya` は拗音優先なら `こにゃ` になるはずで、「ん + 母音/や行」の語を正しく打つ手段が仕様上存在しない。仕様の撥音の記述（`n`/`nn`、語末 `n`）は母音・や行が後続する場合の扱いに触れておらず、ここは仕様の穴だと感じる。

### シナリオ 7: 小文字の英単語・ツール名を含む文を変換する（`bun de test wo hashirasete`）

- 操作: `bun de test wo hashirasete` → Shift + Space。
- 仕様どおりの挙動: `test` は `te` のあと `st` が余り原文維持。しかし `bun` は `bu` + 語末 `n` で完全変換可能、`de` も完全変換可能。結果として `ぶん で test を はしらせて` がモデルへ渡る。モデルは `ぶん` を「文」「分」などへ変換し得て、ツール名 Bun は出力から失われる。
- 体感: これが日常で一番頻繁に踏む地雷になると確信する。`bun`、`sudo`（すど）、`make`（まけ）、`code`（こで）、`demo`（でも）、`mode`（もで）、`node`（ので）、`data`（だた）など、ターミナル開発者の語彙には「偶然ローマ字として完全変換できる小文字英単語」が大量にある。原文維持側の取りこぼし（false negative）は Escape で直せるが、かな化の誤爆（false positive）はモデルに渡った時点で情報が失われ、復元手段がない。回避策は protectedTerms への登録だけだが、語単位完全一致かつ大文字小文字区別なので、登録運用の負担が重い。

## 2. 使いにくい・わかりにくい点

### 2-1. かな化の誤爆は復元不能なのに、防御手段が protectedTerms しかない（high）

シナリオ 7 のとおり、小文字英単語の誤かな化は出力から元の語を失わせる。原文維持の誤り（git が残る）は「直せる失敗」だが、誤かな化は「気づけない・直せない失敗」で、非対称に深刻。protectedTerms（語単位完全一致・大文字小文字区別）だけでは、開発者の語彙の広さに追いつかない。仕様の判定ルール「完全にひらがなへ変換できる語だけかな化」は、英単語側の偽陽性率を一切考慮していないように読める。

### 2-2. `'`（撥音区切り）が「他の記号」扱いで、頻出語が生ローマ字のままモデルへ直行する（high）

シナリオ 6 のとおり。`han'i`、`tan'i`、`kin'yuu`、`hon'ya` などは技術文書の頻出語で、README の看板例文すら該当する。`'` をかな表（撥音区切り）に含めない限り、「正しい打ち方をしているユーザーほど変換品質が落ちる」という逆転が起きる。スコープ外節にも挙がっていないため、単なる仕様の抜けに見える。

### 2-3. なぜ変換されないのかが、ユーザーに一切伝わらない（high）

完全変換できない語が混ざって生ローマ字がモデルへ渡り誤変換が返ってきたとき、画面にあるのは太下線の誤った日本語だけ。現行実装では `KotoError.userMessage` は生成されるが `InputController.render` は status を下線スタイルにしか使っておらず、失敗メッセージも「この語はかな化できなかった」という手がかりも表示されない。シナリオ 3-B（タイポで全文が原文維持）と組み合わさると、ユーザーから見える現象は「同じ操作なのに結果が日によって滅茶苦茶になる」であり、原因（1 文字のタイポ）に自力で到達するのはほぼ不可能。仕様が「表示挙動は現状維持」と決めたことで、この機能の失敗モードが完全に不可視になっている点は再考してほしい。

### 2-4. protectedTerms のデフォルト値と判定方式がかみ合っていない（medium）

仕様は「語単位の完全一致（大文字小文字を区別）」だが、README が案内するデフォルト設定の保護語は `Claude Code` という 2 語のフレーズで、この判定には決して一致しない。出力検証（`ConversionOutputValidator`）は `contains` でフレーズを見るので、同じ protectedTerms が層によって違う意味を持つことになる。設定した語が「かな化からは守られないが出力検証では効く」という挙動は、ユーザーには説明不能。

### 2-5. かな表の範囲「一般的な日本語 IME 互換」が未定義で、誤爆率がそこで決まる（medium）

`th` 行（`the` → てぇ）や `ca` → か、`q` 行などの拡張行を持つ IME は多い。拡張行を入れるほど英単語の誤かな化（2-1）が増える。たとえば `ca` と `the` を受理すると `authentication` すら完全変換可能（あうてぇんてぃかてぃおん）になり得る。どこまでを「互換」とするかが受け入れ基準にも現れておらず、実装者のテーブル選定が体感品質を直接左右する。

### 2-6. 受け入れ基準の期待値文字列の空白規則が曖昧（medium）

シナリオ 2 の気づいた点 2 のとおり、`Claude Code をつかう と git も べんり` の `をつかう` だけ空白が無い。意図（かな語同士は連結する、のか、単なる誤記か）を仕様に明記してほしい。

### 2-7. 変換中・キャンセル時の無反応（low）

シナリオ 1 と 5 のとおり。仕様の現状維持方針は理解するが、「Shift + Space が効いたか分からない」「Escape が効いたか分からない」は新機能で変換頻度が上がるほど目立つ。

### 2-8. 語中の `.` `,` がかな化対象になる副作用(low)

`.` → `。` は文末では嬉しい（`kyouhaiihida.` → `きょうはいいひだ。`）。一方で語中に `.` を含む語（`node.js` 等）も除外対象にならず、たまたま子音が余る語が多いので救われているだけに見える。「`.` `,` は語末のみ変換」のような限定があると安心できる。

## 3. 受け入れ基準のユーザー視点での充足確認

仕様書の受け入れ基準を 1 つずつ、「ユーザーにとって意味があるか」の観点で確認した。

1. RomajiKanaConverter の単体テスト網羅（五十音・濁音・半濁音・拗音・促音・撥音・長音・変換不能語・protectedTerms）。
   - 意味はある。かな表の互換性は打鍵の癖がそのまま通るかどうかで、体感に直結する。ただし「一般的な IME 互換」の範囲（2-5）と `'` の扱い（2-2）が基準に現れていないため、テストが網羅していても自分の打ち方が通る保証にならない。撥音のケースに `han'i` / `konya` 型を含めることを求めたい。
2. `kyouhaiihida` → `きょうはいいひだ` の決定的変換テスト。
   - 意味がある。この機能の存在理由そのもので、ユーザーが最初に試す操作と一致している。
3. `Claude Code wo tukau to git mo benri` → `Claude Code をつかう と git も べんり` のテスト。
   - 意味はあるが、期待値の空白規則が曖昧（2-6）。確定した期待値で固定しないと、テストが通っても「どちらの挙動が正か」をユーザーに説明できない。
4. scripted provider で `ConversionRequest.sourceText` がかな化済みであることの検証。
   - ユーザーから直接は見えないが意味がある。「モデルに何を渡したか」は誤変換の原因切り分けの根拠で、プライバシー方針（ログを残さない）のもとでは、テストだけが唯一の証拠になる。
5. Escape で元のローマ字テキストへ復元される既存挙動の回帰なし。
   - 強く意味がある。タイポ時（シナリオ 3）の唯一のリカバリ手段であり、この機能が増やす「変な変換結果」から脱出する命綱。converting 中の Escape（シナリオ 5）もケースに含めてほしい。
6. 実機確認: `kyouhaiihida` → Shift + Space → 「今日はいい日だ」相当。
   - 意味がある。ただし「相当」の判定はモデルの非決定性に依存するため、確認者の主観が入る。確認時の OS・モデル状態と実際の出力文字列を記録として残してほしい。
7. `docs/terminal-compatibility.md` の対象ターミナルで E2E 確認し、結果を追記。
   - 意味がある。現在マトリクスは全組み合わせ「未検証」のままで、このペルソナの主戦場である Ghostty + Claude Code と iTerm2 + Claude Code が埋まらない限り、自分の環境で使えるかは賭けになる。最低この 2 組み合わせの記録を求めたい。

総評: 受け入れ基準はハッピーパス（基準 2・3・6）と回帰防止（基準 5）を押さえており、ユーザーにとって意味のある構成。一方で、この機能の失敗モード（誤かな化・原文維持時の生ローマ字直行）がユーザーにどう見えるかを確認する基準が 1 つも無い。「完全変換不能語を含む文を変換したとき、何が起きるかが説明可能であること」に類する基準の追加を希望する。

## 4. 今回のスコープ外で構わないが、将来欲しいもの

1. ライブかな表示（打鍵と同時のかな化プレビュー）。スコープ外と明記済みだが、誤かな化（2-1）とタイポ（シナリオ 3）の両方を「変換前に気づける」唯一の根本対策。優先度は最も高い。
2. かな化できなかった語のヒントをモデルへ併記する仕組み。たとえば `[INPUT]` にかな化前の原文を併記すれば、生ローマ字直行時の誤読をモデル側で緩和できる可能性がある。
3. 英語のみ・未変換の composition を Enter 1 回で確定までパススルーするオプション。シナリオ 4 の Enter 2 回問題の解消。
4. protectedTerms の強化。フレーズ一致、大文字小文字を無視するオプション、開発用語（ツール名・コマンド名）の組み込み辞書。2-1 と 2-4 の運用負担を下げる。
5. 変換不能語・失敗状態の視覚フィードバック。下線色やスタイルの変化だけでも、2-3 の「無言の失敗」が大幅に改善する。
6. 1 語内タイポのあいまい補正（スコープ外明記済み）。空白なし長文入力の文化がある以上、いつかは必要になる。

なお `'`（撥音区切り）対応（2-2）は「将来」ではなく今回のかな表に含めることを、ユーザーとして強く希望する。仕様の「一般的な日本語 IME 互換」という目標自体に含まれているはずの挙動だからである。
