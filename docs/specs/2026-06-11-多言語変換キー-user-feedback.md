# 多言語変換キー ユーザーフィードバック

- 対象仕様: [docs/specs/2026-06-11-多言語変換キー.md](./2026-06-11-多言語変換キー.md)（承認済み、これを正とする）。
- 担当 Issue: <https://github.com/susumutomita/koto-input/issues/24>。
- ロール: User（実ユーザーペルソナによるシミュレーション）。
- ペルソナ: Ghostty / iTerm2 で Claude Code / Codex に指示を書く日本人開発者。コミットメッセージや PR タイトルを英語で書く義務があるが、考えるのは日本語で行う。Ctrl + Shift 系のショートカットは VS Code / ターミナルとの衝突が頻発する経験がある。
- 前提: シミュレーション時点で実装は未完了。挙動はすべて仕様書の記述から決定的に追跡した。

## 1. 主要シナリオのウォークスルー

### シナリオ 1: ローマ字で打って Ctrl + Shift + E でコミットメッセージを英語にする（機能の核心）

- 操作:
  1. ターミナルで `git commit` を実行し、エディタを開かずにインライン入力する。
  2. Koto を選択中に `tsugi no commit de nyuusyoku shori wo kaizen suru` と打つ。
  3. `Ctrl + Shift + E` を押す。
  4. 応答を待ち、訳文を確認する。
  5. 気に入ったら `Enter` で確定する。
- 期待される表示列:
  1. 打鍵直後: `tsugi no commit de nyuusyoku shori wo kaizen suru`（下線付き）。
  2. `Ctrl + Shift + E` 直後: ローマ字のまま変化なし（変換中の無音期間）。
  3. 変換完了: `Improve input processing in the next commit`（太下線）。
  4. `Enter` 後: 確定されてカーソルが末尾に移動する。
- 体感: これがこの機能の最大の価値。日本語脳でプロンプトを書きながら、コミットメッセージは英語という使い分けが 1 つのキーコードで完結する。ローマ字のままでも英語に変換できるのは強力で、「日本語変換してから英語変換する」という 2 ステップを踏まなくてよい。
- 懸念点: 変換中、画面にまったく変化がない。手元のターミナルでは Ctrl + Shift + E が効いたのか、ただのタイプミスと同様に無視されたのか、数百ミリ秒どころか場合によっては数秒区別がつかない。オンデバイスモデルが応答するまでの無音時間は、仕様の既知の限定事項だと理解しているが、キー操作列の視覚フィードバックの欠如は新機能で特に目立つ。

### シナリオ 2: 日本語変換した後で「やっぱり英語で」と切り替える

- 操作:
  1. `repojitorinokoukaisettei wo kakunin suru` と打つ。
  2. `Shift + Space` で日本語に変換する。
  3. 「リポジトリの公開設定を確認する」が表示される（converted 状態）。
  4. 「PR タイトルにそのまま貼りたい」と気が変わる。
  5. `Ctrl + Shift + E` を押す。
- 期待される挙動: converted 状態から `Ctrl + Shift + E` を押すと、原文スナップショット（`れぽじとりのこうかいせってい を かくにん する`、仮名正規化済み）を使って英語変換に切り替わる。attempt 0 からやり直し。
- 期待される表示: `Verify repository visibility settings`（太下線）。
- 体感: 「日本語でいったん確認してから英語にする」という流れは自然で、仕様の「原文スナップショット維持」が正確に機能すれば非常に便利。一方で「今、日本語変換の converted 状態なのか、英語変換の converted 状態なのか」がユーザーから一切わからない。どちらも太下線のまま。状態を見分ける視覚的な手がかりがゼロなのは、操作が合っているかどうかの確信を持てないまま次の操作を進めることになる。

### シナリオ 3: 訳が気に入らず同キー連打で再抽選する

- 操作:
  1. `genzai no approach wa sukooshi reigai ga ooi` と打つ。
  2. `Ctrl + Shift + E` を押す。
  3. 変換結果が `The current approach has some exceptions.`（期待する「外れ値が多い」というニュアンスと少し違う）。
  4. もう一度 `Ctrl + Shift + E` を押す。
  5. 再抽選される（仕様: attempt 1 以降は温度 0.8）。
  6. 今度は `The current approach has too many edge cases.` と返ってくる。
  7. こちらの方が意図に近いので `Enter` で確定する。
- 期待される挙動: 同じ言語キーの連打で attempt が増え、温度 0.8 によって揺らぎのある別候補が来る。
- 体感: 「気に入らなければ再抽選」という UI パターンは直感的でよい。Shift + Space の日本語変換の既存挙動（2 回目以降揺らぎあり）と操作が対称なのも覚えやすい。ただし、何回連打した結果なのかが表示されないため、「もう揺らぎが尽きたのか、まだ押し続けていいのか」が分からない。収束するまで押し続けるしかない。

### シナリオ 4: 文の一部だけ英語にする（分割確定の運用）

- 操作（コミットメッセージに説明文を混ぜる例）:
  1. Claude Code への日本語指示コメントの前に英語の summary をつけたい。
  2. `Update authentication flow` と英語でそのまま打つ。
  3. `Enter` で確定する（1 パート目）。
  4. 続けて `認証フローを OAuth2 に移行し、リフレッシュトークンの管理を一元化した` と打つ。
  5. `Enter` で確定する（2 パート目）。
  6. もう一度 `Enter` でターミナルへ送信する。
- 期待される挙動: composition 単位での変換・確定の繰り返しで、パート分けが自然にできる。仕様の「composition が無いときは Ctrl + Shift + 言語キーを消費しない」により、確定後に誤ってキーを押しても誤動作しない。
- 体感: 仕様が「composition 内の部分選択だけを訳す操作はスコープ外」と明示しているので、分割確定の運用は自然な代替。ただしこのワークフローは、「英語パートは英語でそのまま打ち、日本語パートはローマ字で打って変換する」という使い分けが前提で、混在テキストを一発で変換する用途は別途 Shift + Space の日本語変換に依存することになる。この使い分けは README に書いてほしい（第 4 節参照）。

### シナリオ 5: VS Code 統合ターミナルで Ctrl + Shift + E がサイドバーに奪われた場合の体験

- 状況: VS Code のデフォルトキーバインドで `Ctrl + Shift + E` は「エクスプローラーサイドバーにフォーカス」に割り当たっている。VS Code の統合ターミナルで Koto を使っている場合、このキーは VS Code が先に受け取る可能性がある。
- 操作:
  1. VS Code の統合ターミナルを開く。
  2. `tsugi no pull request de test wo tsuyomeru` と打つ。
  3. `Ctrl + Shift + E` を押す。
  4. VS Code がキーを横取りしてエクスプローラーサイドバーが開く。
  5. 入力欄には何も変化しない（composition はそのまま）。
- 期待される挙動: 仕様の「composition が無いときは Ctrl + Shift + 言語キーを消費しない」は、composition がある状態でのキーの競合については記述がない。IME がキーを受け取る前にアプリがキーを消費した場合、Koto にキーイベントが届かない。
- 実際の体験: 英語に変換されない。エクスプローラーが開く。ユーザーは操作を失敗したことには気づくが、「VS Code がキーを横取りした」という原因には気づかない。`Escape` 押下で composition は戻るが、変換に失敗した理由も、回避方法も、画面上に何も出ない。
- 問題の深刻度: このペルソナの主戦場は Ghostty や iTerm2 よりも VS Code 統合ターミナルの比率が高い。「Ctrl + Shift + E が VS Code と衝突する」は機能の発見可能性を根本的に損なう。仕様がキーマップのカスタマイズをスコープ外としているため、現状では「VS Code のキーバインドを自分で変えてください」という説明しかできないが、この情報は README や `docs/terminal-compatibility.md` に書かれていない。最低限、既知の衝突と回避策を文書化することを求めたい。

### シナリオ 6: 変換後に別言語（中国語）に切り替える

- 操作:
  1. `kinou no meeting no youryo wo matome taidesu` と打つ。
  2. `Ctrl + Shift + E` で英語に変換する。
  3. `I'd like to summarize yesterday's meeting.`（英語）が表示される。
  4. 「やっぱり中国語メンバーに送りたい」と気が変わる。
  5. `Ctrl + Shift + C` を押す。
  6. 中国語（簡体字）に変換し直される（attempt 0、原文スナップショットを使う）。
- 期待される表示: `我想总结一下昨天的会议内容。`（太下線）。
- 体感: この「英語で確認してから別言語に切り替える」フローは仕様のユーザーストーリーに明示されており、操作自体はシンプル。ただし、6 言語が存在することをユーザーはどうやって知るのか。README のキー操作表には現状この機能の記載がなく、仕様書を読まない限り機能の存在すら知れない（第 4 節で README の下書きを用意した）。

## 2. 「使いにくい」「わかりにくい」点

### 2-1. キー割当の覚えやすさ: E / C / K / F / G / S の記憶負荷（high）

6 言語のキー割当のうち、E（English）・C（Chinese）・K（Korean）・S（Spanish）は頭文字との対応がある。しかし F は French か Finnish か（Finlandは F だが Finnish も F、Farsi も F）、G は German だが German の G と気づくまでにタイムラグがある。また 6 言語が同時に存在することは、初回使用時に「C が Chinese か Korean の略称 Co か Catalan か」という疑念を生む。

特に G と C の判別は、キーボードを見たままでは判断できず、仕様書か README を参照する必要がある。これは「覚えてしまえば速い」設計だが、「覚えるまでのコスト」が高く、発見可能性が低い。変換後に言語名が何らかの形でフィードバックされれば（たとえば `→ English` のような表示）、誤った言語キーを押したことに気づける機会が増える。

### 2-2. 6 言語の発見可能性がゼロ（high）

現在の README のキー操作表にはこの機能の記載がない。変換中に使えるキーの一覧も表示されない。「Shift + Space を押したことがある」ユーザーが「Ctrl + Shift + E も試してみよう」という動機を持つ経路が存在しない。

日本語変換の再抽選（Shift + Space の 2 回押し）は既存の挙動説明の中に書いてあるので発見できる。多言語変換は全く別のキーを押す操作で、README に一切書かれていなければ「存在しない機能」として永遠に使われない可能性がある。

### 2-3. 変換中のフィードバックの無さ（high）

仕様の「表示挙動は現状維持（marked text の下線のみ）」により、`Ctrl + Shift + E` を押した後から変換完了まで、画面に何の変化もない。

- キーが届いたかどうか不明。
- 何語に変換しようとしているのか不明。
- あと何秒かかるのか不明。
- 変換が失敗したのか、単に遅いだけなのか不明。

オンデバイスモデルの遅延（実感で 0.5〜数秒）が発生するシナリオでは、この無音期間が体感上の不安を増幅する。これはシナリオ 1 と 2 で述べた懸念と同根で、多言語変換では「正しい言語キーを押したか」という追加の不安が乗る分だけ深刻度が上がる。

### 2-4. 失敗時に何が起きたか分からない問題（high）

以下の失敗モードがあり、いずれも UI が何も説明しない。

1. **言語キーの横取り**（シナリオ 5）: VS Code などがキーを消費した場合、変換が起動されないが、画面上の変化もない。ユーザーは「キーが届かなかった」という事実を知る手段がない。
2. **モデルが保護語を訳してしまった場合の拒否**（仕様の「消えた出力は拒否される」）: 拒否された場合、ユーザーから見えるのは「何も変わらなかった」か「元に戻った」だけ。なぜリトライが必要なのかが伝わらない。
3. **composition が無い状態でキーを押した**（仕様の「消費しない」）: この場合はアプリにキーが通るので、たとえばターミナルのショートカットが発動する。それが意図通りなのか誤動作なのか、ユーザーには区別できない。
4. **翻訳精度が低い場合**（意味が壊れた訳文が返ってくる）: 検証ロジックで弾かれる条件（空・膨張率超過・保護語欠落）以外の低品質訳は現状そのまま表示される。ユーザーは再抽選（Ctrl + Shift + E 連打）で対処できるが、「再抽選すれば改善するかもしれない」という情報が提示されない。

### 2-5. 訳調の設定がないことによる予測不能な文体（medium）

仕様の非機能要件に「自然な訳」とあるが、formal / casual / technical の指定ができない。コミットメッセージには命令形（`Fix`, `Add`, `Update`）が慣習的に期待されるが、チャットメッセージには丁寧語が自然。現仕様ではプロンプトに文体指示が含まれず、モデルが状況から推測するしかない。ユーザーは「今日のコミットメッセージは命令形で揃ってくれるか？」を毎回試しながら使うことになる。訳調設定はスコープ外と明示されているが、現在の設定 API に `style` キー（`neutral` / `polite` / `plain`）が日本語変換にはあるにもかかわらず、翻訳には適用されない非対称が分かりにくい。

### 2-6. 「converted 状態」の種類がユーザーに見えない（medium）

日本語変換後と英語変換後のどちらも「太下線の変換済みテキスト」に見える。シナリオ 2 で述べたように、現在の状態（何語に変換されているか）をユーザーが視認できない。

- 誤って日本語変換後に再度 `Ctrl + Shift + E` を押せば英語に切り替わる（正しい操作）。
- 英語変換後に `Ctrl + Shift + E` を押せば再抽選（同じ英語の別候補）になる（也正しい）。
- しかし「今 E ボタンを押したら何が起きるか」という予測を立てるには、状態を知っている必要がある。

### 2-7. かな正規化の前段が全言語に適用される影響（low）

仕様に「かな正規化の前段は翻訳でも適用する（モデルへは `modelInputText` を渡す）」とある。これはローマ字入力に特化した最適化で、直接日本語（ひらがな・漢字）を打った場合や、英語をそのまま打った場合にも前段を通るのか、挙動が仕様から読み取りにくい。たとえば `how to implement OAuth2` と英語で打って `Ctrl + Shift + C` を押した場合、かな正規化前段が `OAuth2` を壊さないか不安が残る。保護語・頭字語（長さ 2 以上の大文字連続）の保護で `OAuth` は守られるはずだが、数字の `2` は保護対象外で、語として切り離される挙動になるかどうかが不明。

## 3. 受け入れ基準のユーザー視点での充足確認

仕様書の受け入れ基準を 1 つずつ、「ユーザーとして満足か」の観点で確認した。

1. **`kyouhaiihida` → Ctrl + Shift + E で英語の自然文になる（scripted provider でターゲット言語付きリクエストを検証、実機で文意を確認）。**
   - 評価: 満足。この受け入れ基準はシナリオ 1 の核心で、「ローマ字のまま打って英語に変換できる」という価値命題の検証として適切。ただし「自然文」の判定基準が主観的で、確認者と実装者の解釈のずれが起きやすい。`claude code ni shitsumon suru` → `Ask Claude Code` が自然か `Query Claude Code` が自然かという揺れを誰がどう判断するかを確認手順に書いてほしい。
2. **converted 状態から同じ言語キーの連打で attempt が増えて再抽選される（テスト）。**
   - 評価: 満足。シナリオ 3 のユースケースの直接的な保証であり、日本語変換の既存挙動と対称な設計を検証する基準として適切。
3. **converted（英語）から Shift + Space で日本語変換へ戻る、別言語キーでその言語へ変わる（テスト）。**
   - 評価: 満足。シナリオ 6 の言語切り替えとシナリオ 2 の日本語への戻りを保証している。ただし「converted（英語）から Shift + Space」が attempt 0 の日本語変換になるという仕様が、ユーザー向けには一切説明されていない。確定した後に「やっぱり日本語で」とするには Escape → Shift + Space の操作が必要かどうかが仕様から読み取れない（仕様の記述は converted 状態での Shift + Space のみ言及）。
4. **Escape で元のローマ字へ復元される（テスト）。**
   - 評価: 満足。失敗時のリカバリ手段の保証として必須で、この基準があることは安心できる。composing 中・converting 中・converted 状態のすべてから Escape が機能することをテストケースに含めてほしい。
5. **翻訳でも保護語（`Claude Code` 等）・頭字語（`SWIFT` 等）が原文のまま残り、消えた出力は拒否される（テスト）。**
   - 評価: 概ね満足だが懸念あり。開発者として `Claude Code`・`SWIFT`・`OAuth2`・`API`・`JSON`・パス文字列（`src/auth/handler.ts` 等）が訳されずに残ることは強い要求。「消えた出力は拒否」は再変換を強制するので安全側に倒れているが、拒否が多発すると `Ctrl + Shift + E` を押すたびに何も変わらない（実は拒否されてリトライ中）という無音の失敗が増える（2-4 参照）。
6. **composition が無いときは Ctrl + Shift + 言語キーを消費しない（テスト）。**
   - 評価: 満足。シナリオ 5 の VS Code での体験に直接関わる基準。ただしこの基準は「消費しない」ことのテストであり、「ユーザーが誤って押したときに混乱しないか」という体験側の検証ではない。テストが通っても VS Code との衝突問題は解決しない。
7. **6 言語すべてについて prompt の指示がターゲット言語を明示する（テスト）。**
   - 評価: 満足。翻訳の正確性に直結する基準で、実装の抜けを防ぐ観点から重要。ユーザーとして「6 言語すべてが実際に使える品質か」は実機確認と組み合わせないと分からないが、プロンプトレベルの確認として必要十分。
8. **CI green（swift ジョブ含む）。**
   - 評価: 必要条件。ユーザーとして直接は見えないが、リグレッションの検出として当然の基準。
9. **実機確認: 英語 + もう 1 言語で `docs/terminal-compatibility.md` の主要ターミナルにて変換・確定・復元を確認し、結果を追記。**
   - 評価: 必要だが不足感あり。現状マトリクスが全組み合わせ「未検証」のままで、これを 1 組み合わせ以上埋めることを基準にすることの意義はある。ただし、このペルソナが最も使う Ghostty + Claude Code と VS Code 統合ターミナルの組み合わせが対象に入っていない。VS Code 統合ターミナルでの `Ctrl + Shift + E` 衝突問題（シナリオ 5）は、実機確認で初めて発見される類の問題で、事前に `docs/terminal-compatibility.md` の対象に追加しておくことを求めたい。

総評: 受け入れ基準は機能の正しさを検証するテストとして概ね適切。ただし、「ユーザーが機能の存在を知れるか（発見可能性）」「キー衝突時に混乱しないか（エラーハンドリングの透明性）」「6 言語全部が実用品質か（翻訳精度の検証）」に対応する基準が存在しない。機能が実装されても使われないリスクと、使われてもキー衝突で諦められるリスクが残っている。

### 多言語変換

composition 中に `Ctrl + Shift + 言語キー` を押すと、現在の入力を指定した言語で変換する。ローマ字入力のまま英語に変換するなど、日本語変換（`Shift + Space`）と同じ操作感で多言語対応できる。

| キー | 変換先言語 |
|------|-----------|
| `Ctrl + Shift + E` | 英語（English） |
| `Ctrl + Shift + C` | 中国語 簡体字（中文 简体） |
| `Ctrl + Shift + K` | 韓国語（한국어） |
| `Ctrl + Shift + F` | フランス語（Français） |
| `Ctrl + Shift + G` | ドイツ語（Deutsch） |
| `Ctrl + Shift + S` | スペイン語（Español） |

- `Ctrl + Shift + E` を連打すると別候補を再抽選する（初回は決定的、2 回目以降は揺らぎあり）。
- 変換後に別言語キーを押すと、元の入力から別言語へ変換し直せる。
- `Shift + Space` で日本語変換に切り替えられる。
- `Escape` でどの変換状態からでも元のローマ字テキストへ戻せる。
- composition が無いときは `Ctrl + Shift + 言語キー` を消費せずアプリへ通す。

**VS Code・JetBrains など IDE の統合ターミナルで使う場合**: `Ctrl + Shift + E` は VS Code のエクスプローラーフォーカス、`Ctrl + Shift + C` はターミナルのコピーなど、アプリ側のショートカットと衝突することがある。その場合は IDE のキーバインド設定で競合を解除するか、ターミナルウィンドウを独立して使用すること。詳細は [docs/terminal-compatibility.md](./docs/terminal-compatibility.md) を参照。

また、「既知の制限」節に以下を追加することを提案する。

---

- 変換中の表示は marked text の下線スタイルのみ。多言語変換キーを押した後、変換完了まで画面上の変化はない。
- VS Code の統合ターミナルなど、一部環境では `Ctrl + Shift + 言語キー` がアプリ側のショートカットと衝突する。`docs/terminal-compatibility.md` に既知の衝突と回避策を記録している。
- 言語キーマップはカスタマイズできない（初期リリース）。
