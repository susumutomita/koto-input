# ハイブリッドかな漢字変換 仕様書

## 概要

かな漢字変換を Apple FoundationModels 単独から、辞書バックボーンと AI 再ランクのハイブリッドへ移行する。辞書が「正解の表記を候補集合に必ず含める」決定的な一次変換を担い、AI が「文脈で正解を自動選択し文体を整える」二次変換を担う。これにより読み未変換の no-op を撲滅し、AI 不可用時も一発変換が成立する状態を作る。

## 背景と課題

現状の変換経路は `RomajiKanaConverter`（規則ベースのローマ字かな化）の後段が FoundationModels 100% であり、かな漢字辞書やラティスを一切持たない。このため次の問題が起きる。

- 小型オンデバイス LLM が greedy デコードで変換を据え置くと、「ほうほう」が「ほうほう」のまま返る no-op が発生する。
- `ConversionOutputValidator` は長さ・保護語・頭字語しか検証せず、全かな未変換の出力を素通しさせるため、未変換が正規候補として確定できてしまう。
- 辞書がないため「方法」という正しい表記がそもそも生成される保証がない。

純 AI は文脈による曖昧性解消は得意だが網羅性と決定性を担保できず、純辞書は同音異義を自力で選べないため従来 IME のように候補切り替えを強いる。両者を組み合わせることで初めて「読みを打って一発・切り替えなし」が成立する。

## ユーザーストーリー

- koto-input の利用者として、読みをローマ字で打って変換キーを 1 回押すだけで、候補リストを選ばずに正しい漢字交じり文を得たい。
- 利用者として、AI モデルが未準備・不可用なときでも、辞書だけで一発変換が成立してほしい。
- 利用者として、まれに変換を外したときだけ、従来どおり候補を切り替えて救済したい。

## 設計判断（ヒアリング確定事項）

- 辞書ソース: mozc 系辞書（BSD-3-Clause、`dictionary_oss`）。Homebrew 公開 tap 配布に同梱できるライセンスを優先する。
- 変換粒度: 段階導入。今回は最長一致の単一最良パス変換を実装し、文節分割と Viterbi ラティスは後続 PR とする。
- AI の役割: 辞書候補の再ランクと文脈整形。辞書が一次、AI が二次。
- 切り替えの位置づけ: 既定は常に一発変換で候補リストを常時表示しない。候補切り替えは外したときだけの任意の保険として据え置く。これは koto-input の差別化の核であり、受け入れ基準として固定する。

## 受け入れ基準

- [ ] 代表的な読み（「ほうほう」→「方法」、「けっきょく」→「結局」など）が辞書経由で決定的に漢字化され、全かな未変換の no-op が起きない。
- [ ] 既定の変換は単一最良出力のみを表示し、候補リストを常時提示しない。切り替えは任意操作としてのみ動作する。
- [ ] AI（FoundationModels）が不可用・未準備のときも、辞書のみで一発変換が成立する（辞書最良をフォールバック確定とする）。
- [ ] `ConversionOutputValidator` が、出力が読み（全かな）と実質同一の未変換を failure 扱いにし、既存の attempt 機構で自動エスカレーションする。
- [ ] 辞書引きと最良パス選択が決定的であり、同一入力に対し常に同一の既定出力を返す。
- [ ] 既存テストが全て green で、カバレッジ 100% を維持する。
- [ ] `bun scripts/architecture-harness.ts --staged --fail-on=error` が invariant 違反ゼロで通る。
- [ ] 入力テキストを外部サービスへ送信しない invariant を維持する（辞書はローカル、AI はオンデバイス）。

## 非機能要件

- パフォーマンス: 辞書引きと最良パス選択はオンメモリのバイナリサーチまたはトライで数ミリ秒以内に収める。composition 開始時の `prewarm` で辞書をロードし、変換キー押下時のレイテンシを既存と同等に保つ。
- セキュリティ: 辞書はローカルに同梱するデータであり外部送信しない。プロンプトはセキュリティ境界ではないため、出力は従来どおり `ConversionOutputValidator` を必ず通す。
- アクセシビリティ: 候補切り替えの表示は marked text 内の巡回選択（ADR-0012）を踏襲する。

## 技術設計

### コンポーネント構成

```
ローマ字 → かな（RomajiKanaConverter, 既存）
        → 読み列
        → DictionaryConverter（新規・KotoCore・決定的）
              → 単一最良表記 + セグメントごとの代替候補集合
        → AppleFoundationModelsProvider（既存・再ランク + 整形）
              → ConversionOutputValidator（既存・no-op 検出を追加）
        → 確定（既定一発）／ ↑↓ で候補プールを巡回（任意）
```

### DictionaryConverter（新規）

- 配置: `KotoInput/Packages/KotoCore/Sources/DictionaryConverter.swift`。決定的なロジックのみを持ち、I/O とモデルに依存しない。
- 入力: `ConversionRequest.modelInputText`（かな化済み読み）。
- MVP アルゴリズム: 読みを左から最長一致でセグメント分割し、各セグメントに対し辞書から最小コストの表記を選んで連結する。辞書ヒットしないセグメントはかなのまま残し、後段の AI とフォールバックに委ねる。
- 出力: 単一最良文字列と、各セグメントの代替表記集合（候補切り替え用）。
- 決定性: 最小コスト選択は安定ソートで一意に定める。同一読みは常に同一の既定出力を返す。

### 辞書データの取り込み

- ソース: mozc `dictionary_oss`（BSD-3-Clause）。
- ビルド前処理: 生の辞書ソースを、読みをキーとしたコンパクトなテーブル（ソート済みバイナリ + バイナリサーチ、または読みトライ）へ変換し、SwiftPM リソースとして同梱する。MVP は高頻度語彙にサブセットして配布サイズを抑える。
- ライセンス遵守: BSD-3-Clause の著作権表示とライセンス本文を `NOTICE` もしくは `THIRD-PARTY-LICENSES` に追加し、同梱データの出典を明記する。
- No Mock 原則: テストは実 mozc 由来の小規模サブセット辞書を使う。スタブ辞書は使わない。

### HybridConversionProvider（新規）

- 配置: provider 層。`TextConversionProvider` を実装し、内部に `DictionaryConverter`（KotoCore）と既存 `AppleFoundationModelsProvider`（再ランク・整形）を合成する。
- `convert(_ request:)` の流れ:
  1. reading = `request.modelInputText`。
  2. dictBest, dictCandidates = `DictionaryConverter.convert(reading)`。
  3. AI に dictBest と reading と `contextEntries` を渡して文脈再ランク・整形し、`ConversionOutputValidator` を通す。
  4. 検証成功なら AI 出力を確定。AI が不可用・失敗・検証 NG のときは dictBest を確定フォールバックとして返す。
- `prewarm`: 辞書ロードと既存のモデルウォームアップを行う。

### ConversionOutputValidator の拡張（先行リファクタ）

- 出力が読み（全かな）と実質同一の未変換である場合を failure として追加する。target が `.japanese` のときに適用する。
- 既存の attempt と temperature による自動エスカレーション（ADR-0008、ADR-0015）に接続し、no-op を確定させない。
- この修正は辞書導入前でも独立して価値があるため、作業順序として先に入れる。

### 候補プールと切り替え

- 既定の確定値は単一最良出力のみ。`ConversionCandidate` の候補プールには、AI 出力・dictBest・dictCandidates の代替表記・別 attempt を順に積むが、常時提示はしない。
- 切り替えは ↑↓ の任意操作のときだけ候補プールを巡回する（ADR-0012 踏襲）。

### ADR

- ADR-0016 として「かな漢字変換は辞書バックボーンと AI 再ランクのハイブリッドで構成する」を Accepted で追加する。純 AI 単独変換を前提にしていた関連 ADR（ADR-0008 など）の前提を更新し、必要なら supersede 関係を明記する。
- harness invariant に辞書段に関する不変条件（辞書はローカル・外部送信なし、既定出力は決定的）を追記する場合は、同 ADR で記録する。
