# ローマ字かな変換層 インタラクション設計

- 対象機能: ローマ字かな変換層（仕様書: [`docs/specs/2026-06-10-ローマ字かな変換層.md`](./2026-06-10-ローマ字かな変換層.md)）。
- 担当 Issue: https://github.com/susumutomita/koto-input/issues/7
- ロール: Designer。
- 日付: 2026-06-10。

## 前提となる表示制約

Koto の表示チャネルは入力クライアントへ渡す marked text のみであり、装飾は下線スタイルの 2 値だけが使える。

| 表示手段 | 実装上の根拠 |
| --- | --- |
| 細下線（`NSUnderlineStyle.single`） | `InputController.render` が `status == .converted` 以外のすべてに適用する。 |
| 太下線（`NSUnderlineStyle.thick`） | `status == .converted` のときのみ適用する。 |

ポップアップ・候補ウィンドウ・スピナー・色・通知は存在せず、本機能でも追加しない（仕様の「UI コンポーネントの追加はない」、スコープ外の「変換候補 UI」に従う）。状態は `CompositionViewState.status`（idle / composing / converting / converted / failed）として renderer へ届くが、視覚上の表現は「marked text の内容」と「下線の太さ」の組み合わせがすべてである。

この機能で追加されるかな化（`RomajiKanaConverter`）は `CompositionCoordinator.startConversion` の内部でのみ起き、**表示には一切現れない**。ユーザーが見るテキストは常に「自分が打ったローマ字」か「モデルの最終出力」のどちらかである。

## 1. 状態ごとの表示仕様

| 状態 | marked text の内容 | 下線 | カーソル | 本機能で変わるか |
| --- | --- | --- | --- | --- |
| idle | なし（marked text 消去） | - | - | 変わらない。 |
| 入力中（composing） | 打鍵したローマ字そのまま | 細 | 編集位置（移動・選択可） | 変わらない。ライブかな表示はスコープ外。 |
| 変換中（converting） | Shift + Space 時点のローマ字スナップショットのまま | 細 | 変換要求時の位置のまま | 変わらない。かな化はこの瞬間に同期実行されるが表示には出ない。 |
| 変換後（converted） | モデル出力（validator 通過済み） | 太 | 末尾 | **テキストの中身だけが変わる**。表示メカニズム自体は不変。 |
| 失敗時（failed） | ローマ字スナップショットのまま | 細 | 変換要求時の位置のまま | 変わらない。エラーメッセージは state に保持されるが表示チャネルが無い（後述の既知の制約）。 |
| Escape 復元後（composing に戻る） | `sourceText`（元のローマ字） | 細 | 末尾 | 変わらない。復元対象はかな化前のローマ字であり、中間ひらがなには決して戻らない。 |

### 変わる点と変わらない点のまとめ

変わる点。

- converted 状態で表示される変換結果の品質（`kyouhaiihida` が「今日はいい日だ」相当になる）。
- 英語混在語（`git`、`Claude` 等）と protectedTerms 一致語が変換後テキストの中に原文のまま残ること。

変わらない点。

- 下線スタイルの割り当て（converted のみ太、それ以外は細）。
- 状態遷移そのもの（`CompositionTransition` のフロー）。
- 入力中の表示（ローマ字のまま。かなのライブ表示はしない）。
- Escape の復元対象（元のローマ字テキスト）。
- 変換中・失敗時の表示テキスト。
- 操作系（Shift + Space で変換、Enter で commit、Escape で復元 / 破棄、編集で変換キャンセル、Control + C で破棄して素通し）。

### 設計判断: 中間ひらがなを表示しない理由

かな化結果（`きょうはいいひだ`）を converting 中に表示する案は採らない。

1. 仕様の決定事項（入力中の表示はローマ字のまま、表示挙動は現状から変更しない）に反する。
2. Escape の心的モデルが「最後に自分が見ていた打鍵テキストに戻る」であるため、見たことのない中間表現を挟むと復元結果との対応が崩れる。
3. converting 中の表示変更は、モデル応答が速い場合に 2 段階のちらつき（ローマ字 → かな → 漢字）を生む。

## 2. インタラクションフロー

基本フロー: 入力 → Shift + Space → かな化（コード側、即時） → モデル変換（非同期） → 表示。

```
ユーザー            InputController        CompositionCoordinator / KotoCore        provider（モデル）
  |                       |                          |                                  |
  | k,y,o,u,h,a,...,d,a   |                          |                                  |
  |---------------------->| .insert                  |                                  |
  |                       |------------------------->| composing                        |
  |   [表示] kyouhaiihida（細下線）                   |                                  |
  |                       |                          |                                  |
  | Shift + Space         |                          |                                  |
  |---------------------->| .requestConversion       |                                  |
  |                       |------------------------->| converting                       |
  |                       |                          | sourceText を                    |
  |                       |                          | スナップショット（ローマ字）        |
  |                       |                          | RomajiKanaConverter で           |
  |                       |                          | "きょうはいいひだ" へかな化         |
  |                       |                          | （同期・O(n)・表示に出ない）        |
  |   [表示] kyouhaiihida（細下線のまま不変）          |                                  |
  |                       |                          |-- ConversionRequest(かな化済) --->|
  |                       |                          |        （非同期・キャンセル可）     |
  |                       |                          |<-- "今日はいい日だ" ---------------|
  |                       |                          | validator 通過 → converted       |
  |   [表示] 今日はいい日だ（太下線）                   |                                  |
  |                       |                          |                                  |
  | Enter                 |                          |                                  |
  |---------------------->| .commit                  |                                  |
  |   [表示] marked text 消去、テキスト確定            |                                  |
```

分岐フロー。

```
converting 中に編集（insert / deleteBackward）
  → 変換タスクをキャンセルし composing へ戻る（表示は編集後ローマ字 + 細下線）。

converting 中に Shift + Space を再押下
  → 旧タスクをキャンセルし、現在の表示テキストで新しい変換を開始する（表示は不変）。

変換失敗（モデル利用不可・生成失敗・validator 拒否）
  → failed。表示はローマ字スナップショット + 細下線のまま。

converted / converting / failed で Escape
  → .restoreSource。元のローマ字 + 細下線で composing へ戻る。

素の composing で Escape
  → .cancel。composition を破棄し marked text を消去する。

converted のまま編集
  → composing へ戻り、以後の復元対象は編集後テキストになる（2 回目の変換で再スナップショット）。
```

## 3. 英語混在時の見え方

例: `Claude Code wo tukau to git mo benri` と打って Shift + Space した場合。

| 段階 | ユーザーに見えるもの | 内部 |
| --- | --- | --- |
| 入力中 | `Claude Code wo tukau to git mo benri`（細下線） | - |
| Shift + Space 直後 | 同上、不変（細下線） | 語単位かな化 → `Claude Code をつかう と git も べんり` をモデルへ送る。 |
| 変換後 | 例: `Claude Code を使うと git も便利`（太下線） | モデル出力が validator を通過。 |

ユーザーが受け取る体験の要点。

- `Claude Code` と `git` は**ラテン文字のまま**変換後テキストに残る。これが「英語が崩れていない」ことの確認手段になる。
- かな化の語単位判定はユーザーから不可視であり、ユーザーが学ぶべきルールは「**空白で区切った英単語はそのまま残る**」の 1 つだけにする。設計・ドキュメント（README 等）の説明もこの 1 文に統一する。

### 完全変換できない語が残ったときの気付き方

スペルミス等で意図せず原文維持になった語（例: `tukua` と打ち間違えた場合）への気付きは、次の 2 層で担保する。

1. **commit 前確認**: 変換結果は必ず太下線付きの marked text として表示され、Enter を押すまで確定しない。日本語文の中にラテン文字が残っていれば、それが「変換されなかった語」のシグナルである。
2. **低コストの修正ループ**: Escape 1 回で元のローマ字全文に戻れるため、「Escape → 該当語だけ修正 → Shift + Space」で再試行できる。変換結果側を直接編集して再変換する経路もある。

留意点（設計上の限界として明記する）。

- 気付きの手掛かりは「ラテン文字が残っている」ことのみで、下線色や強調による位置の指示はできない。長文では見落としやすいため、ユーザー向けドキュメントには「変換は文単位・短い単位で行う」運用を推奨として書く。
- 全語が変換不能（例: `git status` のような英語のみの文）の場合、変換前後でテキストが変わらず、変化は下線が細 → 太になることだけになる。環境によっては知覚困難である（後述のアクセシビリティと UX リスク参照）。

## 4. アクセシビリティ要件（WCAG 2.1 AA 観点）

下線のみ・ホストアプリ描画という制約下で、該当する達成基準と本設計の位置付けを整理する。本機能は表示挙動を変えないため、**新たな後退は発生しない**ことが要件である。

| 達成基準 | 位置付け |
| --- | --- |
| 1.4.1 色の使用 | 状態の区別に色を使っていない（下線の形状差 + テキスト内容の変化）。適合方向。ただし細 / 太の形状差だけに依存させず、「テキストがローマ字から日本語に置き換わる」こと自体を converted の第一シグナルと位置付ける。 |
| 1.4.3 / 1.4.11 コントラスト | marked text と下線の描画はホストアプリ・ターミナル側のスタイルに委ねられ、Koto 側で保証できない。既知の制約として記録し、`docs/terminal-compatibility.md` の E2E 確認時に「細下線と太下線が判別できるか」を観点に含める。 |
| 2.1.1 キーボード | 全操作（入力・変換・確定・復元・破棄）がキーボードのみで完結する。適合。 |
| 2.2.1 タイミング調整可能 | 変換待ちに時間制限を課さない。converting 中も編集・Escape・Enter・Control + C を常時受け付け、ユーザーがロックされない。適合。 |
| 3.2.2 入力時 | Shift + Space という明示的な操作によってのみ変換が起きる。打鍵途中で表示が予告なく変化しない（ライブ変換をしない決定はこの基準にも沿う）。適合。 |
| 4.1.2 名前・役割・値 | marked text は InputMethodKit の標準経路（`setMarkedText`）を通るため、VoiceOver 等の支援技術はホストアプリ経由で内容を取得できる。Koto 独自の UI を持たないことで、独自 UI 起因の AT 非互換を持ち込まない。 |

支援技術ユーザー向けの留意点。

- 下線スタイルの変化（細 → 太）は読み上げでは伝わらない。converted の通知はテキスト内容の変化に依存する。英語のみの文では内容も変わらないため、変換完了が伝わらないケースがある。
- converting 中は表示が一切変わらないため、視覚・聴覚いずれでも「処理中」と「無反応」を区別できない。NSAccessibility 通知等での改善は本機能のスコープ外とし、フォローアップ候補として記録する。

## 5. エラー状態・空状態・ローディング状態の設計

既存方針（スピナー・通知・追加 UI を出さない）の範囲内で設計する。

### ローディング状態（converting）

- 表示はローマ字のまま不変、細下線。専用のローディング表現は持たない。
- 設計上の意味付け: 「表示が変わらないこと」が待機中であり、「テキストが置換され太下線になること」が完了の合図である。
- 待ち時間中もユーザーは主導権を持つ: 編集すれば変換はキャンセルされて composing に戻り、Escape で復元、Enter ならその時点の表示テキスト（ローマ字）のまま commit、Control + C で破棄してターミナルへ割り込みを通せる。
- かな化は同期・O(n) の純関数であり、ローディング時間には寄与しない（体感遅延を生まないという仕様の非機能要件どおり）。

### 空状態

- composition が無い状態の Shift + Space は Koto が消費せず、ホスト（ターミナル）へそのまま通す。既存挙動を変えない。
- composition があっても表示テキストが空白のみの場合、変換要求は noop（状態も表示も変えない）。エラーを見せる価値がないため沈黙する。既存挙動を変えない。
- かな化固有の空状態は発生しない: `RomajiKanaConverter` は純関数で失敗せず、変換できない入力は「原文維持」になるだけでエラーを生まない。

### エラー状態（failed）

発生源はモデル利用不可（準備中・非対応環境）、生成失敗、validator 拒否の 3 系統で、かな化は失敗要因にならない。

- 表示: ローマ字スナップショットのまま、細下線。**テキストは一切失われない**ことを最優先の保証とする。
- 既知の制約: 現行の描画は converted か否かしか区別しないため、failed は composing と視覚上同一になる。エラーメッセージ（`KotoError.userMessage`）は state に存在するが表示チャネルが無い。本機能ではこの制約を変更しない（仕様の「表示挙動は現状から変更しない」に従う）。
- 回復経路（すべて既存操作のままで成立する）。
  1. もう一度 Shift + Space: 同じテキストで再試行する。モデル準備中だった場合の主経路。
  2. 編集して再変換: composing に戻ってから修正できる。
  3. Enter: ローマ字のまま commit する。変換を諦めてもタイプ結果は手元に残る。
  4. Escape: `sourceText` へ復元する（failed では表示と同一のため見た目は変わらないが、素の composing に戻る）。

## 既知の制約とフォローアップ候補

本機能のスコープでは解決しない（実装もしない）が、設計として認識しておく事項。

- failed と composing の視覚的区別が不能（エラーメッセージの表示チャネルが無い）。
- converting 中の進行フィードバックが無い（無反応との区別が付かない）。
- 全語変換不能な文では converted の変化が下線の太さのみになる。
- 中間ひらがなを確認する手段が無く、かな化の解釈（`n` の曖昧性等）起因の誤変換をユーザーが原因切り分けできない。

これらは `/follow-up add` で記録し、別 Issue・別 PR で扱う。
