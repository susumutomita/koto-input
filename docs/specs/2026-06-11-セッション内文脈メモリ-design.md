# セッション内文脈メモリ インタラクション設計

- 対象機能: セッション内文脈メモリ（仕様書: [`docs/specs/2026-06-11-セッション内文脈メモリ.md`](./2026-06-11-セッション内文脈メモリ.md)、設計判断の正本: [ADR-0013](../adr/0013-ローカル文脈メモリは-opt-in-のセッション内記憶から始める.md)）。
- 担当 Issue: https://github.com/susumutomita/koto-input/issues/46
- ロール: Designer。
- 日付: 2026-06-11。

## 前提となる表示制約

Koto の表示チャネルは marked text の下線スタイル（細 = single、太 = thick）のみである。ポップアップ・候補ウィンドウ・スピナー・通知は存在せず、本機能でも追加しない。候補の往復は ADR-0012 の marked text 内巡回選択（↑/↓）をそのまま使う。

本機能は「キールーティング 1 本の追加 + プロンプトへの `[CONTEXT]` セクション付与 + commit テキストの in-memory 収集」であり、状態機械の構造・表示メカニズム・`InputController.render` は変更しない。

確定事項（変更不可）: トリガーは Ctrl + Shift + Space、保持件数 N = 5、合計上限 500 文字（UTF-16 長）、既定 OFF、OFF のときキーは消費せずアプリへ通す。

## 1. キーインタラクション仕様

「消費」= IMK が `true` を返してホストアプリへキーイベントが届かない。「通す」= `false` を返してホストアプリへ届く。

### 1-1. Ctrl + Shift + Space（文脈つき日本語変換）: 状態 × 設定マトリクス

縦軸は `CompositionPhase`、横軸は `contextMemoryEnabled` の ON / OFF。

| 状態 | 設定 OFF（既定） | 設定 ON |
| --- | --- | --- |
| idle | 何もしない。**通す**。収集もゼロで従来挙動と完全一致。 | 何もしない。**通す**。composition が無ければ消費しない（既存キーと同じ原則。アプリ側の Ctrl + Shift + Space 割当が生きる）。 |
| composing | 何もしない。**通す**。 | `.requestContextualConversion` を発行する。displayedText をスナップショットし、store の `snapshot()` を変換タスク開始時に取得して converting へ遷移する。表示はスナップショットのまま細下線。**消費**。 |
| converting | 何もしない。**通す**。 | 既存タスクをキャンセルし、現在の sourceText で文脈つき変換を再開始する（新 requestID）。表示は不変。**消費**。 |
| converted | 何もしない。**通す**。 | 直前の変換も文脈つき（useContext = true、target = .japanese）なら attempt + 1 の再抽選。直前が Shift + Space（useContext = false）や翻訳なら attempt 0 の greedy から文脈つき変換を開始する（attempt 同一性キーは（target, useContext））。候補はクリアせず蓄積を継続し、↑/↓ で通常候補と見比べられる。**消費**。 |
| failed | 何もしない。**通す**。 | composing と同じ扱い。`.requestContextualConversion` で再試行する。**消費**。 |

補足。

- 設定確認はこのキー組み合わせの押下時のみ行う。毎キーストロークでの設定ロードは発生させない（仕様の技術設計に従う）。
- OFF のときは状態を問わず一切消費しない。これにより「既定で従来挙動と完全一致」の受け入れ基準（キーがアプリへ通る・収集ゼロ）が成立する。
- ON でも文脈の有無はキー消費の条件にしない。文脈が空でも `.requestContextualConversion` は発行され、結果は通常変換と同一になる（セクション 3-1）。「押したのに何も起きない」分岐を作らないための判断である。

### 1-2. 既存キーとの衝突がないことの確認（InputController のルーティング検証）

現行 `InputController.process(_:)` のルーティングを Ctrl + Shift + Space（keyCode 49、essentialFlags = [.control, .shift]）で追跡した結果。

| 既存ルート | 判定条件 | Ctrl + Shift + Space との関係 |
| --- | --- | --- |
| Shift + Space（日本語変換） | `KeyCode.space where essentialFlags == .shift` | flags が `.shift` 単独でないためマッチしない。**衝突なし**。 |
| Tab（かな正規化） | `KeyCode.tab` | keyCode が異なる。**衝突なし**。 |
| Escape / Enter / Backspace / ←→ | 各 keyCode + `essentialFlags.isEmpty` 等 | keyCode が異なる。**衝突なし**。 |
| ↑/↓（候補巡回） | keyCode 126 / 125 + 修飾なし + `canCycleCandidates` | keyCode が異なる。**衝突なし**。文脈つき変換の結果も `candidates` に入るため、↑/↓ の消費条件（converted かつ候補 2 件以上）はそのまま機能する。 |
| Ctrl + Shift + 言語キー（翻訳） | `essentialFlags == [.control, .shift]` + `ConversionTarget(languageKey:)` が非 nil | flags は同一だが、space の文字 `" "` は `ConversionTarget(languageKey:)` の `e/c/k/f/g/s` いずれにも該当せず nil を返す。**衝突なし**。判定は仕様どおり keyCode 49 で行い、文字判定（レイアウト依存の言語キー）とは独立に扱う。 |
| Ctrl + C（割り込み） | `.control` を含む + 文字 `"c"` | 文字が異なる。**衝突なし**。 |
| 文字入力フォールバック | `essentialFlags.subtracting(.shift).isEmpty` | `.control` が残るため到達しない（現行はここで `false` となり通している）。**衝突なし**。 |

実装上の注意（Developer 向け）。

- 現行コードで Ctrl + Shift + Space はどのケースにもマッチせず `false` で通っている。つまり OFF 時の「完全一致」は新分岐が `false` を返す限り自動的に成立する。新分岐は `case KeyCode.space where essentialFlags == [.control, .shift]` として switch 文に追加すればよく、既存の `essentialFlags == .shift` ガードとは排他である。
- 矢印キー対策の `essentialFlags = flags.subtracting([.function, .numericPad])` は space に `.function` が立たないため影響しない。
- JIS / US いずれのレイアウトでも space の keyCode は 49 であり、文字判定不要。

### 1-3. 既存キーへの波及（文脈つき変換後の各キーの挙動）

文脈つき変換は reducer 上で `requestConversion(.japanese)` と同じスナップショット・候補蓄積・再抽選規則を共有するため、既存キーの挙動マトリクス（[`2026-06-11-多言語変換キー-design.md`](./2026-06-11-多言語変換キー-design.md) セクション 1）がそのまま成立する。差分のみ示す。

| キー | 文脈つき変換後（converted、useContext = true）の挙動 |
| --- | --- |
| Shift + Space | useContext = false の日本語変換を attempt 0 の greedy から開始する（attempt キー（target, useContext）が異なるため再抽選ではない）。候補は蓄積継続。 |
| Ctrl + Shift + Space | 同一 attempt キーのため attempt + 1 の再抽選（温度つき、ADR-0008 の規則を共有）。 |
| Ctrl + Shift + 言語キー | attempt 0 から当該言語へ変換し直す。翻訳 target には文脈を付与しない（第一版のスコープ外）。 |
| Tab | composing へ戻る（applyEdit）。候補・useContext はリセットされる。 |
| ↑/↓ | 蓄積済み候補（文脈あり・なし・翻訳）を巡回する。候補の重複判定キーは text + target のままなので、文脈の有無で同一テキストになった候補は 1 件に統合される。 |
| Escape | `.restoreSource` で元のローマ字へ復元する。文脈つき変換でも sourceText / isSourcePreserved は不変のため保証は変わらない。 |
| Enter | 表示中の候補だけを commit する（送信しない）。commit テキストは設定 ON なら store へ追記される。 |
| Ctrl + C | `.cancel` で破棄してから通す。変更なし。 |

### 1-4. 状態遷移サマリー（文脈つき変換追加後）

```
idle
  ↓ insert
composing
  ↓ Shift+Space                    → converting（useContext=false）
  ↓ Ctrl+Shift+Space（ON のみ）     → converting（useContext=true）
converting(requestID, target, useContext)
  ↓ conversionSucceeded            → converted（候補へ蓄積）
  ↓ conversionFailed               → failed
converted
  ↓ Ctrl+Shift+Space（同キー再押下）→ converting（attempt+1 再抽選）
  ↓ Shift+Space                    → converting（useContext=false、attempt 0）
  ↓ Ctrl+Shift+言語キー            → converting（翻訳、文脈なし、attempt 0）
  ↓ ↑/↓                            → converted（候補差し替えのみ）
  ↓ Escape                         → composing（sourceText 復元）
  ↓ Enter                          → idle（commit。ON なら store へ追記）
```

### 1-5. ホストアプリ既定ショートカットとの衝突調査（Ctrl + Shift + Space）

| アプリ | 既知の割当 | 衝突の有無と評価 |
| --- | --- | --- |
| Terminal.app | なし | 衝突なし。 |
| iTerm2 | デフォルト未割当 | 衝突なし（デフォルト）。 |
| Ghostty | デフォルト未割当 | 衝突なし（デフォルト）。 |
| tmux | なし（プレフィックスは Ctrl + B） | 衝突なし（デフォルト）。 |
| VS Code | macOS では Trigger Parameter Hints は Cmd + Shift + Space であり Ctrl + Shift + Space は既定未割当 | 衝突なし（デフォルト）。Windows/Linux からのキーマップ移植ユーザーは割当がありうる。 |
| JetBrains 系 IDE（IntelliJ 等） | Smart Type Completion が Ctrl + Shift + Space | **あり（composition 中かつ ON のみ）**。OFF（既定）では常に通り、ON でも idle では通るため、影響は「ON にしたユーザーが composing 中に Smart Completion を押した場合」に限定される。 |
| macOS システム | 入力ソース切替は Ctrl + Space / Ctrl + Option + Space で修飾が一致しない | 衝突なし。 |

結論: 既定 OFF・「ON でも composition 中のみ消費」の二段の絞り込みにより、既知の衝突はすべて実用上回避できる。JetBrains 系の衝突は README のキー操作節に明記して対処する（セクション 5-2）。

## 2. 状態表示

### 2-1. 下線スタイル 2 値ルールの踏襲

| 状態 | marked text の内容 | 下線スタイル | 本機能での変更 |
| --- | --- | --- | --- |
| composing | 打鍵したテキストそのまま | 細（single） | 変更なし。 |
| converting（文脈つき変換中） | 変換要求時のスナップショットのまま | 細（single） | 変更なし。通常変換中と視覚上同一。 |
| converted（文脈つき変換後） | モデル出力（validator 通過済み） | 太（thick） | テキスト内容のみが変わる。スタイルは既存ルールと同じ。 |
| failed | スナップショットのまま | 細（single） | 変更なし。 |
| Escape 復元後 | sourceText | 細（single） | 変更なし。 |

### 2-2. 文脈変換であることの視覚的区別は第一版では持たない（明文化）

文脈つき変換の結果は、通常の日本語変換の結果と視覚上区別できない。これは以下の理由による意図した判断である。

- 表示チャネルは下線 2 値（細 / 太）のみで、既に「変換前後」の区別に使い切っている。第 3 のスタイル（点線・色等）を文脈の有無に割り当てると、最優先のシグナルである「converted かどうか」の判読を阻害する。色の使用は WCAG 1.4.1 の観点からも単独のシグナルにできない。
- 候補の重複判定キーは text + target（ADR-0012）であり、文脈あり / なしで同一テキストになった候補は 1 件に統合される。候補単位で「これは文脈由来」と表示し分けるメタデータ表示チャネルがそもそも存在しない。
- ユーザーへのフィードバックは「自分が Ctrl + Shift + Space を押した」という操作の記憶と、出力テキストの内容（曖昧な参照が具体化されている）で十分成立する。pull 型のみ（ADR-0013 Decision 1）なので、ユーザーが要求していないのに文脈が混入することはない。

将来、候補メタデータの表示手段（IMKCandidates パネル再検討等）が入る場合に文脈由来の表示を再検討する。

## 3. エラー状態・空状態

### 3-1. 文脈が空のとき（空状態）

- store が空（セッション開始直後・ON 直後・OFF→ON 直後・clear 直後）のとき、`snapshot()` は空配列を返し、`PromptBuilder` は従来の `[INPUT]` のみのプロンプトをバイト単位で同一に構築する（仕様の技術設計）。
- したがって文脈が空の Ctrl + Shift + Space は、**Shift + Space の attempt 0 と同一の変換結果になる**。エラーにも空表示にもならない。
- 直前に Shift + Space を押していた場合、同一テキストの結果は候補へ重複追加されず既存候補が選択し直されるだけである（ADR-0012 の重複規則）。「押しても候補が増えない」のは正常系であり、故障ではない。
- 空白のみのテキストは `append` で無視されるため、空白 commit の連続で有効な文脈が押し出されることはない。

### 3-2. 変換失敗時（モデル利用不可・生成失敗）

- `KotoError.modelUnavailable` / `.generationFailed` 等で `.failed(message: error.userMessage)` へ遷移する。通常変換と同じ経路で、文脈固有のエラー型は追加しない。
- 表示はスナップショットのまま細下線。テキストは一切失われない。
- 回復経路: Ctrl + Shift + Space 再押下（文脈つきで再試行）、Shift + Space（文脈なしで再試行）、Ctrl + Shift + 言語キー（翻訳）、Enter（原文のまま commit）、Escape（`.restoreSource` で sourceText へ復元して composing へ）。
- Escape 復元は文脈つき変換でも保証される。文脈は変換タスクへのスナップショット渡しのみで sourceText / isSourcePreserved に触れないためである。

### 3-3. 検証拒否時（ConversionOutputValidator）

- validator は変更しない。膨張率（4.0 × 原文長 + 固定許容）・保護語検証は sourceText 基準のまま適用される。
- **文脈展開は本質的に出力を長くする機能であるため、短い入力（例: 「あれやっといて」）からの展開が膨張率上限を超えて拒否されるケースは通常変換より起きやすい**。これは第一版の意図した安全側の挙動であり、拒否時は `.failed` へ遷移してスナップショットが保持され、Escape で原文へ戻れる。
- 拒否時のメッセージは `KotoError.userMessage`（「変換に失敗しました。元のテキストを保持しています。」等）として `CompositionViewState.status` に乗るが、**現行の render はメッセージを視覚表示しない**（表示チャネルが無い）。failed と composing は視覚上同一という既知の制約は本機能でも変わらない。
- ユーザー側の回避策は `maximumExpansionRatio` の引き上げ（既存設定）である。README には載せず、検証拒否が頻発する場合のフォローアップ判断材料とする（セクション「既知の制約とフォローアップ候補」）。

### 3-4. ON→OFF 切替時

- OFF へ切り替えた後の最初のインタラクション（追記または読み出し）で store が全消去される。以後 Ctrl + Shift + Space は消費されずアプリへ通り、収集も止まる。
- 切替に伴う表示・状態遷移の変化はない（composition が生きていればそのまま継続する）。

## 4. アクセシビリティ

### 4-1. キーボードのみで完結すること

文脈メモリの有効化（UserDefaults の JSON 編集）から、収集（commit するだけ）、文脈つき変換（Ctrl + Shift + Space）、候補比較（↑/↓）、確定（Enter）、復元（Escape）まで、ポインティングデバイスを要する操作は存在しない。WCAG 2.1.1（キーボード）に適合する。

### 4-2. VoiceOver

marked text の設定・確定は既存の `setMarkedText` / `insertText` 標準経路のみを使い、本機能で経路を追加・変更しない。VoiceOver への通知は macOS の marked text 標準挙動に依存する。文脈つき変換の結果も converted のテキスト差し替えとして読み上げ対象になり、通常変換と同じ体験になる。

### 4-3. WCAG 2.1 AA 観点の新規リスク評価

[`2026-06-11-多言語変換キー-design.md`](./2026-06-11-多言語変換キー-design.md) セクション 5 の評価を継承し、後退は発生しない。

| 達成基準 | 本機能での位置付け |
| --- | --- |
| 1.4.1 色の使用 | 状態・文脈の区別に色を使わない（文脈の視覚区別自体を持たない）。新規リスクなし。 |
| 1.4.3 / 1.4.11 コントラスト | ホストアプリ依存。変更なし。 |
| 2.1.1 キーボード | Ctrl + Shift + Space のみで完結する。追加入力を要しない。適合。 |
| 2.2.1 タイミング調整可能 | 文脈つき変換待ち中も編集・Escape・Enter・Ctrl + C を常時受け付ける。文脈収集に時間制限の概念は無い。変更なし。 |
| 3.2.2 入力時 | 明示キーによってのみ文脈つき変換が起きる（pull 型のみ）。収集はユーザーの commit 操作の副作用だが表示・文脈の変化を引き起こさない。変更なし。 |
| 4.1.2 名前・役割・値 | `setMarkedText` 標準経路を維持。変更なし。 |

### 4-4. 既知の制約（変更なし・本機能で解消しない）

- failed と composing の視覚的区別が不能（メッセージ表示チャネルが無い）。
- converting 中の進行フィードバックが無い。
- 下線スタイルの変化（細 → 太）は読み上げでは伝わらない。
- 文脈つき変換と通常変換の区別が視覚・読み上げの双方で不能（セクション 2-2 の意図した判断）。

```markdown
`Ctrl + Shift + Space` は `contextMemoryEnabled` を `true` にしたときだけ Koto が消費する。同じ Koto で確定した直近のテキスト（最大 5 件・計 500 文字）を参照情報として付与し、「あれやっといて」のような省略表現を具体的な文へ展開する。設定が `false`（デフォルト）のとき、および入力中でないときはキーをアプリへそのまま通す。文脈が何も無いときの結果は `Shift + Space` と同じになる。結果は通常の候補と同じく `↑` / `↓` で見比べられ、`Escape` で元のテキストへ戻せる。JetBrains 系 IDE の Smart Type Completion（`Ctrl + Shift + Space`）とは、設定を有効にして入力中のときだけ競合する（Koto が優先される）。
```

### 5-4. プライバシーの節（`## プライバシー` の箇条書きへ追加）

```markdown
- セッション内文脈メモリ（`contextMemoryEnabled`）は既定で無効。有効にしても文脈はデバイスから出ず、メモリ上にのみ保持され、ディスクへ書かない。OFF へ戻すと保持済みの文脈は次の入力処理の時点で全消去され、以後の収集も止まる。アプリ（IME プロセス）の再起動でも消える。文脈テキストは診断ログに残さない。
- パスワード欄では macOS が IME を無効化するため、Koto はセキュアフィールドの入力を受け取らず、文脈にも入らない。
```

### 5-5. 既知の制限の節（`## 既知の制限` へ 1 行追加・任意）

```markdown
- 文脈つき変換（`Ctrl + Shift + Space`）はセッション内の記憶のみを使う。再起動すると文脈は消え、セッションをまたぐ記憶は持たない。
```

---

## 既知の制約とフォローアップ候補

本機能のスコープでは解決しない事項。`/follow-up add` で記録し、別 Issue・別 PR で扱う。

- commit 直後の追記は `Task { @MainActor in ... }` で遅延するため、「Enter で commit → 即座に次の入力 → 即 Ctrl + Shift + Space」の最速操作で直前 commit が文脈スナップショットに間に合わない理論上の窓がある。MainActor 上の逐次実行で実用上は問題にならない見込みだが、QA はこの順序のテストを reducer / coordinator レベルで固定すること。
- 短い入力からの文脈展開が膨張率検証（4.0 × 原文長）で拒否されやすい。第一版は安全側に倒すが、拒否率が高ければ「文脈つき変換時のみ許容を緩める」判断を別 ADR で検討する。
- 文脈つき候補と通常候補の視覚区別が無い（セクション 2-2 の意図した判断）。候補メタデータ表示チャネルの導入時に再検討する。
- JetBrains 系 IDE の Smart Type Completion とは ON かつ composing 中に競合する。実機検証時に `docs/terminal-compatibility.md` へ結果を追記すること。
- アプリ別無効化・incognito モード・翻訳 target への文脈付与は仕様のスコープ外（将来 ADR）。
