# ADR-0007: 保護語の語境界照合とモデル入力の型分離

- **Status**: Accepted
- **Date**: 2026-06-10
- **Deciders**: susumutomita (susumutomita)

## Context

ローマ字かな変換の前段化（[ADR-0006](./0006-決定論ローマ字かな変換の前段化.md)）の初期実装に対し、ローカルセッションのコードレビュー（7 観点の finder + 実証ベースの検証）で、保護語まわりの実害あるバグが実機コンパイルで確認された。(1) 保護語の照合が部分文字列一致のため、語の途中に保護語を含む語が分解される（`tabun` + 保護語 `bun` → 「たbun」、`kotoba` + `koto` → 「kotoば」）。(2) 保護語の正規化（trim・空除去）が normalize・validator・プロンプトの 3 層でばらばらで、前後空白付きの保護語が黙って保護されない。(3) Tab 即時変換の経路だけ保護語が渡っておらず、AI 経路と挙動が分裂する。(4) `'` を無条件に読み飛ばすため英語の省略形が破壊される（`goin'` → ごいん）。(5) モデル入力（かな化後）と検証・復元基準（元テキスト）の区別が `PromptBuilder.prompt` 内部の実装詳細でしかなく、provider が正規化を呼び忘れても型もテストも検出できない。

## Decision

1. **保護語は語境界で照合する**。保護語の出現は「語境界直後に始まり、終端がテキスト末尾・空白・変換可能句読点（`.` `,`）」の場合のみ保護する。複数語フレーズ（`Claude Code`、`bun run`）は長い候補を優先して照合する。語の途中の偶然の一致は保護せず、その語は通常の変換規則（大文字・解釈不能なら原文維持）に委ねる。
2. **保護語の正規化を `ConversionSettings.sanitizeProtectedTerms`（trim + 空除去）へ一元化**し、かな化・出力検証・プロンプト構築のすべてが `sanitizedProtectedTerms` を使う。
3. **Tab 即時変換にも保護語を渡す**。reducer の純粋性を保つため `CompositionTransition.reduce` に環境引数（`protectedTerms`、デフォルト空）を追加し、`.normalizeToKana` のときだけ Coordinator が設定から注入する。
4. **`'` は撥音の区切り（`n` の直後かつ次が母音 / y）としてのみ消費する**。それ以外の `'` を含む語は原文維持。これにより `han'i` → はんい は維持しつつ、`goin'` / `don't` の黙った破壊を防ぐ。初期実装がテストで固定していた `kon'` → こん は原文維持へ変更した（語末 `'` の利便より英語省略形の保全を優先）。
5. **`ConversionRequest` に `modelInputText`（computed property）を導入**し、provider は常にこれをモデルへ渡す。`sourceText` は「打たれたままの元テキスト」として表示・Escape 復元・出力検証の基準であり続ける。検証を元テキスト基準にすることで、かな化後テキスト基準だと起きる二方向の故障（語境界なし出現の保護語消失の見逃し / かな化で合成された文字列へのひらがな保護語の誤一致による恒常的な変換拒否）を防ぐ。両ケースはコーディネータ経由のテストで固定した。

## Consequences

- **Good**: 保護語の挙動が 3 層（かな化・検証・プロンプト）で一貫する。モデル入力と検証基準の区別が型に現れ、provider 実装の呼び忘れがテスト（`receivedModelInputTexts` の観測）で検出できる。Tab と AI の 2 経路で保護語の意味が同じになる。
- **Bad**: 語境界の定義（空白 + 変換可能句読点）がかな表の句読点エントリと暗黙に結合する。表へ句読点を追加する際は語境界の追随を確認する必要がある。
- **Tradeoff**: 語の途中に出現する保護語（`bunwokakunin` の `bun`）はかな化を保護しない。安全網は出力検証（元テキスト基準の保護語喪失検出 → 変換失敗で原文保持）が担う。語中保護まで前段で行う案は、助詞結合の多いローマ字日本語入力で誤保護（過剰な原文維持）を量産するため採らなかった。
