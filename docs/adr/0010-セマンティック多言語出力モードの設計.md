# ADR-0010: セマンティック多言語出力モードの設計

- **Status**: Accepted
- **Date**: 2026-06-11
- **Deciders**: Susumu Tomita (susumutomita)

## Context

多言語変換キー（ADR-0009、PR 29）で「composition をターゲット言語へ直接変換する」第 1 段階を実装した。Issue 30（https://github.com/susumutomita/koto-input/issues/30 ）は、これを「雑なローマ字・混在日本語の意図を汲み、選択した言語の自然な出力へ再構成するセマンティック入力メソッド」へ発展させる製品レイヤーを要求する。子 Issue は 31（設定モデル）、32（プロンプト）、33（検証）、34（候補表示）、36（品質フィクスチャ）、37（アプリ別プリセット）。設計にあたり、ターゲット言語のモデル化・トーンプロファイル・候補と確定の挙動・プライバシー・非対応言語の失敗挙動・テスト戦略を定める必要がある。

## Decision

1. **ConversionMode の独立型は作らず、`ConversionTarget` を唯一の真実とする**。`.japanese` は従来の整文・かな漢字変換パス、それ以外は多言語出力パスを意味する。並行する mode 型を足すと「mode = japanese かつ target = english」のような不正状態が表現可能になるため採らない。Issue 31 の「ConversionMode 相当」はこの決定で満たす。
2. **ターゲット言語のメタデータを `ConversionTarget` に持たせる**。`localeIdentifier`（BCP 47: ja / en / zh-Hans / ko / fr / de / es / ar）、`displayName`（設定 UI・候補ラベル用）、`isRightToLeft`（アラビア語 = true）。実行時の利用可否はメタデータではなく provider が `SystemLanguageModel` の対応状況で判定し、非対応なら `modelUnavailable` で fail-safe（原文保持）する。
3. **アラビア語（`.arabic`）を「キー割当の無い表現可能ターゲット」として追加する**。Ctrl + Shift + 言語キーには割り当てない（`init?(languageKey:)` は解決しない）。設定・プロンプト・検証・フィクスチャの全層で他言語と同じ契約に従い、RTL テキストが Unicode 安全に検証を通ることをテストで固定する。
4. **多言語出力のトーンは `OutputProfile`（neutral / polite / business / casual / technical）で表す**。翻訳 instructions の `[STYLE]` セクションへ写像する。日本語変換の既存 `style`（neutral / polite / plain）と `customInstruction` は変更せず、多言語パスには適用しない（ADR-0009 を踏襲）。プロファイルは保護語・検証の挙動を変えない。
5. **設定は既存の UserDefaults JSON（`conversionSettings`）に後方互換で追加する**。`outputProfile`（既定 neutral）を decodeIfPresent で読み、旧 JSON は既定値で成功する。日本語変換が既定動作のまま変わらないことをテストで固定する。
6. **中間意味表現（intent reconstruction の明示パイプライン）は導入しない**。現行は「かな正規化済み入力 → ターゲット言語の直接生成」1 段で、小型オンデバイスモデルに 2 段生成（日本語確定 → 翻訳）を強いるとレイテンシと誤り伝播が増える。プロンプトの要求（意図・断定度・不確かさの保持）で意味の再構成を表現し、将来 2 段が必要になったら ADR で supersede する。
7. **候補と確定の挙動**: 変換結果は marked text 内の単一候補として表示し、確定は常にユーザーの Enter のみ（自動 commit なし。これは現行の状態機械が既に保証している）。再抽選は同 target 再要求（attempt + 1）、Escape は常に原文へ復元。複数候補の同時表示・ランキング・選択 UI は Issue 34 で別途設計し、本 ADR は単一候補 + 再抽選を当面の確定挙動として固定する。
8. **品質はゴールデン一致ではなくフィクスチャ契約で評価する**（Issue 36）。フィクスチャは `acceptableOutputs`（人間評価の参考）、`mustPreserve`（機械検証: 保護語・識別子の残存）、`mustNotAdd`（機械検証: 入力に無い断定の不在）を持ち、決定的な完全一致は要求しない。モデル非呼び出しでスキーマと契約をテストする。
9. **プライバシー制約**: 入力・出力・プロンプトをログに残さない（診断はリクエスト ID・所要時間・状態遷移・エラー種別のみ）。クラウド翻訳・外部モデルランタイム（OpenAI / Ollama / LM Studio 等）を使わない。アプリ別プリセット（Issue 37）を将来導入してもアプリ内容を保存・送信しない。

## Consequences

- **Good**: 既存の安全機構（原文スナップショット・stale 拒否・Escape 復元・確定は Enter のみ）が多言語出力にそのまま適用され、「見えない書き換え」が構造的に起きない。型の重複が無く、言語追加はケース追加 + メタデータ + フィクスチャで完結する。
- **Bad**: 単一候補 + 再抽選は複数候補比較より試行回数が多くなり得る（Issue 34 で解消予定）。アラビア語はキー割当が無く、設定経由でしか使えない。
- **Tradeoff**: 中間意味表現の見送りは「日本語を経由しない直接生成」の品質をモデル任せにする。品質課題が観測されたらフィクスチャ（Issue 36）を根拠に 2 段生成を再検討する。
