# ADR-0016: かな漢字変換は辞書バックボーンと AI 再ランクのハイブリッド

- **Status**: Accepted
- **Date**: 2026-06-13
- **Deciders**: Susumu Tomita (susumutomita)

## Context

これまでのかな漢字変換は `RomajiKanaConverter`（規則ベースのローマ字かな化、
ADR-0006）の後段が Apple FoundationModels 100% であり、かな漢字辞書やラティスを
一切持たなかった。小型オンデバイス LLM が greedy デコードで変換を据え置くと、
「ほうほう」が「ほうほう」のまま返る no-op が発生する。当時の
`ConversionOutputValidator` は長さ・保護語・頭字語しか検証せず、全かな未変換の
出力を素通しさせていた。さらに辞書が無いため「方法」という正しい表記がそもそも
生成される保証が無い。

純 AI は文脈による曖昧性解消は得意だが網羅性と決定性を担保できず、純辞書は
同音異義を自力で選べないため従来 IME のように候補切り替えを強いる。koto-input の
差別化の核は「読みを打って一発・切り替えなし」であり、どちらか単独では成立
しない。

## Decision

かな漢字変換を「辞書バックボーンが一次変換、AI が二次変換（再ランク・整形）」の
ハイブリッド構成にする。

- **DictionaryConverter（KotoCore・決定的・I/O とモデル非依存）**: 読み（かな）を
  左から最長一致でセグメント分割し、各セグメントに辞書から最小コストの表記を
  選んで連結する。単一最良文字列とセグメントごとの代替候補集合を返す。辞書ヒット
  しないセグメントはかなのまま残す。最小コスト選択は cost 昇順 → 表記コード順の
  安定順序で一意に定め、同一読みは常に同一の既定出力を返す。
- **辞書データ**: mozc `dictionary_oss`（BSD-3-Clause）の高頻度サブセットを、
  読みをキーとしたコンパクトな TSV（reading\tsurface\tcost、reading 昇順）へ
  前処理して SwiftPM リソースとして同梱する。サブセットの抽出条件（内容語 POS・
  cost 閾値・ひらがな読みのみ等）は `Tools/mozc-dictionary-subset/build-subset.sh`
  に明記し、帰属はリポジトリ直下 `THIRD-PARTY-LICENSES` に記載する。辞書は
  ローカルデータで外部送信しない。
- **HybridConversionProvider（`TextConversionProvider` 実装）**: DictionaryConverter
  と AppleFoundationModelsProvider を合成する。convert で reading =
  `request.modelInputText` を辞書変換して単一最良 dictBest を得て、AI に dictBest と
  reading と contextEntries を渡して文脈再ランク・整形し、`ConversionOutputValidator`
  を通す。検証成功なら AI 出力を確定、AI が不可用・未準備・失敗・検証 NG のときは
  dictBest を確定フォールバックとして返す。さらに辞書認識の no-op ガードを持ち、
  AI 出力が全かな（長音符を含む）へ戻ったのに dictBest が漢字化できているときは、
  AI の据え置き（no-op）とみなして dictBest を優先する。availability は辞書
  バックボーンが常に一発変換を成立させるため常に `.available` を返す。
- **非日本語ターゲットは辞書バックボーンの対象外**: 同梱辞書は日本語の表記しか
  持たないため、翻訳ターゲット（`.english` など、ADR-0009 / 0010）の要求に日本語の
  dictBest を返すと「日本語を訳文として確定」してしまう。convert は target が
  `.japanese` 以外なら AI へ完全委譲し、AI が不可用・未準備のときは日本語を返さず
  `KotoError.modelUnavailable` を伝播させる。availability が常に `.available` で
  coordinator の不可用ショートサーキットを通さない分、この分岐で言語安全性を担保する。
- **no-op 判定は provider 層（辞書認識）に置く**: 出力だけを見ても「未変換の echo」と
  「正規のかな表記（あれをやっておいて 等）」は字種では区別できない。区別には辞書が
  漢字化できたかのシグナルが要るため、no-op の真偽判定は字種しか持たない
  `ConversionOutputValidator` ではなく、辞書を持つ HybridConversionProvider で行う。
  validator は空・長さ・保護語・頭字語のみを検証し、全かな出力を字種では棄却しない
  （正規のかな表記を過剰棄却して failed 状態へ落とさないため）。
- **候補プールと切り替え**: 既定の確定値は単一最良出力のみ。候補プールには AI 出力・
  dictBest・辞書代替・別 attempt を順に積むが常時提示はしない。切り替えは ↑↓ の
  任意操作のときだけ巡回する（ADR-0012 踏襲）。

## Consequences

- **Good**: 辞書が既知の読みに必ず漢字表記を与え、provider の辞書認識 no-op ガードが
  AI の据え置きを弾くため、全かな未変換 no-op を撲滅できる。AI 不可用時も辞書だけで
  一発変換が成立し、AI は文脈での再ランクと整形に専念できる。入力テキストは外部送信
  しない invariant を維持する（辞書はローカル、AI はオンデバイス）。
- **Good**: 字種だけを見る validator では「未変換の echo」と「正規のかな表記」を
  区別できないため、no-op 判定を辞書シグナルのある provider 層へ寄せた。これにより
  全かなのまま正しい文（あれをやっておいて 等）を過剰棄却で failed に落とさない。
- **Bad**: 辞書サブセット（MVP は高頻度数千語）の同梱で配布物がわずかに増える。
  サブセット外の読みは辞書で漢字化されず、AI とフォールバックに委ねる。
- **Tradeoff**: 文節分割と Viterbi ラティスによる多文節最小コスト変換、ユーザー学習、
  mozc 全辞書の同梱、送り仮名・用言活用の厳密処理は後続 PR（follow-up）へ送る。

## 関連 ADR の前提更新

- **ADR-0008（再変換は温度付き再抽選）/ ADR-0015（greedy デッドエンドの自動回復）**:
  「変換は AI 単独」を暗黙の前提にしていた。本 ADR でかな漢字変換が辞書一次 + AI
  二次のハイブリッドになるため、provider が返す単一最良は「AI 出力または辞書最良」
  となる。attempt 機構と検証失敗時の自動回復の仕組みそのものは変更せず、
  HybridConversionProvider が `TextConversionProvider` 契約の内側で辞書フォールバック
  に縮退する。ADR-0015 が見送った「かな化結果の最終フォールバック自動提示」とは
  別物で、ここでのフォールバックは辞書が漢字化した一次変換（dictBest）である。
- **ADR-0006（決定論ローマ字かな変換の前段化）**: 辞書変換の入力は前段かな化済みの
  `modelInputText` であり、本 ADR は ADR-0006 の上に辞書段を重ねる。
