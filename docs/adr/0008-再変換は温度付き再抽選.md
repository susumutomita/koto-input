# ADR-0008: 再変換は原文からの温度付き再抽選

- **Status**: Accepted
- **Date**: 2026-06-10
- **Deciders**: Susumu Tomita (susumutomita)

## Context

変換の安定化のため greedy sampling を採用した（同じ入力 → 同じ出力）。その副作用として、誤変換時にもう一度 Shift + Space を押しても同じ結果しか出ず、「再変換ができない」という実機フィードバックを受けた（Issue 19）。従来 IME の候補ウィンドウは MVP のスコープ外（Issue 1）であり、候補列挙のための複数生成はレイテンシ面でも現実的でない。

## Decision

converted 状態から編集せずに再要求された変換を「再変換（候補の再抽選）」と解釈する。

- 再変換は表示中の変換結果ではなく、**原文スナップショット（sourceText）から**やり直す。表示も原文へ戻して、タイプ先行の prefix 整合と Escape の復元先を保つ。
- ConversionRequest に attempt（再抽選回数）を持たせ、provider は attempt 0 を greedy（決定的）、attempt 1 以降を温度付きサンプリング（temperature 0.8）にする。押すたびに別候補が出る。
- 編集を挟んだら attempt はリセットされ、新しいテキストの初回（決定的）変換になる。

あわせて、入力に無い鉤括弧で出力全体が包まれる現象（小型モデルの癖）を、プロンプト規則と ConversionOutputValidator の決定論 unwrap（元テキストに `「` `『` `[` が無い場合のみ外す）で抑止する。

## Consequences

- **Good**: 初回変換の決定性（ADR の greedy 判断）を保ったまま、誤変換時の「もう一回」が機能する。Escape は何回再抽選しても原文へ戻る。
- **Bad**: 再抽選は非決定的で、同じ操作の再現性がない（意図した仕様）。候補の一覧表示はできない。
- **Tradeoff**: 候補ウィンドウ（複数候補の同時生成・選択 UI）を見送った。要望が強ければ別 Issue で再検討する。
