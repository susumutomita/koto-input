# ADR-0011: 出力プリセットはプロファイルの明示的な束

- **Status**: Accepted
- **Date**: 2026-06-11
- **Deciders**: Susumu Tomita (susumutomita)

## Context

Issue 37（https://github.com/susumutomita/koto-input/issues/37 ）は、Slack・メール・GitHub Issue / PR・AI エージェント CLI プロンプトといった出力先ごとの多言語出力プリセットを要求する。ADR-0010 で導入した `OutputProfile` はトーンの軸を表すが、出力先には「メールで件名や挨拶を勝手に足さない」「コードレビューで Markdown 構造を保持する」のようなトーンに収まらない固有の要求がある。一方、コードベースにはフロントモストアプリの検出機構が無く、Issue 37 もアプリ検出をスコープ外とする。プライバシー制約（ADR-0010 決定 9: アプリ内容の保存・送信・ログ禁止）も維持する必要がある。

## Decision

1. **プリセットは `OutputPreset`（standard / chat / email / codeReview / agentPrompt）として「OutputProfile + プリセット固有の追加プロンプト要求（presetInstruction）」の名前付き束で表す**。新しいトーン軸やプロンプトセクションは導入せず、プリセットは翻訳 instructions の [STYLE] セクションに閉じて作用する。追加指示は ADR-0010 の「入力に無い情報を足さない」原則と矛盾しない内容に限る（email は件名・宛名・挨拶・署名を足さない、codeReview は Markdown 構造と Issue・PR 参照の保持、agentPrompt は命令形で簡潔に・コマンドやパスを変えない、chat は短く砕けた表現で挨拶や絵文字を足さない）。
2. **ケース名に特定の製品名・サービス名をハードコードしない**。Claude Code / Codex は agentPrompt の代表例であって唯一の対応コンテキストではない。Slack → chat、GitHub Issue・PR → codeReview も同様で、製品名は displayName の「等」付き例示と既定保護語にとどめる。新しいツールの登場でケース名が陳腐化するのを防ぐ。
3. **アプリ検出は実装せず、`appAwarePresetsEnabled`（既定 false）の opt-in フラグのみ先行導入する**。`effectivePreset` は false の間は常に .standard でプリセット不活性、true のときのみユーザーが選択した `outputPreset` を返す。アプリ連動は明示的かつ設定可能で、ユーザーがいつでも無効化できる。将来アプリ検出を導入する場合もアプリ内容の保存・送信・ログはしない（ADR-0010 の制約を踏襲）。
4. **実効プロファイルの優先規則を `effectiveProfile = (effectivePreset == .standard) ? outputProfile : effectivePreset.profile` とする**。プリセット未使用（無効化を含む）のユーザーが ADR-0010 以降設定してきた outputProfile を壊さないための規則で、PromptBuilder の [STYLE] は outputProfile ではなく effectiveProfile を参照する。プリセット → プロファイルの写像は standard → neutral、chat → casual、email → business、codeReview → technical、agentPrompt → technical で、テストで固定する。
5. **プリセットは保護語・出力検証・候補確定の挙動を変えない**。変換結果は従来どおり marked text 内の単一候補で、確定は常にユーザーの Enter のみ（ADR-0010 決定 7）。プリセット選択はこの保証を迂回しない。日本語変換の instructions もプリセットの影響を受けない（ADR-0010 決定 4 の踏襲）。
6. **設定は既存パターンで後方互換に追加する**。`outputPreset` と `appAwarePresetsEnabled` を decodeIfPresent で読み、フィールドの無い旧 JSON は既定値で成功させる。未知のプリセット文字列は .standard へ決定論的にフォールバックし、設定全体が default へ巻き戻るのを防ぐ。

## Consequences

- **Good**: プリセットの追加はケース追加（profile + displayName + presetInstruction）で完結し、状態機械・検証・プライバシー保証に触れない。プリセット未使用ユーザーの挙動はビット単位で不変（フラグ false なら instructions が同一であることをテストで固定）。
- **Bad**: `appAwarePresetsEnabled` は現時点で「プリセット適用の総スイッチ」でしかなく、名前が示すアプリ連動（フロントモストアプリによる自動切替）はまだ存在しない。ユーザーはプリセットを手動で選ぶ必要がある。
- **Tradeoff**: プリセットごとに独立した自由文トーン定義を持つ案は捨て、既存 5 プロファイルへの写像 + 追加 1 行に制限した。表現力は下がるが、[STYLE] の組み合わせ爆発と「入力に無い情報を足す」指示の混入を防げる。アプリ検出の実装時、またはプロファイル写像で表せないプリセットが必要になったときに、新しい ADR で supersede する。
