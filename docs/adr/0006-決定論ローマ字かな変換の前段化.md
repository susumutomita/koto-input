# ADR-0006: 決定論ローマ字かな変換の前段化

- **Status**: Accepted
- **Date**: 2026-06-10
- **Deciders**: Susumu Tomita (susumutomita)

## Context

実機検証で、オンデバイスの小型モデルにローマ字解釈まで任せると変換品質が揺れることが分かった。ローマ字 → ひらがなは規則ベースで決定論的に解決できる問題であり、Emacs の Boiling Egg（Egg・Tamago のフロントエンド、1990 年代）が「素打ちしたローマ字を後からまとめてかな化する」方式として 30 年前に確立している。LLM に任せるべきは、得意で代替の無い「かな漢字変換 + 整文」だけでよい。

## Decision

決定論的な `RomajiKanaConverter` を KotoCore に実装し、2 箇所で使う。

1. **Tab キー**: composition 中の Tab で表示テキストをその場でひらがな化する（AI 不要・即時）。composition が無いときは Tab を消費しない。
2. **プロンプト前段正規化**: `PromptBuilder.prompt` がモデルへ渡す [INPUT] をかな正規化する。Escape 復元・タイプ先行の splice・保護語検証は引き続き「打たれたままの sourceText」を基準にするため、表示系の挙動は変わらない。

変換規則はヘボン式・訓令式の標準対応（促音の子音重ねと `tch`、撥音の文脈判定、`n'`、ヘボン式 `m`+b/m/p、小書きの `x`/`l`、長音 `-`）。Egg の `its/hira.el`（GPL）は**規則の参考のみ**とし、コードは独自実装する（Koto は MIT のため、GPL コードの移植はしない）。

誤変換を防ぐ安全規則として、変換対象を「小文字だけで構成され、ローマ字として最後まで解釈できる単語」に限定する。

- 解釈できない英単語（application、check 等）はそのまま残る。
- 大文字を含む単語（Claude、URL 等の固有名詞）は変換しない。
- `/` や `.`+拡張子、`_` に隣接する単語（パス・識別子）は変換しない。文末の `.` `,` は許容する。

## Consequences

- **Good**: ローマ字解釈の揺れ（n の切り方、typo の誤読）が構造的に消える。Tab だけで AI を使わない高速なかな入力もできる。モデルのプロンプトが「かな + 英語」になり、few-shot との整合も良い。
- **Bad**: 英文中の短い単語（to、go 等）はローマ字として解釈可能なため、かな化されてしまう。日本語ローマ字文の中では同綴りはほぼ助詞なので実害は小さいが、純粋な英文で Tab や AI 変換を使うと崩れる（使わなければ影響なし）。
- **Tradeoff**: かな → 漢字の決定論変換（辞書）は持たない。漢字化は LLM の仕事のまま。辞書が必要になったら別 ADR で再検討する。
