# ADR-0013: ローカル文脈メモリは opt-in のセッション内記憶から始める

- **Status**: Accepted
- **Date**: 2026-06-11
- **Deciders**: Susumu Tomita (susumutomita)

## Context

「あれやっといて」のような省略表現をそのまま変換しても、受け手に意図が伝わる文にはならない。Koto は全アプリの入力が合流するオンデバイス IME であり、その位置づけを活かして「今セッションで何を書いてきたか」を参照すれば、曖昧な意図を具体的な候補へ展開できる。これは AI Native IME の主要な差別化であり、Issue 43（https://github.com/susumutomita/koto-input/issues/43）が設計を要求している。

一方、ADR-0002 は「入力テキストをログに残さない」と定めており、文脈の収集と診断ログの禁止を混在させると設計の意図が失われる。この 2 つは目的が異なる別概念として定義する必要がある。

## Decision

### 1. 提案は pull 型のみ（明示キー起動）

文脈候補は、ユーザーが明示的に起動したときだけ生成する。push 型の自動提案は実装しない。"Agent suggests text. User acts." の原則を維持し、候補の表示は既存の marked text 巡回選択（ADR-0012 の ↑/↓）を拡張して行う。自動介入によるタイピングの阻害・誤挿入のリスクをゼロにする。

### 2. 文脈はユーザープロンプトの [CONTEXT] セクションに配置する

instructions に文脈を含めると、prepared box の prewarm 効果（ADR-0005）が失われる。prewarm は instructions の一致でのみ機能するため、instructions は固定に保ち、文脈は `[CONTEXT]` セクションとしてユーザープロンプト側に挿入する。[CONTEXT] は「指示ではなく参照情報」と明示し、[INPUT] と同じインジェクション防御（保護語照合・長さ上限）を適用する。

### 3. 診断ログと作業記憶の概念分離

| 概念 | ADR-0002 の扱い | 本 ADR の扱い |
|------|----------------|--------------|
| 診断ログ | 入力・変換テキスト記録禁止（変更なし） | 変更なし |
| 作業記憶 | 対象外（概念が存在しなかった） | opt-in・既定 OFF・ローカルのみ・全消去可能・incognito モードあり |

作業記憶はユーザーが意図して有効にするものであり、ADR-0002 の診断ログ禁止とは目的が異なる。両者の区別をコードと設定 UI に明示する。

### 4. 第一版のスコープ：セッション内 in-memory 保持

第一版は「同一入力セッション内の直近 commit テキスト N 件を in-memory で保持し、明示キーで文脈つき AI 候補を生成する」。

- **ディスク永続なし**（再起動でメモリは消える）。
- **新権限なし**（Accessibility 権限・新エンタイトルメントは要求しない）。
- SQLite 永続・アプリ連動・window title 取得（Accessibility 権限が必要）は将来の ADR へ先送りする。
- `candidate_events` の生テキスト永続は第一版では持たない。

### 5. メモリ層の定義

| 層 | 内容 | 第一版 |
|----|------|--------|
| session | 同一セッション内の直近 commit N 件（in-memory） | 実装する |
| working | セッションをまたぐ短期メモ（SQLite、TTL 付き） | 将来 ADR |
| vocabulary | 頻出語・固有名詞（SQLite） | 将来 ADR |
| style | 文体・トーン（OutputPreset / ADR-0011 が部分実装済み） | 将来 ADR |
| project | プロジェクト固有ルール（ファイルパス等） | 将来 ADR |

### 6. レイテンシ境界

docs/performance.md のレイテンシ収支を維持する。

| パス | 上限 | 手段 |
|------|------|------|
| 決定論変換（hot path） | 10 ms | 変更なし（ADR-0006） |
| 候補 UI 更新 | 50 ms | 変更なし |
| 文脈組み立て（[CONTEXT] 構築） | 20 ms | composition 開始時に非同期で先行実行 |
| AI 文脈候補生成 | 非同期のみ | hot path に入れない |
| メモリ書き込み（commit 後） | hot path 禁止 | background task |

### 7. プライバシーコントロール一覧

- 全体 ON/OFF（既定 OFF）
- アプリ別無効化（特定アプリで収集しない）
- 全消去（即時、UI から実行可能）
- incognito モード（一時的に収集停止、再起動時は自動 OFF）
- セキュアフィールド非収集（パスワード欄への入力は収集しない）

### 8. 評価方針

品質フィクスチャ（`docs/specs/2026-06-11-多言語品質フィクスチャ.md`）を拡張し、`context` フィールドを追加する。文脈あり / なしのペアでテストケースを構成し、文脈注入が候補品質を改善することを回帰テストで固定する。

## Consequences

- **Good**: 既存の安全機構（stale 拒否・Escape 復元・保護語検証・prewarm）をすべて継承したまま文脈機能を追加できる。プライバシー保護がオンデバイス設計の差別化として正の価値に転化する。新権限・新ディスク I/O がなく、第一版のリスクが低い。
- **Bad**: セッション限りのメモリのため、再起動すると文脈が消える。文脈の質は「今セッションで入力した commit テキスト」のみであり、浅い参照に限られる。
- **Tradeoff**: ディスク永続（SQLite）とアプリ連動は、永続メモリの価値を第一版の採用率・品質改善幅で検証してから別 ADR で判断する。先に権限・ストレージ設計を固めると検証前のオーバーエンジニアリングになる。window title 取得は Accessibility 権限を要求するため、費用対効果が明確になるまで採用しない。
