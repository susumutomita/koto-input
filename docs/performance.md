# パフォーマンス設計（なぜ変換が速いのか）

デモを見た人から「動画のスピードのまま動いているのか」と聞かれることがある。Koto の変換は実時間で動いており、速さは単一の魔法ではなく、レイテンシを「発生させない・隠す・最小化する」工夫の積み重ねで実現している。本書はその知見の正本である。

## レイテンシの収支

体感速度を決めるのは「Shift + Space を押してから訳文・変換文が出るまで」の時間で、内訳は以下に分解できる。

```text
ネットワーク往復 + サーバキュー待ち   → ゼロ（オンデバイス）
モデルのロードと instructions 評価    → 先払い（prewarm）
プロンプトの prefill                → 最小化（短い指示 + かな化入力）
デコード（出力トークン生成）          → 最小化（訳文のみ・greedy）
UI ブロック                        → ゼロ（非同期 + タイプ先行）
```

## 工夫 1: オンデバイスモデルで往復ゼロ（ADR-0002）

変換は Apple Foundation Models（Apple Intelligence のオンデバイスモデル）のみで行う。クラウド API のネットワーク往復・サーバ側のキュー待ち・レート制限が構造的に存在しない。モデルは数 B パラメータ級の小型モデルで、Apple Silicon の Neural Engine 上で動くため 1 トークンあたりの生成も速い。

## 工夫 2: prewarm でモデルの起動を先払いする（ADR-0005）

最初のキー入力（idle → composing）の時点で `LanguageModelSession` を作り `prewarm()` を呼ぶ。ユーザーがローマ字を打っている数秒間が、モデルのロードと instructions の評価に充てられる。変換キーが押された時点ではセッションが「温まって」おり、prefill の大部分が完了している。

- 実装: `AppleFoundationModelsProvider` の prepared box。instructions が一致する次の変換で 1 回だけ使う。
- prewarm は日本語のみ（ADR-0009）。翻訳 6 言語分のセッション常駐を避け、メモリと電力を抑える。このため翻訳の初回変換だけは日本語より待ち時間が長い場合がある。

## 工夫 3: キーイベントの同期パスでモデルを呼ばない

`InputController.handle()` はキーをドメインコマンドへ変換して描画するだけで、モデル呼び出しを一切しない。変換要求はまず `converting` 状態を即時描画し、その後の非同期タスクでモデルを呼ぶ。タイピングがブロックされる瞬間がゼロなので、変換中でも入力メソッド自体は常に軽い。

## 工夫 4: タイプ先行で待ち時間を隠す（ADR-0005）

変換中もタイピングを止めない。原文スナップショットが先頭に残る末尾追記なら変換は継続し、結果はスナップショット部分にだけ splice される。「変換を待つ」時間を「次の文を打つ」時間に重ねることで、体感の待ち時間を隠す。

## 工夫 5: 決定論の前段かな化でモデルの仕事を減らす（ADR-0006）

ローマ字 → ひらがなは AI を使わず `RomajiKanaConverter` が決定論・即時に行う（Tab の即時かな化は同じ変換器で 0 ミリ秒級）。モデルへはかな化済みテキストを渡すので、モデルの仕事は「かな漢字変換と整文（翻訳時は翻訳）」に絞られる。ローマ字解釈の揺れによる誤読・やり直しが構造的に消え、再変換の回数自体が減る。

## 工夫 6: greedy デコードと最小プロンプト

- 初回変換（attempt 0）は greedy sampling。温度付きサンプリングより速く、同じ入力には同じ出力が返る（再抽選時のみ温度 0.8。ADR-0008）。
- instructions は型付きセクション + few-shot 1 例だけの短いプロンプト。出力は「変換後テキストのみ」を要求し、説明・前置き・引用符を生成させない。prefill とデコードの両方でトークン数が最小になる。
- 出力長は膨張率上限（元テキスト × 4 + 64）で検証し、暴走した長文生成を受け取らない。

## 工夫 7: 使い捨てセッションで prefill を一定に保つ（ADR-0002・ADR-0005）

セッションは変換ごとに使い捨てで、transcript を次の変換へ持ち越さない。会話履歴が累積しないため、prefill のコストが何回変換しても一定で、長く使うほど遅くなる現象が起きない（プライバシー上の利点と同じ設計判断から来ている）。

## 工夫 8: 古い計算は即捨てる

新しい変換要求・スナップショットを壊す編集・commit・cancel・deactivate は実行中の変換タスクを即 cancel する（ベストエフォート + stale 三重照合）。計算資源を常に「最新の要求」だけに使う。

## 計測とプライバシー

診断はリクエスト ID・所要時間・状態遷移・エラー種別のみを記録し、入力テキスト・変換結果は記録しない（ADR-0002）。レイテンシの計測を理由にプライバシー保証を緩めない。

## 既知の限界

- 翻訳の初回変換は prewarm 対象外のため遅め（ADR-0009 のトレードオフ）。
- 再抽選（attempt 1 以降）は温度付きサンプリングのためわずかに遅くなり得る。
- OS 側のモデル常駐状態（他アプリの Apple Intelligence 利用状況）には依存する。
